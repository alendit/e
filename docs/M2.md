# M2: Make The Core Actually Work

## Goal

Make `e` produce real model-backed answers while preserving the harness-first
architecture established in M1.

M2 should connect the current pure core to a real OpenAI/Codex backend using
ChatGPT subscription auth, while keeping context construction, provider auth,
tool execution, and presentation as separate responsibilities.

## Scope

M2 should build the smallest working real-agent path:

1. Create a provider-neutral context strategy seam.
2. Add the first `transcript-stack` context strategy.
3. Add a real OpenAI/Codex backend adapter using ChatGPT subscription auth.
4. Prove a minimal prompt/response flow against the real backend.
5. Add one harmless concrete tool to validate real tool-call plumbing.
6. Add the minimum async/active-turn behavior needed for real streaming.

M2 should not implement canvas mode, a rich presentation shell, dangerous Emacs
mutation tools, durable persistence, or harness self-modification.

## Step 1: Context Strategy Seam

Add an explicit context-management boundary before implementing the real
backend.

The first strategy is `transcript-stack`: build backend-neutral model input
from the ordered session transcript, including prior user messages, assistant
messages, tool calls, and tool results.

This is intentionally classic and simple. Its purpose is to make the first real
backend work without baking transcript replay into the OpenAI adapter.

Required invariant:

- The OpenAI adapter receives backend-neutral messages, tools, and options.
- The OpenAI adapter does not decide how session state becomes context.
- Future strategies such as `canvas-state` can replace `transcript-stack`
  without changing provider auth or presentation code.

## Step 2: OpenAI/Codex Backend Adapter

Add a real backend adapter for OpenAI access through ChatGPT subscription auth.

Adapter responsibilities:

- locate/read Codex-managed auth
- handle auth refresh or token acquisition details
- map backend-neutral messages/options/tools to OpenAI request shape
- stream assistant deltas, tool calls, done, and errors into the existing loop
  contract
- keep provider-specific headers, request shapes, retries, and model capability
  mapping outside the harness and context strategy

The adapter should be tested separately from the core loop. Fake backend tests
should remain the proof that the core is provider-independent.

## Step 3: Minimal End-To-End Flow

Build one small real flow before adding UI complexity:

```text
create session
-> prompt
-> transcript-stack context
-> OpenAI backend
-> assistant response
-> persisted assistant message
-> structured events
```

This can be exercised through an Emacs Lisp function or a small interactive
command. It does not need a rich buffer UI.

Success means `e` can answer a plain prompt using real ChatGPT-backed Codex
auth and record the response in the session.

## Step 4: One Harmless Tool

Add one concrete low-risk tool to validate real model tool calls.

Good candidates:

- current time
- list buffers
- read current buffer name
- read a named buffer without editing it

Avoid file writes, buffer edits, process execution, elisp evaluation, and
harness mutation in M2.

Success means a real model can request a tool call, the registry can execute
it, the loop can append a structured tool result, and the backend can continue
or settle the turn.

## Step 5: Async And Active Turns

The current loop is synchronous. Real backend streaming needs the smallest
useful async model.

M2 should introduce:

- active turn tracking
- streaming event delivery
- wait/settled state
- cancellation or abort semantics
- surfaced provider errors

Keep this narrow. Do not build a general scheduler before real streaming
proves what the harness needs.

## Deferred To Later Milestones

M2 explicitly defers:

- `canvas-state` context mode
- block-tree canvas storage
- durable session persistence
- rich Emacs presentation buffers
- dangerous Emacs tools
- file writes
- process execution
- elisp evaluation
- harness self-modification
- multi-agent delegation

## Architecture Constraints

- Harness owns lifecycle and policy.
- Context strategy owns backend context construction.
- Backend adapter owns provider auth and provider request/response mapping.
- Tool registry owns tool lookup and structured tool results.
- Execution environment owns concrete Emacs side effects.
- Presentation shells own rendering, keymaps, and user interaction only.

No M2 change should make provider auth, transcript replay, or UI rendering part
of harness policy.

## Relationship To Canvas Mode

Canvas mode remains a deferred experiment documented in `docs/feat-canvas.md`.

M2 should make canvas mode possible by adding the context strategy seam, but it
should not implement canvas behavior. The only context strategy required for M2
is `transcript-stack`.

## Recommended Stop Point

M2 is complete when:

- `e` can send a real prompt to OpenAI using ChatGPT-backed Codex auth.
- The response is streamed or collected through the backend contract.
- The assistant message is stored in the session.
- Events are emitted through the harness.
- One harmless tool works with real model tool-call plumbing.
- Focused adapter tests and the full Eldev suite pass.
