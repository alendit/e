#+title: Project Local Layers
#+startup: overview

* Purpose

Project-local layers let a repository ship model-facing capabilities and
presentation shell manifests together under the project tree.  They are loaded
only for trusted project roots and are flattened into the built-in
=project-local= aggregate layer for the active harness.

* Layout

Preferred package layout:

#+begin_example
<repo>/.e/layers/<layer-id>/layer.el
<repo>/.e/layers/<layer-id>/skills/<slug>/SKILL.md
#+end_example

=layer.el= runs as trusted Emacs Lisp after the project root is allowed in
=e-project-local-allowed-roots=.  It should register one or more factories:

#+begin_src emacs-lisp
(e-project-layer-register
 :id 'topic
 :factory
 (lambda (directory)
   (e-layer-create
    :id 'topic
    :name "Topic"
    :capabilities (list (topic-capability-create directory))
    :shells (list (topic-shell-create directory)))))
#+end_src

The factory receives the layer directory and must return an =e-layer= whose id
matches the registered id.  Capabilities must be =e-capability= values.  Shells
must be =e-shell= values with valid shell and command ids.

* Trust

Project layer loading executes repository Emacs Lisp.  Unknown roots are skipped
with a message.  Add only roots you trust to =e-project-local-allowed-roots=.

The compatibility layout remains supported:

#+begin_example
<repo>/.e/capabilities/<capability-id>/capability.el
#+end_example

Use =.e/layers/= for new project packages.  Keep =.e/capabilities/= for existing
capability-only packages that do not need shell manifests.

* Byte Compilation

Project-local loaders call =load= with extensionless file names and
=load-prefer-newer= enabled.  This lets Emacs choose newer source over stale
=.elc=, or a fresh =.elc= when one is available.  The same policy is scoped
around trusted project layer and capability factories, so helper code inside a
project layer can write ordinary loads and still get fresh byte-code when
present.

Use =M-x e-project-local-byte-compile-project= from a trusted project root to
byte-compile allowlisted project-local files under =.e/layers/= and
=.e/capabilities/=.  The command writes =.elc= files next to the source files
only when the =.elc= file is missing or older than the corresponding =.el=
source.  With a prefix argument, it recompiles every project-local =.el= file.

Automatic byte recompilation on project entry is off by default.  Set
=e-project-local-auto-byte-recompile= non-nil to make Projectile priming call
the byte-compile command before loading an allowlisted project-local layer.
Keep it disabled for projects where writing =.elc= files into the repository is
undesirable.

* Shell Discovery

Built-in shells are still registered globally with =e-shell-register= and read
with =e-shell-list= or =e-shell-get=.

Layer-owned shells are active only for the harness that activated the owning
layer.  Callers that have a harness should use:

- =e-shell-list-active=
- =e-shell-get-active=

Global shell ids take precedence.  If a layer shell id collides with a global
shell id, the layer shell is reported and skipped instead of replacing the
built-in shell.

* Projectile Priming

Project-local layer loading normally happens when an =e= harness activates the
built-in =project-local= layer.  For interactive commands supplied by
presentation shell files, this can be too late: commands such as a project's
=M-x my-project-new-topic= do not exist until the layer file has been loaded.

When Projectile is present, =e-project-local= installs lightweight hooks on:

- =projectile-after-switch-project-hook=
- =projectile-find-file-hook=
- =projectile-find-dir-hook=

The hooks call =e-project-local-prime-projectile-project=, which finds the
current Projectile root and calls =e-project-local-prime-project=.  Priming only
runs for projects that contain =.e/layers/= or =.e/capabilities/= and are already
trusted by =e-project-local-allowed-roots=.  It performs normal project-local
discovery early so trusted shell files can define global interactive commands.
It does not activate tools, resources, or layer-owned shell manifests in a
harness; harness activation remains responsible for those model-facing effects.

Set =e-project-local-projectile-prime-on-project-entry= to nil to disable this
integration while keeping ordinary harness-driven discovery.
