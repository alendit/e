** DONE Medium: Reset left queued prompts alive

*** Problem

The first subagent review found that =e-harness-reset= cleared transcript
messages but left =prompt-queues= intact.  A queued follow-up from before reset
could therefore drain after the old active turn settled, mutating a session the
user had just reset.

*** Change

=e-harness-reset= now clears the session's queued prompts and emits
=queue-changed= before =session-reset=.  The reset behavior does not try to
silently redefine active-turn abort semantics; it only removes stale queued
follow-ups owned by the harness queue.

* Architecture review

The ownership follows the corrected plan: the harness owns active-turn
lifecycle, queue drain, and pending steering input; the loop owns same-turn
drain points; the chat-session layer exposes semantic actions; the shell owns
prefix handling and bottom chrome rendering.  Provider adapters do not fake
mid-stream mutation.

* Design self-check

1. The package leaves the project better off by making active-turn submit
   behavior explicit and adding a safe queue instead of accidental concurrent
   turns.
2. It moves toward live-configurable chat workflows where mid-turn input and
   follow-up work are first-class harness/session behavior.
3. Provider mid-stream mutation remains the uncertain decision and is out of
   scope until a real adapter capability exists.
4. Ownership is cohesive: lifecycle in harness, semantic actions in
   chat-session, presentation in chat shell, transport support in adapters.
5. Coupling is reduced because shell code calls session actions instead of
   mutating harness internals directly for steering/queue semantics.
6. Stable policy does not depend on concrete providers; adapters opt in only
   when a real mechanism exists.
7. Side effects stay at the shell and harness boundaries; provider transport
   remains behind backend requests.
8. The new abstractions are semantically real: queueing and steering are
   distinct operations with different turn-lifecycle behavior.
9. Compatibility is explicit: ordinary submit and noninteractive prompt submit
   remain supported.
10. No legacy path was introduced for fake steering.
11. Expected active steering/queue failures preserve the draft and report a
    visible input failure, while ordinary submit keeps its existing error
    behavior.
12. Core behavior is proven by focused backend, harness, session, and chat tests,
    with live reload and visual verification for user-facing behavior.
