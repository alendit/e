#+title: Emacs Cron Schedules for e
#+startup: overview

* Goal

Give =e= a first-class, declarative way to run agent work on a schedule:
cron-like recurrence ("every weekday at 09:00", "every 30 min") that wakes a
session or enqueues a task, defined in config and inspectable/cancelable at
runtime — built on Emacs timers, no external cron.

The scheduled action should support a deterministic check that decides whether the agent actually needs to be fired/queued, so a wakeup with no work is a cheap no-op instead of an empty turn.

There should be an =e= command that gives an overview of all scheduled crons.

* Status

In-progress. Engine, routing/capability layer, and overview command
implemented with focused ERT coverage; live-reloaded and probed in the running
Emacs.

* Problem

=e-background-session= already supports a =schedule-seconds= trigger, but it
is a single fixed interval per trigger, configured in Lisp, with no calendar
recurrence (time of day, day of week), no central registry to list/cancel
schedules, and no integration with the new task queue. There is no way to
say "run the inbox triage sweep every morning" or "refresh sources nightly"
without hand-rolling a timer.

* Design

** Schedule spec

A declarative schedule entry:

- =id= :: stable identifier.
- =when= :: a recurrence — either an interval (=:every 1800=) or a
  calendar spec (=:at "09:00" :on (mon tue wed thu fri)=). Keep the
  calendar surface small and explicit; do not reimplement full crontab
  syntax in the first slice.
- =action= :: what to fire — =:enqueue= a task-queue prompt, or =:wake= a
  background session, or call a named handler.
- =guard= :: an optional deterministic predicate evaluated at fire time that
  decides whether the fire happens. Non-nil fires; nil skips. This lets a
  schedule ask "is there anything to do?" (unread inbox items, stale sources)
  before waking a session or enqueuing a task. The guard runs synchronously
  in the live Emacs and must be cheap and side-effect free.
- =metadata= :: opaque, passed through to the fired work.

** Engine

- A registry of schedule entries plus an Emacs-timer scheduler that computes
  the next fire time from =when= and re-arms after each fire (calendar specs
  recompute against local time; interval specs re-add).
- Fires route to the existing primitives: a =:enqueue= schedule calls
  =e-task-queue-enqueue=; a =:wake= schedule submits to a background session.
  The engine owns *timing only*; it does not run turns itself.
- Busy-skip / catch-up policy is explicit per entry (skip a missed fire vs.
  run once on resume); default skip, matching background-session semantics.
- When an entry has a =guard=, the engine evaluates it at fire time before
  routing to the action. A nil result skips this fire without disarming: the
  schedule still recomputes and re-arms for its next time. The guard decides
  *whether* to fire; it does not affect *when* the next fire is scheduled.

** Inspect and control

- An =e= command opens an overview buffer listing every schedule with its
  =when=, next fire time, last fire, guard result at last check, and enabled
  state; from that buffer the user can enable/disable/cancel an entry.
- The overview reads the same registry the engine arms from, so what it
  shows is the live schedule state, not a separate copy.
- Schedules are config-defined (global/project) and runtime-inspectable; a
  disabled schedule survives restart as config but does not arm.

** Relationship to other features

- This is the *timing* source; the task queue (47) is the *execution*
  substrate and the background session is the *prompt* substrate. A nightly
  triage is "schedule -> enqueue triage prompt"; a periodic source refresh
  is "schedule -> wake the sources session".
- Annotation-reactive sessions (50) are the push-triggered counterpart;
  together they cover schedule and push, the two triggers the loops idea
  called for.

* Non-goals

- No full crontab string parser in the first slice; a small explicit
  calendar spec only.
- No persistence of fire history beyond the last fire in this slice.
- No distributed/multi-Emacs scheduling; timers run in the live Emacs.
- No new core primitive; the engine is a registry + timers feeding existing
  task-queue / background-session entry points.

* Acceptance

- A calendar schedule ("weekdays at 09:00") and an interval schedule
  ("every 30 min") both fire at the right times in a test clock.
- A =:enqueue= schedule lands a task on the queue; a =:wake= schedule submits
  to a background session.
- A schedule with a =guard= fires only when the guard returns non-nil; a nil
  guard skips the fire and the schedule still re-arms for its next time.
- The overview command lists every schedule with its next/last fire and
  enabled state, and can disable/cancel an entry at runtime.
- A missed fire follows the entry's catch-up policy (default skip).

* Verification

- Focused ERT tests with an injected clock: next-fire computation for
  interval and calendar specs, re-arm after fire, busy-skip/catch-up,
  guard gating (non-nil fires, nil skips and still re-arms),
  enqueue vs. wake routing, disable/cancel.
- =git diff --check=.
- =e-dev-reload= and a bounded live probe arming a short interval schedule
  and asserting one fire when an Emacs server is available and the user has
  not asked to skip reload.
