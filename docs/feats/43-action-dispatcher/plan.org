#+title: Action Dispatcher
#+startup: overview

* Goal

Add an ergonomic, context-bound way to call active capability actions from
Elisp, aligned with the existing =e-tools-call= tool-chaining shape.

* Status

Done.

2026-07-05 update: the follow-up nonblocking work migration removed the legacy
direct function action surface.  Capability actions are now =e-action=
descriptors backed by =e-work= specs, and execution goes through
=e-actions-call= / =e-actions-dispatch=.

* Problem

Capability actions are first-class runtime behavior, but callers historically
had only a low-level direct-function accessor.  That historical shape has now
been removed from the public runtime surface.

That forces generic callers to find the active capability, know the action's
calling convention, inject harness/session state manually, and discover errors
late.  It also made action guidance diverge from the existing
=e-tools-call=/=e-tools-call!= shape used by =run_elisp=.

* Design

Add action descriptors:

- =e-action-create= describes a work-backed action, compact JSON-style
  parameter schema, and session requirement.
- =e-action-cheap-create= is the convenience path for bounded immediate actions;
  it still creates a =:cheap= =e-work= spec.
- =e-capabilities-action-spec= returns the descriptor for callers that need to
  inspect metadata.  It is not an execution API.

Add =e-actions-call=:

#+begin_src elisp
(e-actions-call 'chat-session :compact)
(e-actions-call 'chat-session :rename '(:name "Focused chat"))
#+end_src

The dispatcher resolves the active harness/session from =e-tools-current-context=
unless explicit =:harness= and =:session-id= options are supplied.

* Scope

- Add the core descriptor and dispatcher.
- Convert built-in action surfaces to descriptors where they are part of
  standard layers: =chat-session=, =layer-selection=, =agent-shell-fleet=, and
  =goal=.
- Convert generated/dynamic project-local capabilities to descriptor actions.
- Update =run_elisp= guidance to show both =e-tools-call= and =e-actions-call=.
- Document the runtime vocabulary and new action call shape.

* Non-goals

- Do not make every shell action a dedicated first-level model tool.
- Do not add a generic model-facing action tool; agents already have
  =run_elisp=, and =e-actions-call= is the Elisp affordance for that tool.
- Do not make capability actions model-facing tools.
- Do not change action ownership; capabilities still own semantic actions and
  the harness still owns lifecycle/state.

* Acceptance Criteria

- Elisp can call =(e-actions-call 'chat-session :rename '(:name "..."))= with
  harness/session injected from context or explicit options.
- Missing required action arguments produce clear =e-actions-invalid-arguments=
  errors.
- =chat-session :compact= remains callable through =e-actions-call= from
  context-bound Elisp.
- Legacy direct function action callers are gone; all execution routes through
  work-backed descriptors.
- Runtime docs and =run_elisp= guidance show the aligned tool/action APIs.
