* Design self-check

1. If this stopped here, the project is better off: agents have a stable,
   documented action API and do not need to rediscover calling conventions from
   examples.
2. This moves toward capability-owned semantic operations with the harness
   injecting runtime context.
3. Stable behavior is modeled directly.  The uncertain parts stay local in
   descriptor fields such as =:caller= instead of leaking per-action
   conventions to callers.
4. Capabilities still own their actions.  The dispatcher owns lookup,
   context-injection, argument validation, and common errors.
5. Coupling is reduced for callers.  Interfaces stay narrow:
   capability/action/arguments/options for Elisp.
6. Dependency direction remains correct: generic dispatch depends on stable
   capability and harness contracts, not on concrete shells.
7. Side effects remain in capability handlers and harness/session services; the
   dispatcher only resolves and invokes them.
8. The descriptor abstraction is semantically real because every implementation
   can expose the same handler/caller/schema contract without unsupported
   operations.
9. Compatibility is explicit: old plain function actions still work and direct
   =e-capabilities-action= callers still receive functions.
10. No legacy code is removable yet because shell/tests still intentionally use
    direct action functions.
11. Expected errors surface as typed =e-actions-*= conditions; unexpected
    handler errors are not swallowed by the dispatcher.
12. Focused tests prove dispatcher behavior independently of full runtime, and
    harness/tool tests prove it remains aligned with nested =e-tools-call=
    behavior.
