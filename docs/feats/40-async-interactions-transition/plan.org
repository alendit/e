#+title: Async Interactions Transition
#+startup: overview

* Status

Planned.

* Problem

=e= already has an async turn spine for normal provider and top-level tool
execution, but the runtime still has many user-visible paths that can block
Emacs. Some are explicit waits around network/process/helper work. Others are
CPU-heavy presentation or context work that runs on the UI thread.

The result is that a turn can be architecturally "async" and still feel frozen
when the shell renders progress, materializes a tool result, computes status,
resumes a large session, or waits for a sync tool handler.

This plan turns the research in =docs/research/nonblocking-interactions.org=
into a complete implementation track for removing slow synchronous work from
interactive paths.

* Goal

Make all slow interactions non-blocking in practice:

- no interactive command waits for provider, MCP, web, browser, subprocess, or
  helper completion before returning;
- no long-running tool is exposed only as a synchronous =:handler=;
- no network, helper, subprocess, recursive filesystem search, large transcript
  replay, full Markdown fontification, or heavy context/status computation runs
  unbounded on the Emacs command/UI thread;
- every long operation has a lifecycle visible to the harness and presentation:
  start, progress when useful, finish, failure, and cancellation;
- compatibility sync wrappers remain batch/test conveniences only and are
  guarded away from normal chat shells.

* Non-goals

- Do not make Emacs Lisp itself preemptively threaded. Heavy local computation
  should be made cooperative, cached, or chunked with timers/request handles.
- Do not change model/provider semantics unless required to make an existing
  blocking lifecycle async.
- Do not rewrite the chat UI wholesale. Presentation work should move behind a
  render scheduler and incremental update points.
- Do not remove sync wrappers immediately if tests or batch callers still need
  them. First make interactive misuse detectable.
- Do not solve all performance work in this feature. Feature 39 remains the
  narrow chat hot-path optimization track; this feature owns the complete
  non-blocking contract and migration.

* Current State

Already acceptable:

- =e-chat-session-submit= -> =e-harness-prompt-async= -> =e-loop-start-turn= is
  async in the normal chat path.
- OpenAI and Anthropic ordinary =:start= paths use async request handles.
- =bash= is an async process tool with progress and cancellation.
- Composer =!= command insertion already uses an async process start.

Still blocking or potentially blocking:

- compaction calls =e-harness-compact-session= synchronously and can wait on a
  backend summary request;
- MCP, web search/fetch/browser, and some resource tools still use sync
  handlers or sync helper/HTTP/subprocess waits;
- =file://= and =tmp://= glob/search methods use synchronous =process-file=;
- nested =e-tools-call= waits for the nested tool by contract;
- running-status redraw, final assistant Markdown presentation, tool-result
  previewing, block details, and full session render are synchronous buffer work;
- context/status paths can build full context, run providers, estimate tokens,
  or query Anthropic model info synchronously;
- session append/load/replay and overview read state are synchronous disk I/O;
- composer =@= candidates and file insertion do synchronous discovery/reads.

* Design Direction

The target contract is not "no synchronous code." The target is:

1. Cheap synchronous snapshot reads are allowed.
2. Slow or unbounded interactions must be represented as async requests.
3. CPU-heavy local work must run in bounded chunks or against a cached snapshot.
4. UI rendering must be scheduled and budgeted like runtime work, not treated as
   incidental string insertion.

The stable boundary is a request lifecycle:

- =:start= creates work and returns a cancellable handle immediately.
- callbacks/events report =started=, =progress=, =finished=, =failed=, and
  =cancelled= states.
- the harness owns turn/session lifecycle and durable activity.
- presentation shells observe lifecycle events and schedule bounded rendering.
- compatibility wrappers may wait only in tests, batch commands, or explicitly
  non-interactive helper code.

* Ownership

| Component | Responsibility |
|-----------+----------------|
| =e-tools= | Generic async tool contract, sync-handler guardrails, cancellation/result shape |
| =e-harness= | Turn lifecycle, compaction lifecycle, nested tool policy, event emission |
| =e-loop= | Provider/tool continuation without blocking waits |
| =e-mcp= | Async stdio/HTTP MCP request dispatch and cancellation |
| =e-web-tools= | Async web fetch/search/browser helper operations |
| =e-resources= / resource methods | Async resource operation dispatch or cached snapshots |
| =e-chat= | Render scheduler, bounded presentation jobs, composer/nav preservation |
| =e-context-status= / =e-context-budget= | Cached status snapshots and cheap formatting |
| =e-capabilities= | Cheap context-provider contract and cache-facing providers |
| =e-session= | Durable write queue and chunked replay/loading |
| provider adapters | Async transport, cancellable request handles, async metadata prefetch |

Dependency direction should stay from shells/harness toward core contracts and
adapters. Core policy must not depend on concrete presentation details.

* Implementation Slices

** Slice 1: Instrumentation and blocking guardrails

Purpose: make blocking paths measurable before refactoring them.

Files likely touched:

- =lisp/core/e-tools.el=
- =lisp/core/e-harness.el=
- =lisp/shells/chat/e-chat.el=
- =lisp/shells/e-context-status.el=
- =test/*=

Work:

1. Add a small "interactive hot path" guard used by tests/profile hooks to
   detect calls to known blocking primitives from chat commands, turn startup,
   tool execution, and status/render paths.
2. Add profile spans for:
   - tool handler execution vs async start;
   - compaction prepare/start/finish;
   - final assistant insert and Markdown presentation;
   - running activity redraw;
   - tool-result preview materialization;
   - session render/replay;
   - context/status estimate work.
3. Add render-budget warnings that report when a presentation job exceeds the
   configured foreground budget.
4. Add regression tests proving ordinary chat submit does not call sync backend
   wrappers.

Acceptance:

- A focused test fails if an interactive path calls
  =url-retrieve-synchronously=, unbounded =process-file=, or a known
  =accept-process-output= wait wrapper.
- Profiles can distinguish provider wait, tool wait, context prep, and chat
  rendering cost.
- No prompt text, response text, tool output body, or full context body is
  recorded in profile metadata.

* Cross-Slice Acceptance Criteria

- Commands return promptly even when provider, MCP, web, browser, shell,
  resource search, compaction, or render work continues.
- Cancellation settles every active request type and leaves no stuck turn/tool
  state.
- Progress or lifecycle feedback is visible for long operations.
- Chat drafts, references, point, navigation state, block view, and tool-list
  state survive async rendering.
- Status and mode-line updates are cache reads, not hidden context builds.
- Large outputs and transcripts degrade into previews, references, or lazy
  chunks instead of UI freezes.
- Tests cover the core contracts independently of live providers.

* Risks and Constraints

- Emacs is single-threaded for normal UI work. "Async" CPU work must be
  cooperative and budgeted.
- Request state machines can race with cancellation, buffer kill, session reset,
  and live reload. Each async request needs explicit settlement semantics.
- Progressive rendering can reorder visible UI if not tied to stable block ids
  and generation counters.
- Cached context/status snapshots can become stale. Staleness must be visible or
  safe, and correctness-critical turn context must still be rebuilt when needed.
- Sync wrappers are useful for tests and batch commands. Removing them too early
  would make verification harder.
- Nested tool async semantics may require a larger =run_elisp= contract change;
  the fallback is to forbid long nested calls clearly.

* Compatibility Expectations

- Existing provider =:start= contracts remain valid.
- Existing tool names and model-facing result shapes remain stable where
  possible.
- Existing sessions and compaction records remain readable.
- Sync wrappers stay available for explicit compatibility/test use until a
  separate cleanup proves they are obsolete.
- User-visible chat content should not change except for better progress,
  previews, and visibly progressive rendering.

* Architecture Self-check

1. If this work package stopped here, would the project be better off than if it
   had not been built?
   - Yes. The plan defines the migration boundary and lets each slice remove a
     concrete class of UI freezes without waiting for the whole transition.
2. What final direction does this move toward?
   - A runtime where harness lifecycle, tool execution, resource work, context
     refresh, persistence, and presentation all use explicit non-blocking
     contracts.
3. Which decisions are likely to change, uncertain, or stable, and are
   uncertain ones local and reversible instead of hidden behind speculative
   abstraction?
   - Stable: long work must not block interactive paths. Likely to change:
     render budgets, cache invalidation details, and nested async syntax.
     Uncertain choices are isolated in scheduler, cache, and nested-call
     contracts.
4. What component owns this behavior, and why?
   - Each existing owner keeps its domain: tools own tool contracts, harness
     owns lifecycle, chat owns rendering, resources own resource operations,
     context modules own snapshots, and session owns persistence.
5. Does this change increase or reduce coupling?
   - It should reduce coupling by replacing ad hoc waits with request handles
     and events. The risk is accidental coupling through shared scheduler state;
     keep scheduler APIs narrow and consumer-shaped.
6. Did any dependency start pointing the wrong way?
   - The plan forbids core runtime code from depending on concrete chat
     rendering. Shells observe events; core emits lifecycle.
7. Could any side effect be moved outward into an adapter or shell?
   - Network/helper/browser side effects belong in adapters/tools. Presentation
     side effects belong in =e-chat=. Core policy should only decide and emit.
8. Are the abstractions and interfaces semantically real?
   - Yes: async request lifecycle, cheap snapshot provider, render job, and
     durable session write are real contracts with distinct owners.
9. What compatibility expectations apply?
   - Existing names/results/sessions stay readable. Sync wrappers are retained
     until interactive callers are migrated and tests can prove no dependency.
10. What legacy code can we remove now?
    - None in the planning slice. During implementation, remove obsolete sync
      fallback paths once each async replacement is covered.
11. Where are expected errors handled, and where do unexpected errors surface?
    - Expected cancellation/timeouts/tool failures settle through lifecycle
      events. Unexpected coding errors should surface through existing error
      paths and not be swallowed by schedulers.
12. What tests prove the core behavior independently of the full system?
    - Tool, harness, resource, context, session, and chat scheduler tests use
      fake backends/helpers and synthetic data; live probes only validate the
      integrated user experience after reload.
