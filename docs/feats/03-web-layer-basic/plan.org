#+author: planning
#+date: 2026-05-23

* Status
Implementation plan derived from [[file:sketch.org][sketch.org]]. Keep this plan direct: it is for a coding agent, not a spec ceremony.

* Shape
Build a =web= layer with three semantic capabilities:

- search: use =bx web= behind a tool, not shell
- fetch: passive HTTP read + text extraction
- browser: Playwright-backed rendered/interactive page control

Do not dump every operation into the always-visible instructions. The active prompt should say roughly:

#+begin_quote
Web access is available. Use search for queries, fetch for passive page reads, and browser for rendered/interactive pages. Read =e://web/refs/overview.md= for details when needed.
#+end_quote

Detailed tool names, argument shapes, examples, and boundaries should live in read-only =e://web/refs/...= resources for progressive discovery.

* Agent-facing discovery resources
Implement the web capability with compact instructions plus resources:

- =e://web/refs/overview.md= -- when to use search vs fetch vs browser
- =e://web/refs/search.md= -- =web_search= usage, =bx= notes, examples
- =e://web/refs/fetch.md= -- =web_fetch= usage, output shape, limitations
- =e://web/refs/browser.md= -- browser operations, session model, examples
- =e://web/refs/boundaries.md= -- what the MVP intentionally does not implement

Use =e-capability-with-skills-create= or plain =:resources= providers, whichever fits best. The important part is that only the short overview is pushed up front.

* Files to add
- =lisp/layers/web/e-web.el= -- layer constructor + short instructions
- =lisp/layers/web/e-web-capabilities.el= -- capability constructors/resources
- =lisp/layers/web/e-web-tools.el= -- tool registration + handlers
- =test/e-web-capabilities-test.el=
- =test/e-web-tools-test.el=

Split later only if needed.

* Tool naming
Logical names in logs/docs/resources can be dotted:

- =web.search=
- =web.fetch=
- =web.browser.*=

Provider-facing function names should be conservative aliases:

- =web_search=
- =web_fetch=
- =web_browser_open=, =web_browser_observe=, etc.

Do not put the full browser operation list in the default instruction text. Put it in =e://web/refs/browser.md=.

* Common result shape
Keep tool results easy to inspect and debug:

- logical capability id, e.g. =web.search=
- backend, e.g. =bx=, =http=, =playwright=
- requested URL/query/session where relevant
- normalized result data
- diagnostics for backend failures or truncation

Do not design a permission system here. Just preserve enough context that later logging/policy work can classify what happened.

* Step 1: skeleton
Create the layer and register placeholder tools/resources.

Acceptance:
- =(require 'e-web)= works
- =(e-web-layer-create)= returns a layer
- registering the layer yields the expected provider-safe tool names
- the generated instructions are short and point to =e://web/refs/overview.md=
- =read= can load the web reference resources

* Step 2: search
Implement =web_search= as a direct =bx web= wrapper. Do not route through shell.

Initial model-facing args:

#+begin_src json
{
  "query": "string",
  "count": 5,
  "freshness": "optional bx freshness",
  "include_site": ["example.com"],
  "exclude_site": ["example.com"],
  "include_raw": false
}
#+end_src

Implementation notes:
- configurable =e-web-bx-program=
- build argv list directly
- parse JSON stdout
- normalize =.web.results[]= into ={rank,title,url,snippet?,source?,date?}=
- include raw JSON only when requested
- return useful errors for missing =bx=, non-zero exit, timeout, invalid JSON

Tests use a fake =bx= executable. No real network in unit tests.

* Step 3: fetch
Implement =web_fetch= as passive HTTP retrieval and text extraction.

Initial args:

#+begin_src json
{
  "url": "https://example.com/page",
  "format": "text|markdown|both|html",
  "include_links": true,
  "include_html": false,
  "max_chars": 50000,
  "timeout": 20
}
#+end_src

Implementation notes:
- allow only =http= and =https=
- use Emacs =url= APIs directly
- capture final URL, status, content type, headers
- decode text sensibly, default UTF-8
- for HTML: title, readable text, links, simple Markdown if requested
- reject obvious binary content in MVP
- no browser rendering here yet; tell the agent to use browser for rendered pages

Tests should use mocked/local HTTP responses.

* Step 4: browser
Implement browser support with a small Node/Playwright helper process. Emacs talks to it over NDJSON.

Why helper process:
- keeps Playwright/Node concerns out of Elisp
- easier to fake in tests
- allows a stable Emacs-side protocol even if backend changes later

Minimum operations for MVP live in =e://web/refs/browser.md= and are registered as provider-safe tools:

- open/navigate: browser load
- observe/screenshot: read-only
- click/type/press: browser mutation
- close: session control

Implementation notes:
- default to isolated ephemeral browser context
- start helper lazily
- pending request table keyed by id
- parse stdout NDJSON, capture stderr diagnostics
- return page state with URL, title, text/accessibility summary, element handles where practical
- screenshots should be artifact/temp-file references, not giant chat blobs unless explicitly needed
- fake helper tests first; real Playwright tests behind an env flag

* Step 5: boundaries
Keep the first browser implementation narrow:

- default to an isolated ephemeral browser context
- do not implement downloads, uploads, clipboard, geolocation, camera/microphone, or notifications yet
- if a page or operation needs something outside the implemented surface, return a clear unsupported error

This is not a permissions design. It is just keeping the MVP small.

* Step 6: docs and examples
Keep examples in resources, not the default prompt.

Useful examples:
- search then fetch a result before citing it
- fetch a static article
- browser-open a JS-heavy page, observe, click, screenshot, close
- when not to use browser because passive fetch is enough

* Test checklist
- web layer registers tools and resources
- default instructions stay compact
- resource reads return overview/search/fetch/browser/boundaries docs
- fake =bx= success and failure
- fetch HTML/text/truncation/unsupported scheme
- fake Playwright helper open/observe/click/type/screenshot/close
- unavailable backend errors are understandable
- unsupported browser options return clear errors

* Milestones
1. Layer skeleton + resources.
2. =web_search= with fake =bx= tests.
3. =web_fetch= with mocked/local HTTP tests.
4. Browser helper protocol + fake helper tests.
5. Optional real Playwright integration.
6. Boundary docs and cleanup.
