** Finding 1: Concrete file roots were passed to =fd=

*** Problem

=file://= and =tmp://= concrete file roots could fail because the glob
implementation passed a file path to =fd= as though it were a searchable
directory.

*** Change

Added concrete-resource handling for =file://= and =tmp://= glob roots.  A
regular file root now matches directly against the requested facade pattern and
returns that single resource when appropriate.

** Finding 2: Backend glob syntax leaked through =fd=

*** Problem

The addendum requires a portable facade glob syntax.  Passing facade patterns
directly to =fd --glob= leaked fd-only syntax.  For example,
=literal[abc].txt= was interpreted as a character class rather than a literal
resource name.

*** Change

Added a shared fd translation helper for facade glob syntax.  The translator
keeps facade =*= and =**= semantics, escapes backend-only fd glob metacharacters,
and still post-filters returned names through the facade matcher.

*** Evidence

Focused and full feature 25 tests passed:

#+begin_example

151 expected, 0 unexpected
#+end_example

Live probe for =literal[abc].txt= returned exactly the literal file for both
=file://docs= and =tmp://docs= with =:truncated nil=.

** Finding 3: Bounded fd enumeration could drop facade matches

*** Problem

Using =fd --max-results= before facade post-filtering was unsafe when fd's
prefilter was broader than the facade.  A facade pattern such as =*.txt= should
only match direct children, but recursive fd enumeration could return nested
false positives first and consume the bounded result window before the direct
match was seen.

*** Change

Added a facade max-depth helper.  For glob patterns without =**=, =file://= and
=tmp://= now pass =fd --max-depth= equal to the facade pattern segment count.
Patterns containing =**= remain depth-unbounded.  The implementations continue
to use =--max-results limit+1= and facade post-filtering.
