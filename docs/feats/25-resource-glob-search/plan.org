#+title: Resource Glob and Search Tools
#+startup: overview

* Status

In-review

* Problem

=e= has abstract resource operations for =read=, =write=, and =edit=.  Active
capabilities register per-scheme resource methods, and the harness turns those
methods into model-facing tools only when a session has matching capabilities.

Agents still need to discover resources before reading them.  Today that
discovery is fragmented:

- workspace files can be found with shell commands such as =fd= and =rg=;
- live Emacs buffers can be listed with the separate =list_buffers= tool;
- =tmp://= session outputs are readable once a URI is known, but there is no
  resource-native discovery tool;
- =e://= capability resources can be read when a URI is already present in
  guidance, but cannot be listed or searched through the resource operation
  abstraction.

This pushes agents toward ad hoc shell commands and scheme-specific tools even
when they are working with URI-addressed resources.

* Goal

Add two generic resource operation tools:

- =glob= :: list resources in an active URI scheme by URI root and name pattern.
- =search= :: search text content in resources under an active URI scheme.

These tools should be abstract in the same way =read=, =write=, and =edit= are
abstract: =e-operations= owns the shared model-facing contract,
=e-resources= dispatches by URI scheme and operation, capabilities register
scheme-specific methods, and the harness generates model-facing tools from the
active resource registry.

The first feature slice must include implementations for:

- =file://= workspace resources using =fd= for listing and =rg= for content
  search;
- =buffer://= live Emacs buffers using Emacs-native buffer enumeration and
  buffer search;
- =e://= in-memory capability resources using Emacs-native store enumeration and
  string search;
- =tmp://= session-scoped temporary resources using =fd= and =rg= inside the
  session tmp root.

* Non-goals

- Do not add standalone =file_glob=, =grep=, =search_buffers=, or =search_tmp=
  tools as the primary model-facing API.
- Do not route =buffer://= or =e://= search through shell commands.
- Do not make =e-harness= aware of filesystem, buffer, store, or tmp search
  details beyond generic resource operation registration.
- Do not search binary file contents in the first slice.
- Do not add cross-scheme search that fans out across every active scheme in one
  call.  The caller chooses a URI root such as =file://=, =buffer://=, =e://=,
  or =tmp://=.
- Do not implement replacement, indexing, async streaming search, TRAMP search,
  or external search backends.

* User-facing behavior

When a session has file inspection active, the model sees =glob= and =search=
with =file://= listed as an active scheme.  Example calls:

#+begin_src elisp
(:name "glob"
 :arguments (:uri "file://"
             :pattern "*.el"
             :limit 50))

(:name "search"
 :arguments (:uri "file://lisp"
             :query "e-resource-method-create"
             :glob "*.el"
             :limit 20))
#+end_src

When the Emacs layer is active, the same tools expose =buffer://=.  Example
calls:

#+begin_src elisp
(:name "glob"
 :arguments (:uri "buffer://"
             :pattern "*e-*"
             :limit 25))

(:name "search"
 :arguments (:uri "buffer://"
             :query "e-chat-session-id"
             :limit 20))
#+end_src

When harness-base session tmp resources are active, =tmp://= can be listed and
searched:

#+begin_src elisp
(:name "glob"
 :arguments (:uri "tmp://tool-results"
             :pattern "*.txt"
             :limit 50))

(:name "search"
 :arguments (:uri "tmp://"
             :query "Command exited"
             :glob "*.txt"
             :limit 20))
#+end_src

When capability resources populate the in-memory store, =e://= can be listed and
searched:

#+begin_src elisp
(:name "glob"
 :arguments (:uri "e://"
             :pattern "*/skills/*"
             :limit 50))

(:name "search"
 :arguments (:uri "e://"
             :query "frontmatter"
             :glob "*/skills/*"
             :limit 20))
#+end_src

Tool descriptions must enumerate active URI schemes and scheme-specific URI
patterns just as =read=, =write=, and =edit= do.

* Operation contracts

Add two new standard operations in =lisp/core/e-operations.el=.

** =glob=

=glob= lists resource URIs under a URI root.  The result is a structured plist
with a =:resources= vector/list of entries and a =:truncated= flag.

Parameters:

- =:uri=, required string.  A URI root or scope inside one scheme, for example
  =file://=, =file://lisp=, =buffer://=, =e://base-guidance=, or
  =tmp://tool-results=.
- =:pattern=, optional string.  A resource-name glob applied relative to the
  root using the portable facade glob syntax from =addendum.org=.  Default is
  =*=.
- =:case-sensitive=, optional boolean.  Default t means case-sensitive glob
  matching.
- =:limit=, optional positive number.  Default should be modest, initially 100.

Result entry shape:

#+begin_src elisp
(:uri "file://lisp/core/e-resources.el"
 :name "lisp/core/e-resources.el"
 :kind file
 :metadata (:bytes 4821))
#+end_src

Scheme methods may omit expensive metadata.  The shared contract should require
=:uri= and =:name= only.

** =search=

=search= searches text content in resources under a URI root.  The result is a
structured plist with =:matches= and =:truncated=.

Parameters:

- =:uri=, required string.  A URI root or scope inside one scheme.
- =:query=, required string.  Facade search query.  Literal characters match
  literally; =*= inside a query part is the first-slice wildcard.
- =:glob=, optional string.  Resource-name facade glob filter relative to the
  root.
- =:case-sensitive=, optional boolean.  Default nil means case-insensitive
  search.
- =:whole-word=, optional boolean.  Default nil.
- =:multiline=, optional boolean.  Default nil; when non-nil whitespace gaps may
  cross line boundaries.
- =:limit=, optional positive number.  Default should be modest, initially 100.

Result match shape:

#+begin_src elisp
(:uri "file://lisp/core/e-resources.el"
 :line 71
 :column 1
 :text "(defun e-resources-register (registry method-or-register)")
#+end_src

For non-line-oriented resources, =:line= and =:column= are best-effort.  Text
schemes in this feature all support line and column reporting.

* Architecture

** Core operation layer

Extend =e-operations-standard= to include =e-operation-glob= and
=e-operation-search=.  Keep this as the only place that owns JSON schema,
tool names, and the shared contract text.

Add convenience dispatchers in =e-resources.el=:

#+begin_src elisp
(e-resources-glob registry uri pattern &optional limit case-sensitive)
(e-resources-search registry uri query &optional options)
#+end_src

The dispatch should still flow through =e-resources-call= so unknown-scheme and
unsupported-operation errors stay consistent with =read=, =write=, and =edit=.

** Resource method metadata

Reuse =e-resource-method= for the new operations.  Its existing =:description=,
=:uri-patterns=, =:range-modes=, =:examples=, and =:metadata= fields are enough
for tool descriptions.  Do not add a separate resource-discovery registry.

Method descriptions should use these URI patterns:

- =file://<path-or-directory>= for file glob/search.
- =buffer://<optional-buffer-name-prefix>= for live buffer glob/search.
- =e://<capability>/<path>= for store glob/search.
- =tmp://<relative-path-or-directory>= for session tmp glob/search.

** Harness-generated tools

=e-harness--register-resource-operation-tools= should pick up the new operations
automatically once methods are registered because it already iterates
=e-resources-operations=.  Update tests to prove =glob= and =search= are
generated and dispatch through active methods.

Resource usage metadata for =glob= and =search= should use operation ids
=glob= and =search=.  For =search=, include the searched URI root.  Do not try
to record every matched resource in tool metadata in the first slice; the result
already carries exact match URIs.

* Scheme implementations

** =file://= workspace resources

Owner: =lisp/layers/base/e-base-tools.el= and
=lisp/layers/base/e-file-capabilities.el=.

Listing uses =fd=:

- Resolve the URI address through =e-base-tools--resource-path= so all paths stay
  inside the configured root.
- Treat =file://= as the configured root.
- Treat =file://dir= as a directory scope when it resolves to a directory.
- For file roots, list that file when it matches the pattern.
- Invoke =fd= without shell interpolation by using =process-file= or
  =make-process= with an argv list.
- Prefer =fd=, accept =fdfind= if that is the installed executable name.
- Validate the facade glob pattern before invoking external commands.
- Use =fd= as an implementation engine, then post-filter names with the shared
  facade glob matcher so backend glob extensions do not leak into the contract.
- Use =--glob=, =--color= =never=, and a bounded result count.  Include hidden
  files only if a later config option asks for that; the first slice should use
  =fd='s default visibility.

Search uses =rg=:

- Resolve the URI address to a directory or file scope inside the configured
  root.
- Invoke =rg= without shell interpolation.
- Use =--json= and parse JSON result objects rather than parsing human text.
- Compile the facade query to an =rg=-compatible regexp and escape all backend
  regexp metacharacters from literal query text.
- Validate any facade =:glob= filter before invoking =rg= and post-filter
  matches by resource name.
- Use =--line-number=, =--column=, and =--color= =never=.
- Use =--ignore-case= by default and omit it when =:case-sensitive= is non-nil.
- Treat exit code 1 as no matches; other non-zero exits are tool errors.
- Return matched URIs as =file://= paths relative to the configured root.

Missing =fd= or =rg= should fail clearly with an actionable error naming the
missing executable.

** =tmp://= session resources

Owner: =lisp/layers/harness/e-session-tmp-resources.el=.

Tmp listing/search should mirror =file://= behavior but root every command in
the session tmp directory returned by =e-session-tmp-directory=.

Path safety stays owned by =e-session-tmp--safe-relative-name= and
=e-session-tmp--path=.  The methods must reject unsafe roots before invoking
external commands.

Returned URIs must be =tmp://<relative-path>=, never absolute filesystem paths.

** =buffer://= live buffers

Owner: =lisp/layers/emacs/e-emacs-tools.el= and
=lisp/layers/emacs/e-emacs-capabilities.el=.

Listing should use Emacs-native buffer enumeration:

- enumerate =(buffer-list)=;
- filter exact URI root/prefix when the caller passes =buffer://name-prefix=;
- filter names relative to the URI root with the shared facade glob matcher;
- return buffer metadata already available through
  =e-emacs-tools--buffer-metadata= where useful.

Search should use Emacs-native buffer search:

- search only live buffers selected by the URI root and optional facade =:glob=;
- compile the facade query to an Emacs regexp before using =re-search-forward=;
- bind =case-fold-search= based on =:case-sensitive=;
- return =buffer://<buffer-name>=, line number, column, and the matched line text;
- skip binary/special handling in the first slice, because Emacs buffers are text
  objects for this resource contract.

The existing =list_buffers= tool may remain for explicit Emacs awareness, but
the model should be able to discover buffers through =glob= when =buffer://= is
active.

** =e://= in-memory resources

Owner: =lisp/core/e-store.el=.

Store resources already have =e-store-list= and =e-store-read=.  Add
=e-store-glob-resource-method= and =e-store-search-resource-method=, or replace
=e-store-resource-method= with a function that registers all read/glob/search
methods.

Listing:

- iterate =e-store-list=;
- filter by URI root and glob pattern;
- return =e://= URIs and names relative to =e://=.

Search:

- read each matching entry with =e-store-read-entry=;
- use Emacs-native string/buffer search;
- pass nil range so dynamic readers provide full searchable content;
- return =e://= URIs, line, column, and matched line text.

Keep =e://= read-only for mutation.  This feature adds discovery/search only,
not =write= or =edit=.

* Acceptance criteria

- =glob= and =search= are standard resource operations beside =read=, =write=,
  and =edit=.
- The harness generates =glob= and =search= tools only when at least one active
  resource method supports each operation.
- Tool descriptions list active URI schemes and scheme-specific patterns.
- =file://= glob uses =fd= and =file://= search uses =rg=.
- =tmp://= glob uses =fd= and =tmp://= search uses =rg= inside the session tmp
  root.
- =buffer://= glob/search uses Emacs-native live buffer enumeration/search.
- =e://= glob/search uses Emacs-native store enumeration/search.
- =e://= remains read-only for mutation.
- Results are bounded by =:limit= and report truncation.
- URI roots cannot escape configured file or tmp roots.
- Missing =fd= or =rg= failures are explicit and actionable.
- Focused tests, full tests, live reload, and one bounded live probe are
  completed before marking implementation ready.

* Risks and constraints

- =fd= and =rg= availability varies by environment.  The feature should fail
  clearly for missing executables rather than silently falling back to slower or
  semantically different shell commands.
- Facade pattern compilation is now a compatibility boundary.  The shared helper
  must reject unsupported public syntax and compile only facade wildcards,
  whitespace gaps, and mode options into backend-specific regexps.
- Search result volume can grow quickly.  Keep the required =:limit= behavior
  central in each method and include truncation metadata.
- File/buffer coherence still matters.  =file://= search should search disk or
  the filesystem view in the first slice; it should not try to merge unsaved live
  buffer contents unless resource coherence grows a specific search policy later.
- Tmp resources are session-scoped.  Methods must require harness/session
  context and fail clearly if called without it.

* Architecture self-check

1. If this work stopped after the core operation slice, the project would still
   be better off because resource discovery becomes a real extension point
   instead of an ad hoc shell habit.
2. This moves toward the final direction where capabilities expose resource
   behavior through narrow contracts and the harness remains presentation- and
   scheme-agnostic.
3. Stable decisions: =glob= and =search= are resource operations with a portable
   facade pattern language.  Likely-to-change decisions: exact result metadata
   and future pattern-language extensions.  Those stay in operation docs and the
   shared pattern helper rather than in harness policy.
4. =e-operations= owns model-facing contracts, =e-resources= owns dispatch,
   scheme adapters own side effects, and capabilities own exposure.
5. Coupling is reduced by replacing scheme-specific discovery tools with the
   same resource-method registration path used by existing operations.
6. Dependencies keep pointing from adapters/capabilities into core contracts,
   not from the harness into fd, rg, buffers, tmp, or store internals.
7. Side effects stay in scheme adapters: process execution for file/tmp and
   buffer/store traversal for Emacs-native resources.
8. The abstractions are semantically real because every listed scheme can list
   resources and search content without weakening the contract.
9. Existing read/write/edit compatibility remains intact.  =list_buffers= stays
   available as an Emacs-specific awareness tool.
10. No legacy code should be removed in the first slice.  A later cleanup may
    remove redundant guidance that tells agents to shell out for discovery.
11. Expected errors include unsupported scheme operation, unsafe roots, no
    matches, and missing executables.  Unexpected process and parse failures
    should surface as tool errors.
12. Core tests prove operation dispatch independently of fd, rg, buffers, tmp,
    or store.  Scheme tests prove each implementation without requiring
    unrelated presentation shell rewrites.
