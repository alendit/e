#+title: Session Resource
#+startup: overview

* Status

Done

* Goal

Expose persisted agent sessions as a read-only resource scheme for agents and host code.

=e= sessions are always available through the built-in =e= engine.
Additional local or remote engines can be configured explicitly.

The first slice should let an agent discover prior sessions across enabled engines, inspect metadata, read a transcript or entry stream, and search session text through the existing resource operation tools.

* Problem

=e= has durable session storage, and other installed agent engines such as Codex can have their own durable session stores.
The agent-facing surface is still tied to the current =e= session.

Today an agent can:

- inspect the current session context through actions such as =chat-session/context=, which calls =e-harness-context= for a preview of the current session, or =context-inspection/export-context=, which writes the assembled LLM context to a URI;
- search visible chat buffers through =buffer://= when those buffers are live;
- inspect persisted =e= session files only by knowing implementation paths and using file tools.

That is not a semantic session API.
It also pushes callers toward private filesystem layouts for each engine.
For non-=e= engines, the problem is worse because there is no shared discovery surface at all.

A session is better modeled as a resource than as a new action family.
The caller wants to list, read, and search addressable material.
There is no clear write or edit meaning yet, so the first version should be read-only.

* Resource scheme

Add a =session://= scheme backed by a small engine registry.
The built-in =e= engine is always present and uses the active harness's session store.
Additional engines can be registered through a configuration option.

Initial URI namespace:

- =session://= is a glob root that lists enabled session engines.
- =session://<engine-id>/sessions/= is a glob root that lists sessions for one enabled engine.
- =session://<engine-id>/sessions/<session-id>/= is a glob root that lists readable projections for one session.
- =session://<engine-id>/sessions/<session-id>/<projection>= reads one concrete projection.

The agent-facing rule should be simple: glob first, then read returned leaf URIs.
Agents should not have to synthesize or memorize the full path grammar.
The generated tool description should present this as a three-step workflow:

1. Glob =session://= to discover engines.
2. Glob =session://<engine-id>/sessions/= to discover sessions.
3. Glob =session://<engine-id>/sessions/<session-id>/= to discover readable projections, then read one returned URI.

Initial projection names can include:

- =summary= for one session summary and transcript overview;
- =messages= for normalized messages in insertion order;
- =activity= for durable activity events when supported;
- =events= for session events when supported;
- =compactions= for compaction records when supported;
- =provider-anchors= for provider anchor records when supported.

Keep directory-like URIs for =glob= only.
Do not make =session://=, =session://<engine-id>/sessions/=, or =session://<engine-id>/sessions/<session-id>/= readable.
=read= should only work on leaf projection URIs returned by =glob= or documented as stable examples.

The exact text format can be Org or plain text, but it must be stable enough for agent reading and search snippets.
Prefer a human-readable text projection over raw JSON for the first slice.
Raw JSON can be a later explicit variant if needed.

* Engine configuration

=e= is always registered as the built-in engine with id =e=.
Additional engines are opt-in through a user option such as =e-session-resource-engines=.
Each configured engine entry should declare a stable engine id, a display name, and an adapter that can list sessions and render supported subresources.
Adapters can read local stores or remote APIs, but remote adapters must be explicit because they may send credentials or queries over the network.

The first adapter contract should stay narrow:

- list sessions;
- render one session summary;
- render messages or transcript text;
- report which optional subresources are supported;
- search rendered text, or allow the generic resource method to search the rendered projection.

Codex can be a later configured engine once its local or remote session store shape is understood.

* Resource operations

** Read

Register a =read= resource method for =session://=.

The method should:

- resolve the engine id first;
- resolve =session://e/sessions/<session-id>/...= through the active harness's =e-harness-sessions= store;
- use public =e-session= accessors for =e= sessions instead of reading JSONL files directly;
- call configured engine adapters for non-=e= engines instead of assuming filesystem paths;
- preserve lazy loading for persistent =e= sessions by calling accessors such as =e-session-get= and =e-session-messages=;
- support existing read ranges only where they make sense for the rendered text;
- fail clearly for unknown engines, unknown session ids, and unsupported subpaths.

** Glob

Register a =glob= resource method for =session://=.

Listing =session://= should return compact entries for enabled engines.
Listing =session://e/sessions/= should return compact entries for =e= sessions sorted by =e-session-list=, newest first.
Listing a configured non-=e= engine should use that engine adapter's list operation.

Entries should include:

- =:uri=, e.g. =session://e/sessions/01K.../summary= for a readable projection or =session://e/sessions/01K.../= for a glob root;
- =:name=, preferably the display title or session id;
- =:kind=, e.g. =session=;
- =:metadata= with stable fields such as =:engine-id=, =:id=, =:title=, =:created-at=, =:updated-at=, =:last-message-at=, =:project-root=, and =:harness-instance-id= when present.

A glob pattern should match engine id, session id, and display title.
Subpath listing can be small and explicit: listing =session://<engine-id>/sessions/<session-id>/= may return the known readable child projections above.

** Search

Register a =search= resource method for =session://=.

Search should inspect the text projection for selected sessions and return normal resource match records:

- =:uri= pointing at the closest readable resource, such as =session://<engine-id>/sessions/<id>/messages=;
- =:line= and =:column= in the projection;
- =:text= with the matched line.

The first slice may use Emacs-native string search over rendered session text.
It should search across enabled engines unless a glob or URI narrows the engine.
It should honor the shared facade parameters: =:query=, =:glob=, =:case-sensitive=, =:whole-word=, =:multiline=, and =:limit=.

* Read-only contract

Do not register =write= or =edit= methods for =session://= in this feature.

Session mutation already has explicit semantic operations: submit, steer, reset, rename, compact, metadata updates, and future specialized APIs.
A generic edit against a rendered transcript would be ambiguous and unsafe.

If future work needs mutation, add narrow actions or specialized session operations rather than making =session://= editable by default.

* Architecture

Owner: a harness or runtime self-management capability, not the chat shell.

The resource method needs access to the active harness/session context, but it must not depend on chat buffers or presentation state.
Good candidate owners:

- the existing =e= layer runtime self-management capability; or
- a small dedicated =session-resource= capability enabled by the =e= or =harness-base= layer.

Keep dependencies flowing toward core contracts:

- resource methods depend on =e-resources= and the engine adapter contract;
- the built-in =e= adapter depends on =e-session=;
- formatting helpers can live near the capability or in a small session projection module;
- each engine store remains the source of truth for its own sessions;
- no filesystem path assumptions should leak into callers.

* Privacy and safety

Sessions can contain private data, tool outputs, source snippets, and user text.
The resource is agent-readable only when the layer is active, so the capability description must make that scope explicit.

The first slice should not expose an unbounded global search.
It should search only the built-in =e= store and explicitly configured engines.
Remote engines should be opt-in and should document what endpoint or storage they read.

Add conservative limits for search and read projections.
Large transcripts should be readable by range or by subresource instead of forcing one huge response.

* Resolved questions

All initial review questions now have decisions.

- Summary projection:
  - =session://<engine-id>/sessions/<id>/summary= should render summary metadata, a transcript overview, and links to readable subresources.
  - It should not duplicate the full transcript.
- Search scope:
  - Default search should favor the least surprising transcript search behavior while still giving agents control.
  - In v1, searching a session root should search the =messages= projection by default.
  - Activity, events, compactions, and provider anchors should remain separate searchable projections when explicitly narrowed by URI or glob.
  - If a broad all-projections search becomes useful later, add it as a documented scope rather than hiding it in the default.
- Branch and current-head structure:
  - Do not expose branch or current-head structure in v1.
- Raw projection:
  - A raw JSON projection is useful later.
  - Use =session://<engine-id>/sessions/<id>/raw= as the likely stable name when a consumer needs raw data.
