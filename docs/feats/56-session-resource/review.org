* What looks good

- =session://= is read-only because only =read=, =glob=, and =search= methods are registered.
- The built-in =e= engine uses the active harness session store and public =e-session= accessors.
- Configured engines are opt-in through =e-session-resource-engines= and are accessed only through adapter functions.
- Search defaults to =messages= for session roots, while explicit projection URIs and projection globs can search activity and other optional projections.
- Tool descriptions describe the glob-first workflow instead of requiring agents to memorize the path grammar.

* Architecture review

The owner is =harness-base=, which already owns harness-scoped support resources.
The resource method depends on =e-resources=, =e-session=, and an adapter contract.
It does not depend on chat buffers or presentation shell state.
Configured engines remain explicit because they may read local stores or remote APIs.
Mutation stays outside =session://= and remains owned by semantic session actions or future narrow APIs.

* Design self-check

1. This package leaves the project better off because sessions become discoverable through the existing resource tools instead of private files.
2. It moves toward a semantic session API that can cover =e= and later opt-in engines.
3. The stable decision is the glob-first URI workflow; the uncertain parts are remote engine details and raw JSON, both left outside v1.
4. =harness-base= owns this because the behavior is harness-scoped support, not chat presentation.
5. Coupling is reduced for callers because they depend on resource contracts, not store paths.
6. Dependencies point toward core contracts and configured adapters.
7. Side effects are limited to reading existing session stores or configured adapters.
8. No new durable runtime state is added beyond the user option for configured engines.
9. Search renders text in Emacs and applies limits; no broad unbounded filesystem or network search is introduced.
10. The adapter interface is narrow and substitutable for list and render operations.
11. No legacy path was added.
12. No legacy code was removed.
13. Expected errors surface as specific resource errors; unexpected adapter errors surface to callers.
14. Tests prove discovery, read, search, range, read-only behavior, errors, and configured-engine opt-in behavior.
