* Requirements Checked

- Manual compaction appends an identity-bearing durable compaction record with
  =first-kept-entry-id=, =tokens-before=, =tokens-kept=, metadata, and replay
  support in =e-session-append-compaction= and JSONL replay.
- Context reconstruction uses the latest valid compaction as one backend-facing
  system summary followed by the kept suffix in =e-context--compacted-messages=.
- Historical transcript messages remain persisted; compaction adds a record
  without deleting old messages.
- Repeated compaction includes the previous summary and only newly compacted
  suffix content in =e-compaction-prepare= and =e-compaction-summary-messages=.
- Boundary selection starts the kept suffix at a user message. This conservative
  rule avoids starting inside assistant/tool-call/tool-result groups for the
  supported first-slice transcript shape.
- Tool result text is truncated in summarization input, while affected resource
  metadata is deduplicated into compaction metadata.
- Resource-backed tools record URI/operation metadata, optional free-tool
  metadata can be supplied, and tool result metadata is persisted on tool
  messages.
- Manual orchestration in =e-harness-compact-session= rejects active turns,
  emits lifecycle events, calls the active backend for the summary, and records
  failure events without appending a compaction on expected failures.
- Chat exposes =e-chat-compact-session= and renders compaction lifecycle
  activity.
- Optional compaction instructions are accepted through the command/function
  argument and are included in summary-generation metadata and prompt input.

* Addressed Review Concerns

** Manual instruction prompt

The chat command is callable with an optional =instructions= argument, and the
harness passes those instructions into =e-compaction-prepare=. The summary prompt
includes them under =Additional instructions:= when non-empty, and the durable
compaction metadata stores the same instruction string.

This satisfies the first-slice requirement that optional instructions can be
supplied. A minibuffer prompt on prefix remains a possible UI refinement, but it
is not required for the minimal acceptance criteria because command callers and
shell actions can already pass instructions programmatically.

** Resource usage metadata

The first version intentionally keeps durable resource metadata centered on
=:affected-resources=. Resource tools provide implicit URI/operation metadata;
free tools can accept optional =:resource_usage= metadata; and compaction
preparation deduplicates resources by URI/operation.

The plan's richer suggested fields such as =:tool-usage-summaries=,
=:read-files=, and =:modified-files= are not currently emitted as separate
metadata fields. That is acceptable for this first slice because the minimal
acceptance criteria require captured/deduplicated resource usage metadata, and
=:affected-resources= is the primary durable resource-oriented view. The legacy
file-only compatibility views can be added later if generated summary display or
external tooling needs them.

** Boundary conservatism

Boundary selection uses user messages as safe suffix starts. Given the current
turn representation, this avoids beginning the kept suffix at assistant output,
tool calls, or tool results. The implementation therefore prefers a simple
conservative boundary over a speculative full turn-graph splitter.

A future split-turn compaction feature can refine this, but the current behavior
meets the first manual compaction target.

** Live compaction scenario

After reload, inspected a temporary live harness/session with a fake summary
backend and optional instructions. Manual compaction returned:

#+begin_src emacs-lisp
(:summary "Live summary."
 :instructions "focus on decisions"
 :messages ((:role system :content "Live summary.")
            (:role user :content "new"))
 :history-count 3
 :compactions 1)
#+end_src

This proves the running Emacs uses summary-plus-suffix context, retains full
history, appends exactly one compaction record, and carries optional
instructions into compaction metadata.

* Residual Risks And Follow-ups

- Token counting uses a conservative character-based estimate rather than
  provider tokenizer counts.
- Optional free-tool resource metadata depends on the model/tool caller
  supplying the hint, so resource-backed tools remain the reliable source for
  automatic affected-resource capture.
- The first-slice boundary selector is intentionally conservative and user-entry
  based. Split-turn compaction for unusually large turns remains follow-up work.
- Separate =:tool-usage-summaries= and file-only compatibility metadata can be
  added later if a consumer needs those display forms; the current durable
  metadata already preserves the primary URI/operation resource view.

* Conclusion

Feature 11 is complete for the planned manual context-compaction slice. The
implementation keeps historical storage append-only, reconstructs compacted
context from durable entry ids, surfaces expected failures, and is covered by
batch and live validation.
