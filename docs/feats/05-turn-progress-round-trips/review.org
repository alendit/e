* Implementation Summary

The current implementation uses provider lifecycle events as the stable boundary
between LLM round-trips:

- =e-loop= emits =provider-request-started= when the backend request handle is
  published, and =provider-request-finished= exactly once when the provider
  request completes or errors.
- =e-harness= includes provider lifecycle events, reasoning deltas, tool starts,
  tool finishes, turn settlement, token usage, and failure/cancellation events in
  durable session activity.
- =e-chat= records provider starts as structured =Thinking= entries and converts
  the matching active entry to =Thought for <duration>= when the provider request
  finishes.
- =e-chat= groups tool-start entries into visible =N tool call(s)= chunks and
  keeps tool-finished entries available for tool-list/output details.
- Final assistant rendering marks the turn final, then either renders a settled
  =Turn took <duration>[, <N> tool call(s)].= summary when provider boundary data
  exists or removes no-provider-boundary transient activity.
- Replayed sessions reconstruct summaries only from durable activity events with
  provider boundaries.  Transcript =tool-call= / =tool= messages are ignored by
  presentation replay rather than used as a legacy fallback.

- Finished provider requests render =Thought for <duration>=.
- Duration formatting uses the planned =Nmin Nsec= shape, including minute

- Multiple provider/tool cycles can render the planned history shape: completed

  in =e-loop= / =e-harness= tests.
- Legacy fallback support for old activity shapes has been removed per the

  activity, and no-provider-boundary activity is deleted after final output.
- Existing final assistant rendering, composer behavior, response navigation, and
  progress redraw preservation tests remain stable.

** Previously Deferred / Now Resolved

- The plan says the transient activity block shows one line per progress record.
  Final state: thoughts, reasoning, and tool batches are represented as semantic
  =:activity-records= with explicit round ownership.  Tool calls intentionally

  visible examples.
- The plan says each expanded summary line should be its own block, and that
  expanded tool-batch/thought/reasoning blocks should have individual actions.
  Final state: summary activation now creates child blocks for thought,
  reasoning, and tool-batch lines.  Tool-batch children open the existing

- Expanded thought and reasoning lines now have line-specific copy/open behavior

- Reasoning and tool batches now belong to explicit semantic provider-round
  records instead of depending on display-title parsing.
- Provider errors, turn failures, and cancellations now settle open thinking into
  duration-bearing terminal thought records under live rendering and replay.

* Detailed Findings

** Runtime and durable event boundaries

=e-loop= owns provider boundary detection and emits lifecycle events without
exposing backend handles to =e-chat=.  The request lifecycle tests confirm both a
single final-response request and a tool-follow-up cycle emit ordered events:

#+begin_example
turn-started
provider-request-started
reasoning-delta
tool-started
provider-request-finished
tool-finished
provider-request-started
provider-request-finished
turn-finished
#+end_example

=e-harness= persists these event types in session activity, which gives replay
sufficient timing data for settled summaries.

** Chat presentation

=e-chat= keeps presentation policy local to the chat shell.  It records provider
start/finish entries, tool call/output entries, and summary metadata in the turn
record.  The visible final summary is gated on =:has-provider-activity= plus turn
start/end times, so old/no-boundary activity does not invent timing.

The summary text uses the turn duration from the first known turn start to the
final assistant/turn finish time.  Tool count comes from recorded =Tool call=
entries.  Expanded summary text is derived from visible chunks and then appends
turn timing metadata.

** Replay behavior

Replay now prefers only durable activity events.  Tool transcript records are
recognized as model-context messages and skipped by presentation rendering.  This
matches the updated plan decision that older session records can be dropped and
should not receive legacy display support.

** Navigation behavior

The summary itself participates in normal response navigation and inline details
can be toggled by activating it.  Running activity blocks remain navigable while
progress is active, and progress redraw preserves response navigation, block
view, context-reference composer state, and running tool-list state.

The remaining navigation mismatch is specific to expanded summary children: the
current implementation has one summary block plus inline details, not separate
child blocks per expanded line.

* Review Result

The feature implementation now matches the runtime, persistence, replay, active
progress, final summary, expanded child-block, line action, and failure/cancel
settlement goals in =plan.org=.  The review follow-ups below have all been
resolved, so feature 05 can be marked =Done=.

Recommended follow-ups:

1. Represent reasoning/tool batch entries with explicit round metadata instead
   of relying on intermittent-entry order.
   - Resolution: implemented by
     =improvement-01-explicit-round-records.org=.  =e-chat= now maintains
     semantic =:activity-records= round records with reasoning and tool-batch
     children, and summary/tool detail derivation reads those records when
     present.
2. Render expanded summary history as child blocks if separate navigation/actions
   remain desired.
   - Resolution: implemented by
     =improvement-02-expanded-summary-child-blocks.org=.  Summary activation now
     toggles child blocks for thought, reasoning, and tool batches.  Tool-batch
     children open the existing tool-list flow, thought/reasoning children can be
     copied independently, and collapse removes the child records.
3. Add line-specific copy/open behavior for expanded thought and reasoning
   blocks after child blocks exist.
   - Resolution: covered with improvement 02's child blocks.  The standard
     response-navigation copy/open commands operate on the individual
     thought/reasoning child block content.
4. Decide whether failure/cancellation should rewrite open thinking entries into
   explicit failed/cancelled thought records or leave current failure UI as the
   terminal state.
        - Decision: it should.
        - Resolution: implemented by
          =improvement-03-failure-cancellation-thinking-settlement.org=.
          Provider errors, turn failures, and cancellations now settle open
          thinking into duration-bearing failed/cancelled thought lines under
          live rendering and durable replay.
