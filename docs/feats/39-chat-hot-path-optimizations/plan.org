#+startup: overview

* Status

* Goal

Reduce the measured chat hot paths from the 2026-06-28 developer profile:
progress/activity redraw churn, repeated composer rebuild work, and opaque
turn-start latency.

* Slices

** DONE Slice 1: In-place running activity redraws

Progress ticks update the existing running-status region when the composer is
already clean.  Dirty composer text that needs presentation-property cleanup
still falls back to the full restore path.

Acceptance:

- Progress redraws preserve drafts and context references.
- Progress redraws preserve response-navigation, block-view, and tool-list
  state.
- Clean progress redraws do not delete or reinsert the composer.

** DONE Slice 2: Cache composer spacer layout inputs

Avoid repeating expensive composer spacer screen-line counts when the transcript
and composer visual shape have not changed.

Acceptance:

- Window changes and multiline composer growth still recompute spacer lines.
- Activity/progress redraws do not force unrelated composer spacer work.

** DONE Slice 3: Break down turn-start latency

Add narrower profile spans around async turn startup so future profiles show
whether the 214ms startup outlier is request-body construction, backend start,
or loop setup.

Implemented spans:

- =loop.backend-start=
- =openai.request-context=
- =openai.request-body=
- =openai.request-json=
- =openai.request-headers=

Acceptance:

- Existing =harness.prompt-turn-async-start= remains available.
- New spans do not include prompt text, response text, tool output bodies, or
  unbounded state.
- Reports aggregate the new events without format changes.
