* Findings

** TODO Medium: helper timeout path lacks explicit test coverage

=e-mcp--helper-request= has an =e-mcp-backend-timeout= path when the helper does
not answer before the deadline, and the plan explicitly asks for helper timeout
tests.  Current tests cover helper =ok: false= failures and real helper/server
integration, but not a deterministic timeout.

Expected fix: add a small fixture/helper transport or subprocess fixture that
causes a request to exceed a very short timeout, then assert
=e-mcp-backend-timeout= and infrastructure-error tagging.

** TODO Medium: invalid helper response path lacks explicit test coverage

=e-mcp--helper-request= signals =e-mcp-protocol-error= when the helper response is
not a plist with an =:ok= member.  The plan asks to test invalid helper responses.
Current tests cover helper-reported errors but not malformed responses such as
=nil=, a string, or a plist missing =:ok=.

Expected fix: add fake-transport tests for malformed responses and assert
=e-mcp-protocol-error=.

** TODO Medium: notification-driven stale-catalog transition is not asserted

The helper handles =notifications/tools/list_changed= by setting a session stale
flag, and diagnostics expose =stale=.  The fixture server sends such a
notification during initialization.  Current tests prove explicit refresh and
catalog changes, but they do not assert the notification-driven stale transition
itself.

Expected fix: add a fixture or fake helper test where =notifications/tools/list_changed=
arrives after an initialized catalog, assert diagnostics show stale before
refresh, and assert refresh clears the flag.  If stale diagnostics are not meant
to be public, document the intended observability boundary and test the chosen
internal behavior.

* Architecture review

The implementation preserves the intended boundary:

- MCP protocol/subprocess concerns live in =e-mcp= and =e-mcp-helper.mjs=.
- Capabilities remain the extension seam.
- The harness sees ordinary capability/tool/resource records and does not know
  about MCP sessions or JSON-RPC state.
- Default layer behavior is unchanged; MCP is opt-in.

The Node helper is a real adapter boundary because it owns the side effects that
would otherwise leak into core harness policy: subprocess lifetime, stdio
framing, server stderr, request timeouts, and protocol diagnostics.

* Design self-check

1. If this work stopped here, E is better off: configured MCP stdio servers can
   contribute usable normal =e= tools through opt-in capabilities.
2. This moves toward capability-owned external tool adapters that keep protocol
   details below the harness boundary.
3. Stable decisions: capabilities are the seam, generated tools are normal =e=
   tools, stdio helper owns MCP protocol.  Likely to change: resources, prompts,
   auth UX, non-stdio transports, richer content blocks; those remain local to
   =e-mcp= and the helper.
4. =e-mcp= owns MCP capability construction and result mapping; the Node helper
   owns server subprocesses and JSON-RPC framing.
5. Coupling stays narrow by registering ordinary =e= tools and storing MCP origin
   metadata on tool definitions/results.
6. No harness dependency on MCP runtime state was added.  =e-core= loads =e-mcp=
   as part of the core surface, but harness policy consumes only capabilities and
   tools.
7. Protocol and subprocess side effects are already outside the harness in the
   helper boundary.
8. The abstraction is semantically real: MCP catalogs map to tool definitions and
   MCP calls map to tool handlers.  Unsupported content is summarized truthfully
   rather than exposed as a fake capability.
9. This is new opt-in behavior; existing default tools and backend-neutral tool
   definitions remain compatible.
10. No legacy MCP code exists to remove.
11. MCP execution errors become model-visible tool errors; helper/process/
    timeout/protocol failures surface as infrastructure errors.  Timeout and
    invalid-response tests should prove those paths directly.
12. Tests prove validation, fake transport, registration, result mapping,
    explicit refresh, real helper integration, child respawn, and harness tool
    execution.  Remaining narrow tests are timeout, malformed helper response,
    and notification-driven stale transition.
