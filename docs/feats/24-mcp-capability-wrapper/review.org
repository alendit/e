* Implementation Summary

- Added =e-mcp-server= and =e-mcp-tool= specs with validation for server ids,
  commands, tool names, and input schemas.
- Added =e-capability-with-mcp-create= so MCP catalogs register generated tools
  through the existing capability tool contribution path.
- Added the Node helper process for stdio MCP server lifecycle, JSON-RPC
  framing, =initialize=, =tools/list=, =tools/call=, refresh, stderr diagnostics,
  stale catalog tracking, and process reset.
- Added result mapping for text blocks, basic structured JSON values, combined
  text plus =structuredContent=, MCP =isError= execution errors, unsupported
  content summaries, and helper infrastructure failures.
- Kept MCP opt-in. No default layer integration was added.

* Plan Comparison

Implemented plan items:

- [[file:plan.org::*Task 1: MCP server and catalog data model][Task 1]]:
  =e-mcp-server= and =e-mcp-tool= data models exist with validation and focused

- [[file:plan.org::*Task 5: Tool call result and error mapping][Task 5]]:
  Generated handlers call the original MCP tool name with supplied arguments,
  map success and =isError= results into normal =e= tool results, and signal
  helper failures through infrastructure-error conditions.
- [[file:plan.org::*Task 6: Refresh command and diagnostics][Task 6]]:
  =e-mcp-refresh= asks the helper for fresh catalogs without mutating an active
  registry during a turn.
- [[file:plan.org::*Task 7: Default integration smoke path][Task 7]]:
  The smoke test constructs an explicit MCP capability/layer, activates it on a
  harness, observes the generated tool, and executes it through =e-tools=.

* Architecture Self-Check

1. If this work package stopped here, would the project be better off than if it
   had not been built?
   - Yes. MCP stdio tool discovery and calls are usable through opt-in
     capabilities without changing default tools.
2. What final direction does this move toward?
   - Capability-owned external tool adapters that keep protocol details below
     the harness boundary.
3. Which decisions are likely to change, uncertain, or stable, and are
   uncertain ones local and reversible instead of hidden behind speculative
   abstraction?
   - Stable: capabilities remain the extension seam. Likely to change:
     resources, prompts, auth UX, non-stdio transports, and richer content
     blocks. Those remain local to =e-mcp= and its helper.
4. What component owns this behavior, and why?
   - =e-mcp= owns MCP capability construction and result mapping. The Node
     helper owns server subprocesses and MCP JSON-RPC framing.
5. Does this change increase or reduce coupling?
   - It keeps coupling narrow by registering ordinary =e= tools and storing MCP
     origin metadata on the tool record.
6. Did any dependency start pointing the wrong way, especially from stable
   policy toward concrete adapters, providers, UI, transport, persistence, or
   tool clients?
   - No harness dependency on MCP was added. =e-core= loads the MCP module as
     part of the core runtime surface, and the harness still consumes only
     capabilities and tools.
7. Could any side effect be moved outward into an adapter or shell?
   - The protocol and subprocess side effects are already outside the harness in
     the helper process boundary.
8. Are the abstractions and interfaces semantically real, and can
   implementations substitute for each other without narrowed preconditions,
   weakened behavior, ignored requirements, or unsupported-operation errors?
   - The public contract is concrete MCP server specs, catalog tools, refresh,
     and generated =e= tools. Unsupported MCP content is summarized truthfully
     rather than exposed as a false capability.
9. What compatibility expectations apply, and were obsolete paths removed unless
   required?
   - This is new opt-in behavior. Existing tool definitions remain backend
     neutral; optional metadata is stored without changing provider payloads.
10. What legacy code can we remove now?
    - None in this first slice.
11. Where are expected errors handled, and where do unexpected errors surface?
    - MCP =isError= results become model-visible tool errors. Helper, timeout,
      process, and protocol failures surface as infrastructure errors.
12. What tests prove the core behavior independently of the full system?
    - =test/e-mcp-test.el= covers validation, fake transport, registration,
      result mapping, refresh, helper process integration, child respawn, and
      harness tool execution. The remaining coverage gaps are listed in
      [[*Findings][Findings]].
