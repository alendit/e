#+title: MCP Capability Wrapper
#+startup: overview

* Goal

Let =e= consume tools exposed by configured Model Context Protocol servers by
turning discovered MCP tools into ordinary =e= tools contributed by an ordinary
=e-capability=.

The feature should preserve the current architecture boundary: capabilities are
the runtime extension seam, and the harness should not gain a separate MCP
runtime category.

* Status

Planned.

* Background

MCP support should follow the same broad shape as the skills wrapper:
construction-time helper code returns an ordinary =e-capability=, and the
harness consumes the result through existing capability contribution paths.

Unlike skills, MCP servers are active protocol endpoints. They require
subprocess ownership, JSON-RPC framing, initialization, discovery, tool calls,
timeouts, and diagnostics. Those concerns should stay outside core harness
policy.

The chosen direction is discovery/registration, not a generic
=mcp_call_tool=. =e= should discover MCP tools and register them as normal
model-facing tools with stable names, descriptions, and JSON schemas.

* Non-goals

- Do not add MCP as a peer runtime primitive next to capabilities.
- Do not teach =e-harness= about MCP server state, JSON-RPC framing, or MCP
  lifecycle details.
- Do not expose a single generic =mcp_call_tool= as the first user-facing API.
- Do not implement MCP resources, prompts, sampling, auth UX, non-stdio
  transports, or rich non-text content in the first slice.
- Do not dynamically mutate the active tool registry during a model turn.

* Design

** Capability builder

Add an MCP-specific capability builder:

#+begin_src elisp
(e-capability-with-mcp-create
 :id "github-mcp"
 :name "GitHub MCP"
 :mcp-servers
 (list (e-mcp-server-create
        :id "github"
        :command '("npx" "-y" "@modelcontextprotocol/server-github")
        :env '(("GITHUB_TOKEN" . "...")))))
#+end_src

The builder returns an ordinary =e-capability=. Its =:tools= contribution is a
registration function that registers generated =e= tools based on the MCP tool
catalog.

The harness sees only a capability. It registers the generated tools through
the existing =e-tools-register= path.

** Helper process boundary

Introduce a small MCP wrapper/helper process owned outside the harness.

The helper owns:

- MCP server subprocess startup and shutdown.
- MCP stdio JSON-RPC framing.
- =initialize= and capability negotiation.
- =tools/list= discovery.
- =tools/call= execution.
- =notifications/tools/list_changed= handling.
- stdout/stderr capture and diagnostics.
- timeout and protocol error normalization.

Emacs talks to the helper using a small framed JSON protocol, following the
same broad process pattern as the existing browser helper: lazy startup,
request ids, per-request timeouts, stderr diagnostics, and reset support.

** Discovery and registration flow

On capability activation or explicit refresh:

1. Ensure the MCP helper process is live.
2. For each configured server, ask the helper to initialize or reuse the MCP
   session.
3. Ask the helper for the server's =tools/list= catalog.
4. Convert each MCP tool into an =e= tool definition.
5. Register each generated tool with =e-tools-register=.

Generated tool names are deterministic and namespaced:

#+begin_example
mcp.<server-id>.<tool-name>
#+end_example

Example:

#+begin_example
mcp.github.create_issue
#+end_example

The generated =e= tool:

- uses the MCP tool description, prefixed with the MCP server id when useful;
- uses the MCP =inputSchema= as =e= =:parameters=;
- stores metadata with the originating server id and MCP tool name;
- calls the helper with the original MCP tool name and supplied arguments.

** Tool call flow

When the model calls =mcp.github.create_issue=:

1. The generated =e= handler receives the already schema-shaped arguments.
2. The handler sends a helper request equivalent to MCP =tools/call=:

   #+begin_src json
   {
     "server": "github",
     "tool": "create_issue",
     "arguments": {
       "title": "Bug",
       "body": "Details"
     }
   }
   #+end_src

3. The helper sends MCP =tools/call= to the server.
4. The helper returns normalized result content and diagnostics to Emacs.
5. The generated handler returns a normal =e= tool result.

MCP tool execution errors where the server returns =isError: true= should be
model-visible tool errors. Infrastructure failures, such as dead helper
process, invalid JSON-RPC, timeout, missing server, or missing MCP tool, should
surface as hard backend/tool infrastructure errors.

** Tool catalog refresh

When the MCP server emits =notifications/tools/list_changed=, the helper marks
that server catalog stale.

The first implementation should support explicit refresh only:

#+begin_example
M-x e-mcp-refresh
#+end_example

or an equivalent internal refresh function used by tests. A later slice can add
live shell notification or automatic refresh before the next turn. The first
slice should avoid mutating the active registry during an in-flight turn.

** Result mapping

Initial result support should cover text content.

MCP result:

#+begin_src json
{
  "content": [
    {
      "type": "text",
      "text": "Created issue #123"
    }
  ],
  "isError": false
}
#+end_src

maps to a normal =e= tool result whose model-visible content is the text.

If multiple text blocks are returned, join them with blank lines and preserve
order. Unsupported content blocks should be summarized with their type and
omitted payload details in the first slice, so the model gets a truthful result
without pretending to support media or binary artifacts.

* Acceptance Criteria

- A configured stdio MCP server can be discovered through the helper.
- MCP =tools/list= entries become normal =e= tools with deterministic names.
- Generated tools use MCP descriptions and input schemas.
- Calling a generated tool sends MCP =tools/call= with the original MCP tool
  name and arguments.
- Text results are returned as model-visible =e= tool content.
- MCP =isError: true= is visible to the model as a tool execution error.
- Helper/process/protocol failures surface as infrastructure errors.
- The harness and core capability contract stay MCP-agnostic.
- MCP support is opt-in and does not alter existing default tools.
