* Findings

** TODO High: missing shell and command ids validate as valid ids

=e-shell--validate-id= accepts any symbol, and =nil= is a symbol in Emacs Lisp
(=lisp/shells/e-shells.el=).  That regresses the previous manifest boundary: a
shell with no id can be registered into =e-shell--registry= under the =nil= key.
The same helper validates command ids, so a command with no id also passes
=e-shell-validate=.

Evidence from this review pass:

#+begin_example
(e-shell-register (e-shell-create :id nil :name "Bad")) => accepted
#+end_example

Why this matters: project layers load trusted project Elisp and then validate
shell manifests through this path.  A malformed project shell can occupy the
=nil= key, appear in active shell lists with no usable id, and make command
lookup/completion ambiguous instead of failing at the manifest boundary.

Expected fix:

- Make =e-shell--validate-id= reject =nil=.
- Add tests for missing shell ids and missing command ids.
- Verify both global registration and layer-owned registration reject invalid
  manifests.

** TODO High: layer-scoped skills are documented/planned but not exposed

The plan and reference document describe layer-level skills at:

#+begin_example
.e/layers/<layer-id>/skills/<slug>/SKILL.md
#+end_example

and suggest a URI shape like:

#+begin_example
e://project-local/layers/<layer-id>/skills/project/<slug>
#+end_example

The current implementation only folds in skills under legacy capability
directories via =e-project-local--capability-skill-resources=.  Project layer
loading aggregates capabilities and shells, but does not discover/register
=.e/layers/<id>/skills/= resources.

Why this matters: a project author following =docs/references/project_local_layers.org=
can add a layer-scoped skill file and get no runtime resource for it.  That makes
the documented preferred layout partially inert.

Expected fix: either implement layer-scoped skill resource exposure with the
chosen URI shape and tests, or explicitly mark layer-scoped skills as deferred in
both the plan and =docs/references/project_local_layers.org=.

** TODO Medium: focused verification currently has an unrelated defaults test failure

The current worktree fails =e-defaults-test-startup-specs-only-include-chat-default=:

This looks like drift from uncommitted default-harness work rather than a direct

** TODO Low: shell listing UX still relies on callers choosing active APIs

The implementation adds =e-shell-list-active= and =e-shell-get-active=, but the
broader listing/completion UI remains mostly implicit.  This is acceptable as a
first slice if documented, but Slice 3's user-facing discovery/completion polish
is only partially complete.

If no broader UI is intended in this feature, mark that portion deferred and keep
the active APIs as the accepted shell-discovery contract for now.

* Architecture review

The architecture moves in the intended direction:

- =e-project-local= owns trust-gated project discovery and aggregate layer
  construction.
- =e-harness= owns active layer lifecycle and invokes scoped shell registration.
- =e-shells= owns manifest validation, global/scoped registration, collision
  handling, and active lookup.
- Project code declares layers; it does not directly mutate global shell state.

The validation gap around =nil= ids is the main architecture defect because it
weakens the manifest boundary.  The missing layer-scoped skills are a contract
mismatch between the documented package shape and runtime resource exposure.

* Design self-check

1. If this work stopped here, E is better off: layers now have a real shell slot
   and project packages can bundle capabilities and shells.
2. This moves toward trusted project-local layers as the package concept for
   model-facing and presentation extensions.
3. Stable decisions: harness-owned layer lifecycle, scoped shell registration,
   aggregate =project-local= first implementation.  Unsettled: layer-scoped skill
   URI/resource shape; keep it local to =e-project-local=.
4. =e-project-local= owns trusted discovery, =e-harness= owns activation, and
   =e-shells= owns shell registry validation/collision policy.
5. Coupling is reduced by deriving active shell visibility from active layers;
   =e-shell-list-active= and =e-shell-get-active= are narrow interfaces.
6. No reversed dependency was found.  Harness depends on shell registry
   contracts; project-local discovery produces layer data.
7. Project file loading stays in the trust-gated project-local adapter.  Shell
   registration is correctly performed by harness lifecycle.
8. The layer-shell relationship is real.  Nil-id validation currently weakens
   substitutability and must be fixed.
9. Legacy =.e/capabilities/= remains supported as compatibility.  No obsolete
   path is removable yet.
10. No legacy code can be removed in this slice.
11. Untrusted roots and duplicate ids are handled.  Invalid nil shell/command ids
    currently fail to surface and should become expected validation errors.
12. Tests cover scoped shell registration, lifecycle, sync cleanup, trusted
    project layer contribution, allowlist gating, duplicate ids, and legacy
    compatibility.  Missing tests: nil id rejection and layer-scoped skills.
