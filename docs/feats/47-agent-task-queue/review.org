* Suggested review use cases

Derived from =plan.org= acceptance, plus surface-driven additions:

- Enqueue returns a =queued= record immediately; dispatch may start it before
  return.
- With cap 2, a third enqueue waits until one running task settles.
- =queued= -> =running= -> =done= over a real harness turn; =read-task= returns
  outputs.
- Failing turn lands =failed= with =:error= and does not block the dispatcher.
- Cancel of a queued task: =cancelled=, never runs; cancel of a running task:
  interrupts and a late settle does not resurrect it.
- Per-task =harness-instance-id= wins; omitted uses the queue default;
  unregistered id fails just that task without stalling dispatch.
- List shell: opens newest-first, refreshes on queue change, row cancel and
  show-outputs, workspace-aware display; empty-queue state; repeated use.

* What looks good

- Dispatch-time harness resolution is implemented exactly as the plan argues
  for: =e-task-queue--start= resolves the instance just before running, so
  re-pointing the default or fixing a bad id still affects queued tasks
  (=e-task-queue.el= lines ~204-216).  The unregistered-instance case fails only
  that task and the dispatcher keeps serving, proven live and by
  =e-task-queue-test-unregistered-instance-fails-task=.
- Re-entrancy guard (=dispatching=) lets a synchronous runner settle without
  recursive dispatch, and the loop still picks up freed slots.  Verified live: a
  runner that calls =on-settle= inside its own body settles =done= and the next
  queued task starts.
- =e-task-queue--settle= is correctly a no-op unless the task is still
  =running=, so a late settle from a runner cannot resurrect a cancelled or
  terminal task.  Verified live and by =e-task-queue-test-cancel-running-task=.
- The capability is actions-only with a readable skill resource, and the
  =is-action-not-tool= test asserts the queue never reaches model-facing tool
  definitions, matching the =elisp-job= precedent.
- Core module depends only on =e-harness= and =e-harness-instances=, never a
  shell, so the queue genuinely runs headless (confirmed by the test/probe
  paths that never load the shell).

* UX and performance playthrough

Exercised in the running Emacs against the reloaded build (run_elisp; the
=e-dev-reload= helper itself is blocked under the interactive load guard, but
the queue and shell symbols were confirmed live and the focused suite ran
clean):

- =enqueue= via =e-actions-call='s action handler returns a record immediately
  (=running= under the cap), =list-tasks= reflects it, =cancel-task= moves it to
  =cancelled=.  Round-trip clean.
- Default queue is constructed and resolves a default chat instance
  (=:chat-default=) live; =e-task-queue-list-buffer= and =e-task-queue-layer-create=
  are bound.
- Synchronous-settle runner: task settles =done= with outputs and session id
  recorded, next queued task dispatches.  No recursion.
- Cancel of a terminal (=done=) task is a safe no-op (returns =done=, the
  handle's cancel thunk is never called).
- Change-hook registration dedupes across repeated =e-task-queue-list-buffer=
  opens (=add-hook= keeps a single entry), so reopening the buffer does not
  multiply refresh work.

Performance: dispatch is event-driven (enqueue + settle), =--running-count= and
=--oldest-queued= are linear over the in-memory record set, which is fine for
the intended task volume.  No timer polling.  No regression observed.

* Findings

** TODO Medium: =harness-instance-id= action schema invites a string the queue rejects

*** Problem

The =enqueue= action advertises =harness-instance-id= as =(:type "string")=
(=e-task-queue-actions--enqueue-parameters=), and the skill text describes it as
=KEYWORD=.  The queue resolves the id through
=e-harness-instance-get-or-create=, whose =e-harness-instance--validate-id=
requires a =keywordp= value.  A caller that follows the JSON schema and passes
the string ="chat-a"= lands the task in =failed=:

#+begin_example
(e-task-queue-enqueue queue :prompt "go" :harness-instance-id "chat-a")
;; => status failed  ("Cannot resolve harness instance chat-a: ...")
#+end_example

Verified live.  A keyword (=:chat-a=) works; the advertised string never does.

*** Why it matters

The action surface is self-contradictory: the only documented way to target a
specific instance from a model/JSON caller is the field whose advertised type
guarantees failure.  Per-task instance selection is a named acceptance item and
the contract grimoire triage is meant to use.

*** Smallest fix direction

Either coerce a string instance id to an interned keyword at the action boundary
(=e-task-queue-actions--enqueue=), or change the validator/contract to accept a
string id.  Coercion at the boundary keeps the core keyword-typed and matches
the skill if the skill is also corrected to say "instance id string, e.g.
\"chat-project-e\"".  Add a round-trip test that enqueues with a string id and
asserts it resolves.

** TODO Low: =g= is bound to a non-command and errors

*** Problem

=e-task-queue-shell-mode-map= binds =g= to =e-task-queue-shell--refresh=, a
private helper with no =(interactive)= form.  =(commandp
'e-task-queue-shell--refresh)= is nil; pressing =g= signals
=(wrong-type-argument commandp e-task-queue-shell--refresh)=.

*** Why it matters

Manual refresh is the operator's escape hatch when the change hook misses a
wakeup; today it throws instead of refreshing.

*** Smallest fix direction

Give the refresh an interactive command wrapper (or rely on
=tabulated-list-mode='s built-in =g= =revert= wired to the queue refresh) and
bind that.  Add the keybinding to the shell smoke test.

** TODO Low: stale =:handle= retained on a synchronously-settled task

*** Problem

When a runner settles inside its own call (synchronous settle),
=e-task-queue--settle= sets =:handle nil=, but control then returns to
=e-task-queue--start=, which runs =(plist-put record :handle handle)= afterward,
leaving a dead handle plist on the now-terminal record.  Observed live:

#+begin_example
;; after synchronous on-settle :status done
(:status done :handle (:session-id "s" :cancel ignore))
#+end_example

*** Why it matters

Harmless today because cancel is a no-op on a terminal task, so the stale cancel
thunk is never invoked.  It is dead state that contradicts the settle's own
cleanup and could mislead a future change that trusts =:handle= as "live".

*** Smallest fix direction

In =e-task-queue--start=, only store the handle when the task is still =running=
after the runner returns (or have the runner-return path skip the =plist-put=
when the record already settled).  Cover with a synchronous-settle test.

* Architecture review

- *Ownership and dependency direction*: core registry/dispatcher in
  =lisp/core/=, capability in =lisp/layers/agents/=, shell in =lisp/shells/=.
  Dependencies flow from shell/capability toward the stable core; the core
  depends only on =e-harness= / =e-harness-instances=.  Correct direction, one
  reason to change per module.
- *Dispatch-time vs enqueue-time resolution*: resolving the harness instance at
  dispatch is the right call -- it keeps the late-binding semantics the plan
  wants and avoids pinning a queued task to a harness that may be re-pointed.
- *Runner seam*: =(task harness on-settle) -> handle= is a genuinely narrow,
  substitutable contract.  The dispatcher owns instance resolution and hands a
  live harness in, so an Agent Shell Fleet or =elisp-job= runner could drop in
  without touching the registry.  No leaked provider concerns.
- *Side effects*: harness turns and subscriptions live in the default runner,
  not the registry; the registry is pure state plus event hooks.  Good
  isolation.
- *State lifetime*: in-memory only, matching =e-agent-shell-work= and
  =e-elisp-job=; persistence is an explicit later slice.  The change hook is the
  right seam for the shell to observe without the core knowing about UI.
- *Fail-vs-default on unregistered instance*: failing the task (rather than
  silently falling back) is the correct loud-failure choice and is tested.

No dependency points the wrong way; no speculative abstraction; the one
medium finding is a contract/marshalling bug at the action boundary, not a
structural problem.
