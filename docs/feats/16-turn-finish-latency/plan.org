#+title: Plan: Turn Finish Latency
#+author: e planning
#+date: 2026-05-29
#+startup: overview

* Status

In-review

* Goal

Reduce the main-thread pauses users see when a turn settles or emits dense
activity, while keeping session persistence durable and chat presentation
accurate.

The first priority is removing stop-the-world work from the hot path. The live
investigation measured the current durable-entry ID path at roughly six seconds
per 100 generated entry ids because =e-session--random-80-bit= calls
=garbage-collect=. Removing that explicit GC is expected to eliminate the worst
freeze class before broader rendering or persistence work.

* Non-goals

- Replace JSONL persistence with SQLite or a new storage backend.
- Remove durable session activity history.
- Change the public shapes returned by =e-session-messages=,
  =e-session-activity-events=, or =e-harness-session-activity-events=.
- Rework backend streaming or provider request semantics.
- Treat byte compilation as the primary fix.
- Rewrite the chat shell or split =e-chat.el= as part of this feature.

* Current behavior

Turn completion and related activity callbacks do several kinds of synchronous
work on the Emacs main thread:

- =e-session--random-80-bit= builds entropy by calling =garbage-collect=, so
  every new durable entry id can force an expensive global collection.
- =e-harness--emit-turn-event= persists durable activity through
  =e-session-append-activity-event= before subscriber callbacks render the event.
- Message and activity append paths append the JSONL record and call
  =e-session--write-index= immediately. The index writer serializes the whole
  session list.
- =e-chat--set-status= calls =e-chat--refresh-mode-line-status= for almost every
  status transition. Without fresh provider usage, that can rebuild full harness
  context and estimate context tokens.
- Running activity redraw paths delete and recreate transient status/composer
  text and call =(redisplay t)=.

Byte compilation can help pure Elisp loops and regex-heavy rendering, but it does
not address the dominant live finding: forced GC inside entry-id generation.

* Accepted decisions

** Remove forced GC from durable entry id generation first

Do not keep =garbage-collect= in =e-session--random-80-bit=. Entropy should come
from cheap process-local and time/random inputs, and monotonicity should continue
to be provided by =e-session-generate-ulid=.

Implementation requirements:

- Change only =e-session--random-80-bit= and directly related tests in the first
  slice.
- Add a regression test that would fail if =garbage-collect= is called by
  =e-session-generate-ulid=.
- Keep the ULID string length and monotonic same-millisecond behavior intact.
- Benchmark the live running Emacs after reload with a scalar bounded probe.

** Defer broad byte-compilation work

Do not make this feature depend on installing =.elc= or =.eln= files. Development
reload currently loads source, and the measured freeze is dominated by explicit
GC and synchronous main-thread work.

Implementation requirements:

- Mention byte compilation only in documentation or review notes as an optional
  later optimization for pure Elisp CPU work.
- Do not add build, package, or startup changes for compiled files in this
  feature.

** Persist less synchronously without losing turn history

Keep durable activity, but avoid rewriting broad metadata for every activity
event. Session index freshness matters for session lists and resume ordering;
reasoning/tool activity does not need a full index rewrite on every event.

Implementation requirements:

- Add an internal append option or helper that can skip immediate index writes
  for activity events whose only effect is activity history.
- Ensure user and assistant message appends still update the index promptly
  because they affect session title, summary, recency, and message count.
- Ensure terminal turn events flush or refresh the index once for the turn if
  needed.
- Do not change JSONL record order.

** Make mode-line context refresh event-aware

Status text should not rebuild model-facing context on every event. Token usage
or cached estimates are enough during a running turn, and full context estimates
should be explicit, cached, or delayed.

Implementation requirements:

- Split cheap header/status updates from expensive mode-line context/token
  estimation.
- Make =e-chat--set-status= avoid recomputing context unless the caller
  explicitly requests it.
- Keep =token-usage= events updating the mode line from durable usage without
  rebuilding context.
- Add tests around =e-chat--set-status= proving ordinary status transitions do
  not call =e-harness-context=.

** Coalesce running activity redraws

Progress and tool activity should render at a bounded cadence. Several activity
events in one callback burst should not each force a full transient redraw and
redisplay.

Implementation requirements:

- Introduce a chat-local deferred redraw helper that schedules one near-future
  redraw for the active turn.
- Keep immediate rendering for final assistant messages, failures, and user
  submissions.
- Remove unconditional =(redisplay t)= from ordinary activity events, or gate it
  behind the deferred redraw helper.
- Preserve the existing response-navigation, block-view, tool-list, and composer
  state guarantees.

* Design summary

The work should stay in the current ownership boundaries:

- =e-session.el= owns durable entry id generation, append behavior, JSONL writes,
  and session index writes.
- =e-harness.el= owns deciding which turn events are durable and when turn-level
  persistence/index flushes happen.
- =e-chat.el= owns presentation status, mode-line text, and redraw cadence.
- Tests remain focused around these seams instead of timing wall-clock UI
  freezes directly.

The plan is ordered from highest confidence and lowest coupling to broader UI
work. Each slice should be useful if the feature stops there.

- Modify =lisp/core/e-session.el=

   =e-session-generate-ulid= several times.

   (ert-deftest e-session-test-generate-ulid-does-not-force-garbage-collect ()
     (cl-letf (((symbol-function 'garbage-collect)

   =(garbage-collect)=.

   #+begin_src emacs-lisp
   (defun e-session--random-80-bit ()
     "Return a sufficiently random 80-bit integer."
     (let* ((seed (format "%S" (list (current-time) (random t) (emacs-pid)
                                     (system-name))))
            (hex (substring (secure-hash 'sha1 seed) 0 20)))
       (string-to-number hex 16)))

** Slice 2: Avoid per-activity index rewrites

Files:

- Modify =lisp/core/e-session.el=
- Modify =lisp/core/e-harness.el=
- Modify =test/e-session-test.el=
- Modify =test/e-harness-test.el=

Steps:

1. Add an internal optional argument to =e-session-append-activity-event= such as
   =&key write-index=, defaulting to current behavior for direct callers.

2. Add a helper in =e-harness.el= for durable turn activity persistence. For
   non-message activity events, call =e-session-append-activity-event= with
   immediate index writes disabled. For terminal events such as =turn-finished=,
   =turn-failed=, and =turn-cancelled=, flush the index once after appending the
   event.

3. Add a test that wraps =e-session--write-index= and emits a representative
   sequence through =e-harness--emit-turn-event=:

   #+begin_src emacs-lisp
   (ert-deftest e-harness-test-activity-index-write-is-coalesced ()
     (let* ((store (e-session-store-create))
            (harness (e-harness-create
                      :backend (e-backend-fake-create :items nil)
                      :sessions store))
            (write-count 0))
       (e-harness-create-session harness :id "session-1")
       (cl-letf (((symbol-function 'e-session--write-index)
                  (lambda (_store)
                    (setq write-count (1+ write-count)))))
         (e-harness--emit-turn-event harness "session-1" "turn-1"
                                     'provider-request-started
                                     '(:status started))
         (e-harness--emit-turn-event harness "session-1" "turn-1"
                                     'reasoning-delta
                                     '(:type reasoning-delta :content "thinking"))
         (e-harness--emit-turn-event harness "session-1" "turn-1"
                                     'tool-started
                                     '(:name "read" :arguments nil))
         (e-harness--emit-turn-event harness "session-1" "turn-1"
                                     'tool-finished
                                     '(:tool-call (:name "read") :result "ok"))
         (e-harness--emit-turn-event harness "session-1" "turn-1"
                                     'turn-finished
                                     '(:reason done)))
       (should (= write-count 1))
       (should (equal '(provider-request-started reasoning-delta tool-started
                        tool-finished turn-finished)
                      (mapcar (lambda (event)
                                (plist-get event :event-type))
                              (e-session-activity-events store "session-1"))))))
   #+end_src

4. Run:

   #+begin_src sh
   rtk eldev test -f test/e-session-test.el
   rtk eldev test -f test/e-harness-test.el
   #+end_src

5. For slices with live loaded chat buffers, reload and verify a live temporary
   turn still persists ordered activity events. Keep the live probe scalar:
   counts and event types only. If no Emacs server is available, record the
   exact =emacsclient= failure.

* Compatibility expectations

- Existing JSONL session files remain readable.
- Existing in-memory sessions continue to generate durable ids without format
  changes.
- Session recency and title behavior remain message-driven.
- Existing response navigation, block view, tool list, and composer behavior
  remain stable during progress redraws.
- No user config changes are required.

* Risks

- Skipping index writes too aggressively could make resume ordering stale until a
  terminal event flushes the index.
- Deferred redraw could make activity feel laggy if the timer interval is too
  long.
- Mode-line refresh changes could hide token usage if =token-usage= events do not
  request the explicit refresh.
- Removing explicit GC from ID generation must preserve enough uniqueness for
  practical durable entry ids. The monotonic suffix path should cover
  same-millisecond calls.

* Review questions

1. Does this reduce the worst live freeze path before touching broader UI code?
2. Does persistence still record every durable activity event in order?
3. Are session index writes tied to real index consumers rather than every
   append?
4. Does chat status update cheaply by default?
5. Does activity redraw preserve scrollback, response navigation, block view,
   tool list, and composer state?
6. What remaining latency is pure Elisp CPU work that byte compilation could
   reasonably improve after these structural fixes?
