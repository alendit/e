#+title: Plan: Session Append Performance
#+author: e planning
#+date: 2026-05-25
#+startup: overview

* Goal
Make session mutation scale linearly instead of quadratically as transcripts grow.

The immediate problem is that append paths copy whole lists on every mutation. A
session with =n= messages currently pays roughly:

#+begin_example
1 + 2 + 3 + ... + n = O(n²)
#+end_example

This feature keeps the public session API and insertion-order semantics intact,
while changing internal append-only list maintenance to O(1) per append.

* Non-goals
- Change the public return shape of =e-session-messages= or activity/session APIs.
- Replace plist-backed sessions with a new storage object model.
- Implement message/turn identity, branching, or compaction.
- Rewrite context construction.
- Change JSONL persistence format except for optional in-memory-only derived/tail fields.
- Optimize backend calls or UI rendering beyond avoiding session-store append hotspots.
- Add range/slice APIs for messages or activity events.

* Current behavior
Several session mutation paths append by copying the existing list:

#+begin_src elisp
(plist-put session :messages (append messages (list message)))
#+end_src

Similar patterns exist for:

- =:messages=
- =:activity-events=
- =:branch-summaries=
- =:compactions=

This means each append is O(length-of-list). Building a long transcript becomes
O(n²). The cost is currently masked by provider/tool latency, but it becomes
visible for long sessions, dense activity timelines, replay-heavy tests, and
future identity/compaction work that may add more durable entries.

There is a second related cost: append paths call broad derived metadata refresh
functions after each mutation. If those refreshes scan message lists, they also
contribute to O(n²) growth.

* Accepted decisions
** Session tail-field initialization
Use one internal, destructive session initialization helper for normal session
construction and replay finalization. The helper initializes every known
append-only list tail field and related internal list state in one place.

The append helper must still be defensive: if it sees a non-empty list with a
missing tail field, it computes and stores the tail once before appending. This
supports legacy in-memory session plists and narrow tests without relying on
non-uniform state for normal operation.

Implementation requirements:

- Add an internal helper, for example =e-session--initialize-list-state=.
- Call it from session creation, replayed session creation/finalization, and
  unloaded index stub construction as appropriate.
- Keep the helper internal; do not introduce a broad public session factory.
- Make mutation behavior clear in the docstring.

** Slice APIs
Do not add message or activity slice APIs in this feature.

This work package is limited to append-time complexity and derived metadata
refresh behavior. Existing full-list accessors remain the supported API. Slice or
range APIs should be considered later, preferably after message/turn identity and
compaction clarify whether ranges should be offset-based, id-based, or
branch-path-based.

Implementation requirements:

- Leave =e-session-messages= and =e-session-activity-events= public behavior
  unchanged.
- Do not add new public range/slice API in this feature.
- Do not change evidence tools as part of this work unless a test reveals a
  direct append-performance dependency.

** Branch summaries and compactions
Optimize branch summary and compaction append operations with the same tail
pointer mechanism, but do not add new derived metadata for them.

Message-derived metadata already feeds session list/index behavior, so it should
be maintained incrementally. Branch summaries and compactions currently have no
count/latest consumers, so adding metadata for them would be speculative.

Implementation requirements:

- Add tail fields for =:branch-summaries= and =:compactions=.
- Use the O(1) append helper for both lists.
- Do not add =:branch-summary-count=, =:compaction-count=, or latest timestamp
  fields in this feature.

** Activity-event metadata
Optimize activity-event append with tail pointers only. Do not add
=:activity-event-count= or latest activity timestamp fields in this feature.

Activity events are currently consumed as timelines. If future turn-progress or
history UI needs aggregate activity metadata, it should add that with a concrete
consumer-shaped contract.

Implementation requirements:

- Add =:activity-events-tail=.
- Use the O(1) append helper for activity events.
- Reset =:activity-events= and =:activity-events-tail= on message/session clear.
- Do not add activity count/latest metadata now.

** Public accessor copy behavior
Keep public accessors returning shallow copies exactly as they do today.

The canonical list spine is owned by the session store. Because tail pointers use
destructive =setcdr= on that spine, exposing canonical lists would make external
mutation able to corrupt append state. Deep copies would add cost and are not
necessary for this feature.

Implementation requirements:

- Keep =copy-sequence= behavior for full-list accessors.
- Do not return canonical internal lists from public APIs.
- Do not introduce deep-copy behavior unless a separate mutation-safety bug
  requires it.

** Complexity regression testing
Use deterministic structural and behavioral tests rather than wall-clock
thresholds.

The tests should prove order preservation and append-after-replay behavior, and
should verify that broad derived-field refreshes are not performed once per
append. Avoid global advice around =append= and avoid timing thresholds in the
automated suite.

Implementation requirements:

- Add behavioral order tests for messages, activity events, branch summaries,
  and compactions.
- Add append-after-replay tests that prove tails are initialized correctly.
- Add clear/reset tests that prove stale tails do not retain old entries.
- Add refresh-call counting around =e-session--refresh-derived-fields= or the
  replacement derived-refresh path to prove append hot paths do not perform full
  refresh per append.
- If practical, add a helper-level structural test showing the list head is
  preserved while the tail advances.

* Design summary
Keep canonical session lists in insertion order, but maintain internal tail cells
for append-only fields.

Add internal in-memory tail fields:

| List field | Tail field |
|------------+------------|
| =:messages= | =:messages-tail= |
| =:activity-events= | =:activity-events-tail= |
| =:branch-summaries= | =:branch-summaries-tail= |
| =:compactions= | =:compactions-tail= |

A small internal helper appends by mutating the previous tail cell. It also
repairs a missing tail field once when given a non-empty existing list:

#+begin_src elisp
(defun e-session--append-list-item (session field tail-field item)
  "Append ITEM to SESSION FIELD in O(1), tracking TAIL-FIELD."
  (let ((cell (list item))
        (tail (or (plist-get session tail-field)
                  (when-let ((items (plist-get session field)))
                    (let ((computed-tail (last items)))
                      (plist-put session tail-field computed-tail)
                      computed-tail)))))
    (if tail
        (setcdr tail cell)
      (plist-put session field cell))
    (plist-put session tail-field cell)
    item))
#+end_src

The session store owns these lists. Public accessors continue returning shallow
copies where they already do, so callers cannot accidentally corrupt canonical
list spines.

* Derived metadata strategy
Avoid full derived-field refreshes on every append. Maintain append-affecting
metadata incrementally:

- =:message-count= increments on message append and resets on clear.
- =:summary= is set when the first user string message appears, and otherwise
  left unchanged.
- =:last-message-at= is set to the appended message's =:created-at=.
- =:file= is set from store/session metadata when persistence is active.

Do not add new derived metadata for activity events, branch summaries, or
compactions in this feature.

Reserve full recomputation for load/replay and recovery paths where a single
O(n) pass is appropriate.

* Replay and load semantics
JSONL replay currently prepends entries and finalizes by reversing list fields.
Keep that strategy because it is efficient for load.

After reversing each replayed field, initialize its tail pointer in the same
finalization step:

#+begin_src elisp
(defun e-session--list-tail (items)
  (when items
    (last items)))

(defun e-session--finalize-list-field (session field tail-field)
  (let ((items (nreverse (plist-get session field))))
    (plist-put session field items)
    (plist-put session tail-field (e-session--list-tail items))))
#+end_src

Then refresh derived fields once per loaded session.

* Clear/reset semantics
When transcript state is cleared, reset both the list fields and their tail
fields:

- =:messages= nil
- =:messages-tail= nil
- =:activity-events= nil
- =:activity-events-tail= nil
- =:message-count= 0
- =:summary= nil
- =:last-message-at= nil

Branch summaries and compactions are not currently cleared by
=e-session-clear-messages= and should not be changed by this feature.

* Compatibility
The new tail fields are implementation details.

They should not be written into JSONL records or included in public index
entries. Existing session files remain loadable. Existing callers should still
see insertion-order lists through the same accessors.

* Implementation steps
** 1. Add list metadata constants and initialization helper
Add a mapping from append-only list fields to their tail fields, for example:

#+begin_src elisp
(defconst e-session--list-tail-fields
  '((:messages . :messages-tail)
    (:activity-events . :activity-events-tail)
    (:branch-summaries . :branch-summaries-tail)
    (:compactions . :compactions-tail)))
#+end_src

Add an internal destructive initializer that sets missing tail fields from the
current list values.

Acceptance criteria:

- The mapping and initializer are internal to =e-session.el=.
- No public API exposes tail fields.
- New sessions, replay-created sessions, and index stubs get consistent internal
  list state.

** 2. Add O(1) append helper
Add =e-session--append-list-item= and use it for append-only session lists.

Acceptance criteria:

- Message appends preserve insertion order.
- Activity event appends preserve insertion order.
- Branch summary and compaction appends preserve insertion order.
- No append path uses =(append existing (list item))= for these canonical lists.
- If a legacy/test session has a non-empty list and missing tail, the helper
  repairs the tail once and appends correctly.

** 3. Finalize replay tails after load
During replay finalization, reverse list fields and initialize tail fields.

Acceptance criteria:

- Persistent replay returns messages/activity in insertion order.
- Appending after replay attaches to the existing tail, not to a copied list.
- Replay still refreshes derived fields once per loaded session.

** 4. Maintain message-derived metadata incrementally
Add small helpers for message append metadata and file metadata.

Acceptance criteria:

- =:message-count= is correct after appends, reloads, and clears.
- =:summary= remains the first user string message.
- =:last-message-at= follows the most recent message timestamp.
- Persistent index metadata remains correct after appends and metadata-only
  changes.
- Append hot paths do not call full derived-field refresh once per append.

** 5. Update clear/reset behavior
Reset message/activity tail fields and message-derived metadata when messages
are cleared.

Acceptance criteria:

- Appending after clear starts a clean list.
- Activity events do not retain stale tail pointers after clear.
- Persistent replay after clear still returns empty messages/activity.

** 6. Add regression tests
Add tests that prove both behavior and the intended performance shape.

Suggested tests:

- append preserves order for messages and activity events;
- append preserves order for branch summaries and compactions;
- persistent replay preserves order, then append after replay preserves order;
- clear resets tails by appending after clear and checking no old entries return;
- derived fields stay correct without full refresh per append;
- helper-level structural test showing list head identity is preserved while the
  tail advances, if practical.

* Risks
- Destructive list mutation with =setcdr= can corrupt state if callers hold and
  mutate canonical list cells. Existing accessors return shallow copies; keep
  that contract.
- Tail pointers can become stale after any path that replaces a list field
  directly. Centralize list field replacement and finalization helpers.
- Index-backed unloaded sessions should not gain behavior that forces full
  transcript replay just to list sessions.
- Full derived refresh is still useful after load; removing it entirely could
  break legacy or partially replayed metadata.
- Defensively repairing missing tails with =(last items)= is O(n), but only on
  non-normal legacy/test paths where a non-empty list lacks its tail field.

* Follow-up candidates
- Add id/path-aware message and activity range APIs after message/turn identity
  and compaction clarify the right address model.
- Add activity aggregate metadata only when a concrete UI or diagnostic consumer
  needs it.
- Consider stronger entry-copy isolation separately if callers mutating returned
  entry plists becomes a real bug.

* Minimal acceptance for this feature
The feature is complete when:

1. Appending to messages, activity events, branch summaries, and compactions is
   O(1) with respect to the existing list length in normal initialized sessions.
2. Public accessors still return insertion-order data with unchanged shapes and
   shallow-copy list spines.
3. Replay/load initializes tail pointers correctly.
4. Clearing messages resets affected list and tail state.
5. Derived session metadata remains correct after append, load, and clear.
6. Tests cover order preservation, replay, append-after-replay, clear, and
   derived metadata correctness.
