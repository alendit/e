* Findings

** Open: live clear/reset leaves a dangling parent head

=e-session-clear-messages= clears =:messages= and =:activity-events= in live state but does not clear =:current-head-id=. The replay handler for =messages-cleared= does clear =:current-head-id=, so live state and the intended reset semantics diverge.

Impact:

- The first entry appended after a live reset receives =:parent-id= equal to the id of an entry that was removed from the session entry collections.
- The entry behaves like a path root because =e-session-current-path= cannot resolve the old parent, but it is not represented as a root: its =:parent-id= is non-nil.
- Persistent append after reset writes the dangling =parent-id=, and replay preserves it because the parent id is stored on the message record.
- This weakens the feature's parent invariant and future branching/compaction assumptions that parent ids reference reachable durable entries.

Suggested fix:

#+begin_src elisp
(defun e-session-clear-messages (store session-id)
  ...
  (plist-put session :messages nil)
  (plist-put session :activity-events nil)
  (plist-put session :current-head-id nil)
  ...)
#+end_src

Add a regression test that clears a session, appends a new message, and asserts:

- the new message has no =:parent-id=;
- =e-session-current-path= contains only the new root message;
- persistent replay preserves that root shape.

** Recommendation: audit non-entry JSONL records as event-log scope, not identity exceptions

The implementation clearly treats messages, activity events, branch summaries, and compactions as durable entries. Other append-only JSONL records are written into the same session log but are not first-class identifiable entries:

- =session=
- =session-info=
- =current-branch=
- =messages-cleared=

The sharper design rule should be: if a record is durable inside the session transcript/event log, it should represent a meaningful historical session event; if it is merely cursor, UI, view, cache, or derived-index state, it should not live in that log.

Implications for the current records:

| Record | Review recommendation |
|--------+-----------------------|
| =session= | Treat as the historical root event, or explicitly keep it as file header metadata outside the event chain. If it stays in the event log, giving it an =id= would make the root invariant cleaner. |
| =messages-cleared= | Treat as a historical transcript mutation. It should not be an anonymous side effect; it should either become a first-class event with an =id= and intentional parent semantics, or the next message should become a true new root after the clear. |
| =session-info= | Split by meaning. User-visible/context-affecting changes such as explicit name, model, reasoning effort, or project-root changes are historical session events and should be identifiable if stored in the log. Pure cache/index/derived display metadata should move out of the transcript log. |
| =current-branch= | If this is a checkout cursor or current UI/view state, it should not be in the session transcript log. Store it in sidecar state instead. If it is intended as a historical branch operation, rename/model it as such and give it entry identity. |

For future tree navigation, session comparison, replay diagnostics, time travel, and branch-aware context construction, the clean target is a unified durable event log containing only historical events. Every event in that log should be addressable by id. Ephemeral cursor/UI/cache state should be stored separately rather than exempted from identity.

This makes the reset bug more significant: =messages-cleared= is a historical transcript mutation, so treating it as anonymous mutable state creates ambiguity around whether later entries should parent to the old head, parent to the clear event, or start a new root.

** Test gap: synthetic branch path is not directly covered

The query helpers support reconstructing a path from an explicit optional =head-id= when entries have parent links, but the current tests do not appear to include a focused synthetic branch case from the feature testing plan. This is lower priority than the reset bug because the minimal acceptance criteria only require the current linear path, but a small synthetic branch test would protect the future branching intent.
