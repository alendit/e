* Review summary

The implementation delivers the core =output-style= capability design.

The first blocks reliable command discovery from a fresh package load.
The second weakens the promised error message for invalid configured style ids.

* UX and performance playthrough

- First use after a fresh package load is not coherent.
  =e.el= loads layer specs lazily and does not require =e-output-style=.
  The interactive commands are therefore unavailable until some path loads =e-base=.
  This is finding 1.
- Repeated set and describe through loaded helpers is responsive and scalar.
  The live probe returned the registered ids, active description, and resolved capability instructions immediately.
- Bad config recovery is understandable only at the generic capability-config level.
  It names the bad value but does not list valid style ids.
  This is finding 2.
- No profiling was needed.
  The feature performs small alist and plist lookups and does not add a redraw, network, or transcript hot path.

* Findings

** TODO Medium: Fresh package loads do not expose the output-style commands

*** Expected

The plan says the deliverable includes a command to switch the active style.
Step 1 explicitly says to modify the =e.el= require list or the reload list to include the new module.
Step 4 adds =M-x e-output-style-set= and =M-x e-output-style-describe= as user-facing entrypoints.

*** Observed

=e-dev.el= reloads =lisp/layers/base/e-output-style.el=, but =e.el= does not require it.
The default layer registry records =os-base= lazily with =:feature e-base=.
A fresh =-l e.el= package load leaves =e-output-style-capability-create= undefined.
By the same mechanism, =e-output-style-set= and =e-output-style-describe= are not discoverable as =M-x= commands until =e-base= has been loaded through another path.

Evidence:

#+begin_example
void-function
Symbol’s function definition is void: e-output-style-capability-create
#+end_example

*** Why it matters

The feature is meant to be selectable by command, not only by Lisp after a chat layer happens to load.
A user can install or start e, try =M-x e-output-style-set=, and fail to find the command.
That misses a direct plan contract and makes the feature feel absent.

*** Smallest useful fix

Load the command-owning module from the package entry path, or add proper autoloads and ensure the package autoload flow installs them.
The direct fix that matches the plan is to add =(require 'e-output-style)= to =e.el= after load-path setup and before startup completes.
Keep =e-base= requiring it too, because the layer factory depends on the capability constructor.

*** Verification needed after fix

Run a fresh-load probe that requires only =e.el= and confirms all three entrypoints exist:

      (fboundp 'e-output-style-capability-create))
#+end_src

Expected result:

#+begin_example

#+end_example

Then rerun focused ERT and warning-as-error compile.

** TODO Low: Invalid configured style ids do not list known ids

*** Expected

The plan says an unknown style id should raise a clear error naming the id and the known ids.
The resolver implements that behavior for direct calls.
The command helper also calls the resolver before writing config.

*** Observed

When the bad id comes through =e-capability-config=, the config option validator rejects it before =e-output-style--resolve= can emit its specific message.
The error is generic:

#+begin_example
e-capability-config-invalid-value
Invalid capability config value: output-style, :style, bogus
#+end_example

*** Why it matters

Directory-local and global config are core selection paths for this feature.
When a user mistypes a style id there, the error should tell them what ids are valid.
The current message names the bad value but does not help them recover.

*** Smallest useful fix

Use an option normalizer or explicit resolver path that preserves =e-capability-config= validation while letting unknown non-nil ids surface the resolver message.
One simple direction is to normalize =:style= by accepting nil, requiring a symbol, calling =e-output-style--resolve= for validation, and returning the original symbol.

*** Verification needed after fix

Add a focused test with =e-capability-config='((output-style :style bogus))= that asserts the error message contains =bogus= and at least one known id such as =hemingway=.
Rerun focused ERT.

* Architecture review

The dependency direction is sound.
The base layer depends on a base-layer capability module.
The capability module depends on the stable capability and config contracts.
No provider or shell code depends on output-style-specific behavior.

The side effect in =e-output-style--set-config= is appropriately at the interactive/config boundary.
The core resolver and constructor stay small and testable.

The main architectural gap is load ownership.
The command-owning module lives behind a lazy layer factory, but the commands are global user entrypoints.
Global entrypoints need either package-level load ownership or autoload ownership.

* Design self-check

1. If this work stopped here, would the project be better off than if it had not been built?
   Yes, because the core capability works and is tested, but the command discoverability gap blocks full acceptance.
2. What final direction does this move toward?
   It moves toward configurable system-channel guidance that remains provider-neutral and capability-owned.
3. Which decisions are likely to change, uncertain, or stable?
   The registry shape, single active style, and config ownership are stable enough for v1.
   The built-in style set and command UX may change, and those choices remain local to =e-output-style.el=.
4. What component owns this behavior, and why?
   =e-output-style.el= owns it because output style is a base guidance capability with its own registry and config contract.
5. Does this change increase or reduce coupling?
   It adds a narrow dependency from =os-base= to the new capability and avoids provider coupling.
6. Did any dependency start pointing the wrong way?
   No provider, adapter, or shell depends on the style implementation.
7. Could any side effect be moved outward into an adapter or shell?
   No, the only new side effect is writing =e-capability-config= from an interactive command.
8. Are the abstractions semantically real?
   Yes, style registry, config option, and capability are real extension seams tied to current behavior.
9. What compatibility expectations apply?
   No legacy compatibility path is needed.
   The capability is inert when unset, so adding it to =os-base= preserves default behavior.
10. What legacy code can we remove now?
    None found in this review.
11. Where are expected errors handled, and where do unexpected errors surface?
    Unknown style ids surface through the resolver in direct/helper paths.
    Config-path error quality needs the low-severity fix above.
12. What tests prove the core behavior independently of the full system?
    =test/e-output-style-test.el= proves the registry, constructor, helpers, and system-message contribution.
    =test/e-base-test.el= proves default-layer presence and ordering.
