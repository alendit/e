* Status

* Review summary

The implementation follows the intended architecture: =output-style= is a
capability owned by =lisp/layers/base/e-output-style.el=, is wired into
=os-base=, is inert by default, and contributes selected style prose as a
=static-prefix= system fragment at priority 260.

The review found no issue with the core capability composition or priority.
However, the user-facing command is not available after a fresh =(require 'e)=,
configured unknown style ids do not produce the planned known-style diagnostic,
and the interactive command currently updates only the global config while its
message can be wrong in a project with directory-local overrides.

- First-use command discovery was checked from a fresh package load. =M-x

  to set a style before starting or rebuilding chat state.

* Findings

** TODO High: =e-output-style-set= is not available after a fresh package load

*** Expected

The plan's deliverable includes a command to switch the active style, and Step 4
explicitly names =M-x e-output-style-set= and =M-x e-output-style-describe= as
the interactive user flow.

Step 1 also says to modify the =e.el= require list / reload list so the new
module participates in package loading.

*** Observed

=e-output-style= is loaded by =e-base=, and =e-base= is loaded lazily when the
=os-base= layer is created. A fresh package load does not define the command:

#+begin_example
(:feature nil :command nil)
#+end_example

Source evidence:

- =lisp/layers/base/e-base.el= requires =e-output-style=.
- =e.el= adds =lisp/layers/base= to =load-path= but does not require or autoload
  =e-output-style= or its commands.
- =lisp/dev/e-dev.el= reloads =e-output-style.el=, so reload coverage exists,
  but fresh command discovery does not.

*** Why it matters

The primary user-facing selector is not available to a user who has loaded =e=
but has not yet caused =os-base= to be instantiated. That makes the advertised
=M-x e-output-style-set= flow unreliable and blocks first-use configuration.

*** Smallest corrective direction

Add explicit autoloads in =e.el= for =e-output-style-set= and
=e-output-style-describe=, or require =e-output-style= during package startup if
that startup cost is acceptable. Add a startup smoke test that checks
=(commandp 'e-output-style-set)= after =(require 'e)=.

** TODO Medium: Invalid configured style ids bypass the planned known-style diagnostic

*** Expected

The plan says an unknown style id should produce a clear error naming the bad id
and listing known ids. The resolver does this for direct calls:
=e-output-style--resolve= signals =Unknown output style `ID'; known styles: ...=.

*** Observed

When an unknown id comes from =e-capability-config=, option validation fails
before the constructor calls the resolver:

#+begin_example
(e-capability-config-invalid-value
 "Invalid capability config value: output-style, :style, bogus"
 (output-style :style bogus))
#+end_example

The message names =bogus= but does not list known styles, so a typo in global or
directory-local config loses the actionable recovery path promised by the plan.

*** Why it matters

The most likely way users will hit unknown ids is through configuration. The
current diagnostic tells them the value is invalid but not which values are
valid.

*** Smallest corrective direction

Either:

- make the =:style= option validator check only nil/symbol shape and let
  =e-output-style--resolve= own unknown-style diagnostics during capability
  construction; or
- make the validator/normalizer signal an output-style-specific error that uses
  the same known-id message as =e-output-style--resolve=.

Add a regression test for configured =:style bogus= whose error message contains
both =bogus= and at least one known id such as =hemingway=.

** TODO Medium: The set/clear command updates global config while reporting an effective project result

*** Expected

The feature is documented as selectable globally and per project via
=e-capability-config=. The interactive command reports that new turns pick up the
selected style, and clearing reports that new turns use the default voice.

*** Observed

=e-output-style--set-config= mutates only the global =e-capability-config= value:

#+begin_src emacs-lisp
(setf (alist-get 'output-style e-capability-config)
      (plist-put plist :style style-id))
#+end_src

But =e-output-style--configured-id= resolves global config first and
then directory-local config, so a project-local =:style= overrides whatever the
command just wrote globally. In that situation:

- =M-x e-output-style-set= can say the style was set, while the current project
  still resolves to the directory-local style.
- Clearing writes =(:style nil)= globally, while the current project may still
  have an active directory-local style.

*** Why it matters

This is a misleading UX for the documented per-project workflow. Users can run
the command in a project, see a success message, and still get a different style
on the next turn because directory-local config has higher precedence.

*** Smallest corrective direction

Make the command's scope explicit. Viable small fixes include:

- message that the global default changed and then report the effective style
  for =default-directory= when a project override still wins;
- add separate global/project/harness-scope commands or prefix behavior; or
- set a harness/runtime override when invoked from an active chat buffer, if the
  intended command semantics are current-session/current-project rather than
  global default.

Add tests for the pure helper/message builder with a simulated directory-local
or higher-precedence project style.

** TODO Low: Adapter-specific output-style regression tests are incomplete

*** Expected

Step 5 acceptance asks for end-to-end system-channel assertions at the adapter
boundary:

- Anthropic: style prose appears in top-level =:system= and is absent from the
  =messages= array.
- OpenAI chat: style prose appears in a system-role message.

*** Observed

=test/e-output-style-test.el= proves the configured style becomes a
backend-neutral =:role 'system= context message and not a non-system turn. The
review's live probe confirms current Anthropic and OpenAI routing, but the
feature test suite does not pin that adapter behavior for output-style prose.

Existing adapter tests cover generic system-message handling, so this is a
coverage gap rather than an observed runtime failure.

*** Why it matters

The feature's central contract is that style guidance conditions the whole turn
through the system channel. A future adapter refactor could keep the
provider-neutral test green while accidentally routing output-style prose
through ordinary input for a specific provider.

*** Smallest corrective direction

Add focused adapter tests that build messages from a configured
=output-style= capability and assert the Anthropic and OpenAI wire shapes. For
OpenAI, match the current adapter contract precisely: Responses/Codex folds
system messages into =:instructions=, while Chat Completions emits system-role
messages.

* Architecture review

The main architecture direction is sound. The behavior is owned by a base-layer
capability, layers remain packaging, the harness does not special-case style,
and providers continue to consume ordinary backend-neutral system messages.

The open issues are at the presentation/config boundary rather than in the core
composition model:

- command discoverability needs a package-startup/autoload decision;
- command scope needs to say whether it edits a global default, project config,
  or active harness/session runtime config;
- configured invalid values should preserve capability-owned diagnostics instead
  of surfacing only a generic config validator error.

* Design self-check

1. If this work stopped here, the project is better off because it has a clean
   output-style capability path and tests, but the user-facing selector still
   needs polish before acceptance.
2. It moves toward capability-owned model behavior that is configured through
   the shared =e-capability-config= contract.
3. Stable decisions: output style is one capability and uses system-channel
   instructions. Uncertain decision: command scope for global/project/session
   selection; that uncertainty is currently hidden behind a global write and
   should be made explicit.
4. =e-output-style.el= owns the behavior cohesively: registry, resolution,
   config option, capability construction, and commands.
5. Coupling stays low; =os-base= depends on the capability owner and providers
   see ordinary system messages.
6. No stable policy was made to depend on concrete providers or shells.
7. The remaining command side effect could be moved outward or made explicit as
   a presentation/action choice if it needs active harness/session semantics.
8. The abstraction is semantically real: a named style resolves to prose carried
   by one capability.
9. Compatibility expectation: nil/unset remains inert; existing adapters keep
   their system-message contracts.
10. No obvious legacy path was introduced.
11. Expected unknown direct style errors surface well; expected unknown
    configured style errors currently surface too generically.
12. Core behavior is covered by focused tests; adapter-boundary and fresh-startup
    command tests should be added.
