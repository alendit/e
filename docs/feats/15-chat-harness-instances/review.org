* Review summary

Feature 15 adds the intended user-facing chat target catalog and wires it into
new-chat, resume, context insertion, overview rows, and live buffer ownership.
The implementation is small in core, keeps live harness objects in the existing
registry, preserves direct harness injection for tests, and has good focused
coverage for the per-instance-store case.

The review found one blocking ownership ambiguity before this can be marked
=Done=: when two chat harness instances share the same session store, every
session is listed once for each instance.  That is not just duplicated UI; it can
resume the same transcript through the wrong configured target/provider.  The
planned user config for a second default-style chat instance calls
=e-default-chat-harness-create= without a separate session store, so this is a
likely real configuration rather than an edge case.

The follow-up slice resolves that ambiguity.  New chat sessions created through
a configured chat instance now persist =:harness-instance-id= metadata.  Resume,
context insertion, and overview rows all use one candidate policy: owned
sessions appear only under their persisted owner; legacy unowned sessions in
shared stores appear once under the default instance; legacy unowned sessions in
unique per-instance stores remain listed under that instance.  The obsolete
=e-chat--context-session-picker= helper was removed, and =e-chat-new= now
documents that its prefix argument both prompts for a target and uses the pop
display path.

* What looks good

- =e-harness-instances= is a narrow catalog over =e-harness-registry= rather
  than a second live harness owner.  It validates ids/kinds, registers lazy
  factories with the low-level registry, tracks kind-scoped defaults, and sorts
  catalog output deterministically.
- Default startup registers the legacy =:chat-default= spec as a default chat
  instance and still registers the low-level factory, so older registry-based
  code paths remain available.
- =e-chat-open= accepts an optional =:instance-id= while still allowing direct
  =:harness= injection.  Buffer reuse checks both harness and instance id, so
  colliding session ids in different stores do not accidentally reuse the wrong
  live buffer.
- =e-chat-harness-instance-id= gives attached chat buffers, overview rows, and
  context insertion paths enough ownership metadata for the per-instance-store
  case.
- =e-chat-new= prompts when multiple chat instances are configured and falls
  back to the default/legacy path when no instances are configured.
- Resume, context picker, latest-context fallback, and overview rendering all use
  the cross-instance candidate model when chat instances are registered.
- Overview read markers include the instance id when available, avoiding marker
  collisions for equal session ids across different configured targets.
- The focused tests cover registry behavior, new-chat selection, resume across
  two instances, overview open with colliding session ids, and context insertion
  across instances.

* Findings

** TODO High: shared session stores duplicate every session across all chat instances

=e-chat--session-candidates= iterates over every registered chat instance,
resolves each instance to a harness, then pushes every =e-harness-session-list=
entry from that harness.  This assumes each chat instance has a disjoint session
store.  The plan explicitly called out that current default harnesses share
=e-default-session-store= and that metadata tagging may be needed to avoid
ambiguity.

The implemented config migration example encourages a second instance like:

#+begin_src emacs-lisp
(e-harness-instance-register
 :id :chat-opus
 :name "Opus 4.7"
 :kind 'chat
 :factory (lambda ()
            (e-default-chat-harness-create
             :provider 'local-go-llm-proxy-opus))
 :metadata '(:model "claude-opus-4-7"))
#+end_src

Because =e-default-chat-harness-create= defaults =:sessions= to
=e-default-session-store=, this second instance naturally shares the same store
as =:chat-default=.  In that configuration, resume/context/overview will show
all sessions under both targets.  Selecting the duplicate for the other target
can continue the same transcript with a different provider/model than the one
that originally owned it.

Expected behavior should make session ownership explicit.  Possible fixes:

- store an owning harness-instance id in session metadata when sessions are
  created through an instance, then filter cross-instance candidate lists by that
  owner;
- or assign separate default session stores per configured instance;
- or explicitly model shared-store sessions as provider-neutral and show them
  only once, with a deliberate target selection step when continuing.

Add regression coverage for at least:

1. two chat instances backed by the same store do not produce duplicate resume
   candidates for one session;
2. a session created through =:chat-beta= is resumed/opened with =:chat-beta=
   ownership, even when another instance shares the same store;
3. overview rows and context picker use the same ownership rule as resume.

** TODO Medium: new sessions do not persist their creating harness instance

=e-chat-open :new-session t :instance-id ...= creates the session through
=e-chat--create-session=, whose metadata currently includes only project root.
The live buffer tracks =e-chat-harness-instance-id=, but the persisted session
record does not remember which configured target created it.

This is closely related to the shared-store bug.  Without persisted ownership, a
future Emacs process or reloaded registry cannot distinguish which configured
chat target a shared-store session belongs to.  If the chosen fix is metadata
ownership, session creation should write that owner and older sessions should get
an explicit compatibility policy.

** TODO Low: stale =e-chat--context-session-picker= remains after the new candidate model

=e-chat--context-session-picker= still implements the old single-harness session
picker, but the context commands now use =e-chat--context-session-target=.  A
repository grep found no source caller outside generated/.eldev artifacts and an
old bug investigation note.

This is not a behavior blocker, but removing the obsolete helper would reduce
confusion and avoid future callers accidentally reintroducing single-harness
context selection.

** TODO Low: =e-chat-new= prefix semantics now combine target prompting and side-window display

=e-chat-new= still documents the prefix argument as the pop display path, and it
also passes that same prefix through to =e-chat--select-chat-instance= to force
prompting.  This preserves the existing prefix-display behavior and satisfies
the plan's "prefix prompts" direction, but it means there is no way to force a
one-instance target prompt without also using pop display.

This is acceptable for now if intentional, but the command docstring should make
that combined behavior explicit or the prompt/display controls should be split.

* Architecture review

The catalog boundary is directionally sound: =e-harness-instances= owns
configured-target metadata, the low-level registry still owns lazy live harness
resolution, and =e-chat= owns chat-specific selection/presentation.  The core
catalog does not depend on providers or UI buffers.

The main design gap is session ownership.  The feature introduced configured
runtime targets, but persisted chat sessions remain target-agnostic.  That is
fine only if every instance has a separate store.  Because the default chat
factory encourages a shared store, ownership needs to be modeled either in the
session domain or in an explicit shared-store policy instead of being inferred by
which harness happened to enumerate the same store.

* Design self-check

1. If this work stopped here, the project is better than before for users with
   one instance or disjoint per-instance stores, but not safe enough to mark done
   because shared default-style stores duplicate and mis-own sessions.
2. This moves toward a generic configured runtime-target catalog that lets chat
   commands work across backends without backend-specific UI commands.
3. Stable decisions: configured chat targets need ids/names/kinds and lazy
   harness resolution.  Uncertain decision: whether sessions belong to a target,
   a provider-neutral store, or a per-instance store.  That uncertainty is not
   yet local enough because candidate enumeration currently assumes disjoint
   stores.
4. =e-harness-instances= owns target catalog metadata.  =e-chat= owns chat
   selection and presentation.  Session ownership likely belongs in the chat
   session/session-store boundary or in a narrow chat target policy helper.
5. Coupling is reduced for UI commands, but the missing ownership model couples
   correctness to an unstated storage topology assumption.
6. No dependency from core catalog policy to concrete providers or presentation
   was found.
7. Side effects remain in startup registration, harness resolution, session
   creation, and shell commands.
8. The configured-target abstraction is semantically real.  The session
   candidate abstraction is not yet substitutable across shared-store and
   per-instance-store implementations because it duplicates shared sessions.
9. Compatibility keeps =e-chat-default-harness-id= and direct harness injection.
   Older sessions without target metadata need an explicit compatibility policy
   if ownership metadata is added.
10. =e-chat--context-session-picker= appears obsolete and can likely be removed.
11. Expected missing-instance and missing-harness cases surface as =user-error=.
   Shared-store ambiguity is currently not handled as an expected condition.
12. Tests cover registry behavior and per-instance-store chat flows.  Missing
   tests are shared-store ownership/deduplication and persisted target ownership
   across restart/reload boundaries.
