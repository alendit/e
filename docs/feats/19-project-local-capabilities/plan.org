#+title: Project-Local Capabilities
#+startup: overview

* Goal

Let a repository contribute its own =e= *capabilities* — not just static
=AGENTS.md= text and read-only =SKILL.md= bodies, but real model-facing tools,
resource methods, and (optionally) stateful behavior — discovered and loaded
the way =agents-std-context= already discovers project =AGENTS.md= and
=.agents/skills=. EDIT: Also shells.

Motivating consumer: the =grimoire= knowledge/agenda repo wants a repo-scoped
"topic" capability (state in Org =:PROPERTIES:= drawers, =index.org=
regeneration, =topic-new/close/open=, =daily=, =graduate=) that ships *inside
the repo* and activates only when =e= is operating in that repo.

* What exists today (inventory)

- =agents-std-context= (=lisp/layers/agents/e-agents-std-context.el=) walks
  ancestor directories from the project root and discovers:
  - project =AGENTS.md= files -> a context provider (system message).
  - project =.agents/skills/*/SKILL.md= -> =e-skill-spec= records advertised in
    a catalog message and exposed as read-only
    =e://agents-std-context/skills/project/<slug>= resources.
- Skills are *guidance only*. A =SKILL.md= can describe a workflow but cannot
  register a tool, a resource method, or mutate state. The =reader= just returns
  the file body.
- Project root comes from =e-harness-project-root= (session metadata, else
  harness default). Layer factories that accept a =directory= arg
  (=e-layer--factory-accepts-directory-p=) get it at construction.
- Capabilities (=lisp/core/e-capabilities.el=) already support everything a
  project capability would need: =instructions=, =tools=, =resource-methods=,
  =resources=, =context-providers=, =actions=, =hooks=, =config-options=.
- Layers are registered in =e-default-layer-specs= and activated by
  =e-default-chat--activate-configured-layers=.

*Gap:* no path from "files in a repo" to "an activated capability that
contributes tools/resource-methods/state". Skills stop at read-only text.

* Design

** Discovery: =.agents/capabilities/= alongside =.agents/skills/=

Comment: shouldn't probably be .agents. Go with .e

Mirror the existing skills-discovery walk. From the project root, walk ancestors
collecting =.agents/capabilities/= directories. Each subdirectory is one
project-local capability:

#+begin_example
<repo>/.agents/capabilities/<id>/
  capability.el            # loader entrypoint (see contract)
  CAPABILITY.md            # optional: description, frontmatter id/name
  skills/<slug>/SKILL.md   # optional: capability-scoped skills
#+end_example

Discovery is *opt-in by presence*: a repo with no =.agents/capabilities/=
behaves exactly as today.

** Loading contract (trust-gated)

Loading project elisp is a real trust boundary (arbitrary code from a repo).
Contract:

1. Discovery finds =capability.el= files but does *not* load them automatically.
2. A user-controlled allowlist (=e-project-capabilities-allowed-roots=, a
   persisted defcustom) gates loading per project root. Unknown roots are
   reported, not executed.
3. For an allowed root, =e= loads =capability.el=. The file makes a
   registration call when loaded:
   #+begin_src elisp
   (e-project-capability-register
    :id 'topic
    :factory (lambda (directory) (grimoire-topic-capability-create directory)))
   #+end_src
   The factory is called with the discovered directory and must return an
   =e-capability= (built via =e-capability-create=).
4. The returned capability is wrapped in an anonymous layer
   (=e-harness-activate-capability=) and participates in normal turn context,
   tool dispatch, and resource registration like any built-in capability.

** Project-defined layers (namespaced)

*Can projects define their own layers?* Yes. The only real objection to repo-defined
layers is id collision in the global registry (=e-layer--registry=,
=e-default-layer-specs=) — a repo could shadow a built-in id (=e=, =web=,
=os-base=, ...). That is avoidable by *namespacing*: project layer ids are
mechanically prefixed with the project (e.g. =project/<repo>/<id>=) at
registration time, so they cannot collide with built-ins or with other
projects. With that, a repo may contribute *multiple* layers, each grouping a
cohesive set of capabilities, just like built-in layers do.

What stays harness-owned is the *trust gate*, not the layer seam: discovered
repo layers are registered but only *activated* for allowlisted roots, and
activation reports (never silently overrides) on collision after namespacing.
The registration call a =capability.el= makes returns one or more
=e-layer= objects (or capabilities the loader wraps into a project layer); the
harness assigns/validates the namespaced id rather than trusting a
repo-supplied raw id.

A built-in discovery layer (=project-local=, feature =e-project-local=,
factory =e-project-local-layer-create= taking =directory=) drives this. It:

- discovers =.e/capabilities/= from the (session) project root,
- filters to allowlisted roots,
- loads each =capability.el=, collecting whatever layers/capabilities the repo
  registers,
- namespaces each repo-supplied layer id (=project/<repo>/<id>=) and registers
  the resulting layers, reporting any post-namespacing collision instead of
  overriding,
- activates the namespaced layers for the allowlisted root,
- additionally folds in any capability-scoped =skills/= via the existing
  skill-spec machinery (factor the =e-agents-std-context= skill helpers into a
  shared =e-skills= discovery util to avoid duplication).

Because layer factories are re-run on =e-default-chat-sync-harness-layers=,
edits to project =capability.el= refresh on reload (consistent with =e='s live
reload direction).

** Config + precedence

- Reuse =e-capability-config= so a project capability can declare
  =config-options= and be tuned per project via existing config plumbing.
- Project capabilities load *after* built-in layers so they can rely on base
  resources (=file://=, =buffer://=) but cannot silently replace core tools;
  id collisions with built-ins are an error surfaced to the user, not a silent
  override.

** Boundary rules honored

- Harness still owns lifecycle/dispatch; the new layer is just another
  capability source.
- Side effects (loading elisp, touching the allowlist) live in the layer/shell,
  not in core.
- Trust decision is explicit and user-owned; =e= does not auto-execute repo
  code. Failures surface (no silent swallow), per =AGENTS.md= error policy.

* Open design questions

1. *Allowlist UX*: prompt-on-first-encounter (like =dir-locals= "mark safe")
   vs. a pure defcustom the user edits. Leaning prompt + persist.
2. *Entrypoint convention*: registration call (chosen above) vs. fixed
   function-name-by-id. Registration is more flexible and avoids name templating.
3. *Sandboxing*: none initially (it is the user's own repo), but document that
   =capability.el= runs with full Emacs privileges.
4. *Skills-only fast path*: capability-scoped =skills/= without a =capability.el=
   should still work (pure guidance), reusing existing discovery.

* Verification

- ERT: discovery walks ancestor =.agents/capabilities/= correctly; empty repo
  is a no-op.
- ERT: allowlist gating — disallowed root yields no loaded capability and a
  reported entry; allowed root yields an activated capability.
- ERT: a fixture =capability.el= contributes a tool + resource method that show
  up in =e-harness-tools= / =e-harness-resources=.
- Live: load the grimoire topic capability in a scratch repo, confirm =topic-*=
  tools are dispatchable and =index.org= regenerates.

* Status

In-progress.

* Implementation notes

Implemented as a single built-in discovery layer rather than per-repo
namespaced layers in the global registry.  The registry contract
(=e-layer-create-registered=) requires a registered factory to return a layer
whose id equals the registered id, so repo-supplied raw layer ids cannot be
returned through that seam without weakening the contract.  Instead:

- =project-local= (feature =e-project-local=, factory
  =e-project-local-layer-create= taking =directory=) discovers
  =.e/capabilities/= from the project root, loads allowlisted =capability.el=
  loaders, and bundles the resulting capabilities into one =project-local=
  layer.  This keeps the trust gate and id-collision handling harness/shell
  owned and avoids a speculative per-repo namespacing layer that no consumer
  needs yet.
- Capability ids are de-duplicated within the layer (nearest directory wins);
  collisions are reported via =message=, never silently overridden.
- =.e/capabilities/= replaces the =.agents/capabilities/= sketch per the
  inline review comment ("go with .e").
- Skill discovery helpers were factored into shared =e-skills-*= functions in
  =lisp/core/e-skills.el=; =e-agents-std-context= now delegates to them, so
  capability-scoped =skills/= reuse the same machinery without duplication.
- Capability-scoped skills register under =e://<id>/skills/project/<slug>=.

Per-project config tuning via =e-capability-config= is available to project
capabilities because they declare =config-options= like any built-in
capability; no project-specific config plumbing was added.

* Self-check (design-sensitive review)

1. *Better off if stopped here?* Yes: repos can now ship real tools, resource
   methods, and skills that activate only in their own tree, which is the
   substrate the grimoire topic capability needs.
2. *Final direction:* live-configurable, repo-contributed capabilities loaded
   the way project AGENTS.md/skills already are.
3. *Likely/uncertain/stable decisions:* The registration-call entrypoint and
   single-layer bundling are the stable choices.  The allowlist UX (defcustom
   vs. prompt-and-persist) is the uncertain decision and is kept local to the
   layer via =e-project-local-allowed-roots=; switching to a prompt later does
   not touch core.  No speculative per-repo namespacing layer was added.
4. *Owning component:* the =project-local= layer owns discovery, the trust
   gate, and capability/layer wrapping (one reason to change: how repo
   capabilities are discovered/loaded).  The harness still owns lifecycle and
   dispatch.
5. *Coupling:* low.  Core gained no dependency on the new layer.  The shared
   =e-skills= discovery helpers are consumer-shaped and depended on by both
   discovery layers.
6. *Dependency direction:* unchanged.  Stable core (=e-capabilities=,
   =e-skills=, =e-harness=) is depended on by the new layer, not vice versa.
7. *Side effects:* loading repo elisp and reading the allowlist live in the
   layer, not core.
8. *Real abstractions:* the registration record and capability factory are
   semantically real; project capabilities substitute for built-in
   capabilities without narrowing preconditions.
9. *Compatibility:* none required; this is new surface.  No legacy paths added.
10. *Legacy removal:* duplicated frontmatter/skill discovery in
    =e-agents-std-context= was removed in favor of shared =e-skills= helpers.
11. *Errors:* the allowlist gate reports unallowed/duplicate ids via =message=;
    a factory returning the wrong type or id signals and surfaces to the top
    rather than being swallowed.
12. *Tests:* =test/e-project-local-test.el= proves discovery, allowlist gating,
    duplicate-id precedence, the load-context guard, bad-factory signalling,
    and tool/resource/skill contribution independently of the full chat shell.

* Relationship to grimoire

This feature is the substrate. The grimoire topic capability is the first real
consumer: it ships as =<grimoire>/.agents/capabilities/topic/capability.el=
contributing topic state I/O + index regeneration tools and =topic-*= /
=daily= / =graduate= skills.
