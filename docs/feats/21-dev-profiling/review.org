* Summary

Implemented opt-in developer profiling traces for normal E usage. Developers can
start a trace, use chat/harness/session paths normally, stop tracing, and inspect
raw JSONL or an aggregate report.

Implemented pieces:

- =lisp/dev/e-dev-profile.el= owns profiler state, JSONL writing, thunk/macro
  measurement helpers, report aggregation, and interactive commands.
- =e-dev-reload= reloads the profiler module.
- =e.el= autoloads the developer profiling commands.
- =.e/dev-profiles/= is ignored as generated local trace storage.
- Session persistence records =session.append-record= and
  =session.write-index= spans when profiling is enabled.
- Harness records =harness.context= and =harness.tool-start= spans.
- Chat records =chat.status= and =chat.activity-redraw= spans.

* Review notes

- Current instrumentation intentionally covers the first useful normal-usage
  boundaries, not every possible rendering or backend callback.
- Core/session/chat modules avoid a hard runtime dependency on the dev module by
  calling profiling helpers only when =e-dev-profile= is loaded and enabled.
- Raw traces are generated under ignored local storage and are not part of
  session transcript persistence.

* Architecture self-check

1. If this work stopped here, E is better off because future performance work can
   start from live trace evidence rather than one-off probes.
2. This moves toward a snappier interactive runtime by making slow boundaries
   observable during normal dev testing.
3. Stable decisions: opt-in profiling, JSONL records, scalar metadata. Likely
   changing decisions: exact event set and report formatting; both are local to
   the dev profiler and call sites.
4. The dev profiler owns trace policy, storage, and reporting. Harness, session,
   and chat own only small boundary call sites.
5. Coupling is limited by dynamic availability checks; normal runtime modules do
   not require the dev profiler.
6. Stable runtime policy does not depend on a concrete profiling adapter at load
   time; instrumentation is inert unless the dev module is loaded and enabled.
7. File writes and reporting side effects stay in =lisp/dev/e-dev-profile.el=.
8. The profiling helper abstraction is real: call sites need only "measure this
   thunk with scalar metadata".
9. Compatibility expectation: disabled profiling preserves existing runtime
   behavior and public APIs. No obsolete public paths were added.
10. No legacy profiling code exists to remove.
11. Expected report/no-event cases are handled. Unexpected errors in measured
    bodies are recorded and re-signaled.
12. Tests prove recorder behavior independently and verify integration at
    session, harness, and chat boundaries.
