* Review summary

Feature 21 adds a useful opt-in developer profiling workflow: traces are written
as JSONL under ignored local storage, call sites can measure spans through a
small dev-owned helper, and reports aggregate slowest events and totals.  The
review verified the recorder and harness/session integration tests, plus a live
probe in the running Emacs.

The implementation is useful enough for performance investigations, but a few
plan items remain incomplete before marking =Done=: prompt/turn lifecycle and
some chat presentation spans are not instrumented, trace file names can collide
when two traces start in the same second, and the report omits the wall-clock
range promised by the plan.

* What looks good

- =e-dev-profile.el= owns profiling state, JSONL writing, measurement,
  report-data aggregation, report rendering, and interactive commands.
- =e-dev-profile-record= records only scalar metadata supplied by call sites; the
  tests explicitly assert that prompt/content/tool-result fields are not added by
  the helper.
- =e-dev-profile-measure-thunk= preserves return values, records durations, and
  records error metadata before re-signaling unexpected errors.
- Generated traces are ignored via =.e/dev-profiles/= in =.gitignore=.
- Public developer entrypoints are autoloaded from =e.el=, and =e-dev-reload=
  knows about =lisp/dev/e-dev-profile.el=.
- Runtime modules avoid a hard load dependency by checking whether profiling
  helpers are fbound and enabled before measuring.
- Current instrumentation covers useful first boundaries:
  - =session.append-record= and =session.write-index=;
  - =harness.context= and =harness.tool-start=;
  - =chat.status=, =chat.activity-redraw=, =chat.render-turn-transient=,
    =chat.render-event=, =chat.insert-entry=, and
    =chat.refresh-session-display=;
  - =context-status.text=.

* Findings

** TODO Medium: prompt/turn lifecycle spans from the plan are not instrumented

The plan's first instrumentation set called out harness prompt/turn entrypoints,
backend callback handling where local, message append, and durable activity
append.  Current harness instrumentation covers =harness.context= and
=harness.tool-start=, but not =e-harness-prompt-async=, =e-harness-prompt=,
=e-harness--run-prompt-turn= / async turn start, provider callback handling, or
harness-owned message/activity append boundaries.

This is not required for the current probe to be useful, but it falls short of
the stated feature scope.  Either add the missing spans with narrow tests or
update the plan/review to explicitly defer them.

** TODO Medium: chat instrumentation does not cover composer operations called out in the plan

The plan specifically listed composer insert/delete/restore spans.  Current chat
instrumentation covers status, activity redraw, render-event, transient render,
insert-entry, and refresh-session-display, but not the composer
insert/delete/restore helpers as distinct events.

Add those spans/tests or document that composer profiling is deferred.

** TODO Medium: starting two traces in the same second reuses the same file path

=e-dev-profile--timestamp-file-name= formats names at one-second granularity.
Starting two traces quickly can return the same path and append unrelated trace
sessions into one file.

Evidence from this review pass:

#+begin_example
:same t
:first  .../20260614-222105.jsonl
:second .../20260614-222105.jsonl
#+end_example

Expected behavior: trace file names should be unique per start call, e.g. by
including subsecond time, a sequence, or a random suffix.  Add a regression test
that two immediate starts produce distinct paths or deliberately rotate/close the
previous trace according to a documented policy.

** TODO Low: report formatter omits wall-clock trace range

The plan says reports should include the wall-clock trace range.  The report data
records =:started-at= and =:finished-at=, but =e-dev-profile-format-report=
prints only trace path, event count, aggregates, and slowest spans.

Add a compact range/duration line to the report and a test for it.

** TODO Low: =e-dev-profile-stop= opens a report as a side effect

=e-dev-profile-stop= disables profiling and then opens the report buffer when a
trace file exists.  That may be fine for an interactive command, but the plan
phrases stop as disabling/flushing and separately provides =e-dev-profile-report=.
Consider splitting the implementation into a noninteractive stop helper and an
interactive command that opens the report, so scripted probes can stop without
presentation side effects.

* Architecture review

The architecture is sound for an opt-in dev facility:

- Profiling policy, storage, JSONL shape, and reports are isolated in
  =lisp/dev/e-dev-profile.el=.
- Runtime call sites use dynamic availability checks and do not require the dev
  module during normal operation.
- Trace records contain bounded scalar metadata, not prompts, assistant content,
  tool results, or whole context payloads.
- File-writing side effects stay in the dev module.

The main remaining design choice is scope: either complete the originally listed
instrumentation set or explicitly declare this slice as a first useful subset.
Do not expand the profiler into always-on telemetry or session transcript state.

* Design self-check

1. If this work stopped here, E is better off because developers can collect
   real profiling traces during normal usage.
2. This moves toward evidence-guided performance work for chat, harness, session,
   and presentation paths.
3. Stable decisions: opt-in profiling, JSONL traces, scalar metadata, local
   ignored storage.  Likely-to-change decisions: exact event set and report
   formatting; those are local to =e-dev-profile= and call sites.
4. =e-dev-profile= owns tracing policy/storage/reporting.  Harness/session/chat
   own only small measurement boundaries for their work.
5. Coupling is limited by dynamic fbound/enabled checks and small call-site
   wrappers.
6. No stable runtime module depends on a concrete profiling adapter at load time.
7. File writes and report buffers are dev-module side effects.
8. The measurement helper abstraction is real: callers provide an event/options
   plist and a thunk/body.
9. Compatibility expectation: disabled profiling preserves behavior and public
   APIs.  Existing runtime semantics are preserved in the passing focused tests.
10. No legacy profiling code exists to remove.
11. Expected empty/no-trace cases are handled.  Unexpected errors inside measured
    bodies are recorded and re-signaled.
12. Tests prove recorder behavior plus session/harness integration.  Remaining
    narrow tests should cover unique trace names, report range formatting, and
    any additional spans chosen from the original scope.
