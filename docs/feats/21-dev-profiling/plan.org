#+title: Developer Profiling Traces
#+startup: overview

* Status

Planned.

* Goal

Add an opt-in developer profiling facility that records structured timing traces
while E is used normally in a live Emacs session, so slow chat, harness, tool,
rendering, and persistence paths can be investigated from real interaction
instead of one-off benchmark probes.

* Motivation

Feature 16 removed several known turn-finish latency hotspots, but its benchmark
handoff stayed ad hoc. E needs to feel snappy across normal usage, and the next
slow path may be in context assembly, provider callbacks, tool lifecycle, chat
rendering, session persistence, canvas integration, or mode-line refresh. A
developer should be able to enable tracing, use E naturally for a few turns,
stop tracing, and inspect a compact report plus the raw trace.

* Non-goals

- Do not add always-on production telemetry.
- Do not persist profile events into user session transcripts.
- Do not record prompt text, assistant text, tool result bodies, full context
  payloads, or broad buffer-local/session state.
- Do not solve every known latency issue in this feature. The output is
  instrumentation and a report that makes those fixes grounded.
- Do not introduce byte-compilation, startup optimization, or broad asynchronous
  rendering changes here.

* Current behavior

- Developers can use Emacs built-ins such as =benchmark-run=, =elp=, and
  =profiler-start= manually, but E does not provide a repo-specific workflow for
  tracing normal chat/canvas/tool usage.
- Existing latency docs include scalar probes for specific suspected hotspots,
  but there is no reusable trace file or aggregate report.
- =e-dev-reload= owns live development reload support. It currently reloads
  development helpers from =lisp/dev/e-dev.el=.
- The harness emits lifecycle events and the chat shell subscribes/render them,
  but there is no timing wrapper that correlates slow spans with session/turn
  identifiers.

* Accepted decisions

** Use opt-in dev tracing, not always-on runtime telemetry

Profiling is developer-operated. Disabled tracing should add only cheap predicate
checks at instrumented boundaries. Enabled tracing writes JSONL records to a
developer trace file while normal E commands continue to run.

Why:

- The performance problem is real, but the first useful artifact is evidence.
- Runtime semantics stay unchanged when profiling is off.
- Trace overhead, privacy, and storage policy remain contained in the dev layer.

** Keep profile traces outside source-controlled docs by default

Raw traces should be written under an ignored runtime/cache path, not under
=docs/feats/=. The implementation should use a default such as:

#+begin_example
.e/dev-profiles/YYYYMMDD-HHMMSS.jsonl
#+end_example

If =.e/= is not already ignored, add the narrow ignore entry that keeps generated
developer traces out of commits.

Feature reviews can quote selected report output in =review.org= when needed,
but raw traces are local artifacts.

** Use structured JSONL events with scalar metadata

Each record should include:

- =timestamp= :: floating-point seconds from =float-time=.
- =event= :: stable symbol/name such as =harness.context= or
  =chat.render-transcript=.
- =duration-ms= :: elapsed wall-clock milliseconds for measured spans.
- =session-id= :: optional short or full session id when cheaply available.
- =turn-id= :: optional turn id when cheaply available.
- =buffer-name= :: optional buffer name for presentation shell spans.
- =metadata= :: optional compact plist/alist with counts or ids only.

Do not include prompt text, assistant content, tool output bodies, complete
context messages, full activity payloads, or unbounded object prints.

** Build a small dev-owned recorder module

Create =lisp/dev/e-dev-profile.el=. It owns:

- enabled state;
- active trace file path;
- latest trace file path;
- event write helper;
- measured-span helper/macro;
- report parsing and aggregation helpers;
- interactive commands.

Core, harness, session, and chat code may call narrow helpers from this module,
but profiling policy and trace/report formatting stay in =lisp/dev/=.

** Instrument useful boundaries first

The first instrumentation set should cover boundaries that affect perceived
snappiness and were involved in prior latency work:

- harness turn lifecycle:
  - context build;
  - prompt/turn entrypoints;
  - backend callback handling where the callback boundary is local;
  - tool lifecycle dispatch;
  - message append;
  - durable activity append;
- session persistence:
  - JSONL append record;
  - index write;
  - session load/list paths when used by visible UI;
- chat presentation:
  - status updates;
  - running activity redraw flush;
  - transcript refresh/render;
  - composer insert/delete/restore.

Keep each span small enough to identify an owning component. Avoid wrapping a
large top-level turn in a way that hides which child boundary was slow.

** Reports should be useful inside Emacs

Provide =e-dev-profile-report= to open a human-readable buffer summarizing the
latest or selected trace:

- total event count;
- wall-clock trace range;
- slowest individual events;
- aggregate count/total/average/max by event name;
- optional grouping by session id and turn id when present.

The report should be compact enough to paste into =review.org=.

* Proposed API

Developer commands:

- =e-dev-profile-start= :: create the trace directory if needed, open a new JSONL
  trace file, enable profiling, and return the trace path.
- =e-dev-profile-stop= :: disable profiling, flush/close any active state, and
  display a report for the latest trace.
- =e-dev-profile-open-latest= :: open the latest trace file.
- =e-dev-profile-report= :: render an aggregate report for the latest trace or a
  selected trace path.

Internal helpers:

- =e-dev-profile-enabled-p=
- =e-dev-profile-record=
- =e-dev-profile-measure=
- =e-dev-profile-report-data=
- =e-dev-profile-format-report=

If a macro is introduced for lower call-site noise, keep it private to the dev
module unless tests show a function is too awkward for the instrumentation sites.

- Create =lisp/dev/e-dev-profile.el= for recorder state, JSONL writes, report

- Modify =lisp/dev/e-dev.el= so =e-dev-reload= loads =e-dev-profile.el=.
- Modify =e.el= if a public autoload or package entrypoint is needed for
  developer commands.
- Modify =lisp/core/e-harness.el= to wrap selected harness spans.
- Modify =lisp/core/e-session.el= to wrap session persistence spans.
- Modify =lisp/shells/chat/e-chat.el= to wrap selected presentation spans.
- Add =test/e-dev-profile-test.el= for recorder/report behavior.
- Update =test/e-dev-test.el= if reload expectations need to include the new dev
  file or command availability.
- Add a narrow generated-trace ignore rule if needed.

* Compatibility expectations

- Disabled profiling must preserve current runtime behavior.
- Existing public session, harness, and chat APIs should not change.
- Developer commands may be new public interactive commands, but trace recorder
  internals should remain dev-owned.
- Trace JSONL shape should be stable enough for reports and review snippets, but
  it is not yet a user-facing persistence format.

* Risks

- Profiling can perturb the timings it measures. Mitigate by making disabled
  paths cheap and enabled records compact.
- Instrumentation call sites can accidentally allocate metadata even when
  disabled. Guard expensive metadata construction behind
  =e-dev-profile-enabled-p=.
- Broad trace metadata can leak user content. Keep metadata scalar and never
  record prompt/tool/result bodies.
- Async callback instrumentation can change callback semantics if wrapped
  carelessly. Preserve return values and error propagation; unexpected errors
  should still surface.
