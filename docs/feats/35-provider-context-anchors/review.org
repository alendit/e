* Review loop

** Finding 1: Provider-anchor compatibility is too weak

*** Problem

Subagent review found that provider-anchor compatibility only compared
provider id, model, current path, covered entry id, and non-history segment
fingerprints.  The plan requires continuation anchors to invalidate when
request-semantic inputs change, including reasoning options, active layer ids,
tool schema names/definitions, and compaction/current-state compatibility.

*** Change

Anchor compatibility now fingerprints non-history context segments, active
layer ids, tool definitions, reasoning options, provider request-shaping
options, and latest compaction boundary data.  Legacy opaque fingerprints are
handled conservatively: identical opaque fingerprints remain compatible, while
old opaque anchors compared against the new structured fingerprint shape are
rejected instead of becoming over-broadly compatible.

*** Evidence

=e-session-test-latest-provider-anchor-requires-current-path= covers the
legacy opaque fingerprint fallback.  Harness regressions cover invalidation
for current-state segment changes, reasoning changes, tool schema changes,
active layer changes, and provider option changes.

** Finding 2: Anthropic cache/context anchors are not persisted

*** Problem

Subagent review found that Anthropic support only builds request metadata.
The plan calls for Anthropic cache/context metadata to use the same opaque
provider-owned anchor slot while preserving the full-history invariant.

*** Change

Anthropic successful cached/context-managed responses now emit
=provider-anchor-candidate= items carrying sanitized full-history cache,
container, beta-header, and context-management metadata.  The harness persists
them through the same provider-anchor slot while Anthropic request construction
continues to send the full logical history.

*** Evidence

=e-anthropic-test-emits-cache-anchor-candidate= proves the adapter emits an
Anthropic provider anchor candidate before the terminal success item.
=e-harness-test-provider-anchor-candidates-are-persisted= proves provider
anchor candidates are stored in the session, and
=e-harness-test-provider-anchor-invalidates-on-provider-option-change= proves
Anthropic container/cache shaping changes invalidate reuse.

** Finding 3: Diagnostics do not report invalidation reasons or exact Anthropic breakpoint

*** Problem

Subagent review found that OpenAI request metadata can distinguish used/full
continuation, but not why a candidate anchor was rejected.  Anthropic metadata
reports a broad breakpoint class but not the specific segment id/fingerprint
chosen for the cache boundary.

*** Change

OpenAI request metadata now includes
=:provider-anchor-invalidation-reason= when continuation mode falls back to a
full replay.  Anthropic cache metadata now records the exact breakpoint segment
id and fingerprint in addition to the broader breakpoint class.

*** Evidence

=e-openai-test-request-context-reports-full-replay-metadata= proves the
invalidation reason is surfaced.  =e-anthropic-test-request-context-reports-cache-metadata=
proves the exact Anthropic breakpoint segment id and fingerprint are reported.

** Finding 4: Responses continuation is enabled by default for every Responses profile

*** Problem

Subagent review found that continuation defaults to enabled for every
Responses profile, making OpenAI-compatible gateways opt out rather than
explicitly opt in.

*** Change

Responses continuation is now profile opt-in.  The built-in Codex Responses
profile sets =:continuation t=; generic Responses profiles only get
=:provider-continuation= defaults when they explicitly request continuation.

*** Evidence

=e-openai-test-responses-profile-can-disable-continuation= proves an unmarked
Responses profile uses full replay defaults.  =e-openai-test-responses-profile-can-enable-continuation=
proves explicit opt-in still enables provider continuation, and
=e-openai-test-default-harness-uses-gpt55-high-effort= proves the default Codex
profile still uses continuation.

** Initial implementation summary

The implementation adds:

- backend-neutral context segments for static prefix, stable context,
  current-state context, and transcript history;
- replay-safe provider-anchor records in the session store, with compatibility
  checks against provider id, model, current transcript path, covered entry id,
  and provider-relevant segment fingerprints;
- OpenAI Responses continuation support using stored response ids,
  =previous_response_id=, current instruction resend, full-replay fallback, and
  current-state invalidation;
- Anthropic cache-aware request construction using segment boundaries, optional
  top-level =cache_control=, optional =container=, gated raw
  =context_management= with beta headers, and full-history diagnostics;
- sanitized provider request metadata that reports continuation/cache decisions
  without dumping prompt contents.

* Architecture review

Core stores opaque provider anchors and compares provider-neutral
compatibility data, but provider-specific meanings stay in the adapters:
OpenAI owns response ids and =previous_response_id=; Anthropic owns
=cache_control=, =container=, beta headers, and =context_management= request
shape.  The local session remains authoritative because Anthropic never omits
history and OpenAI continuation only applies after local compatibility checks.

* Design self-check

1. The package is useful if stopped here because existing providers keep flat
   replay compatibility while OpenAI can safely use continuation anchors.
2. The work moves toward provider-managed request optimizations without making
   provider state authoritative.
3. Provider API details are likely to change and are local to adapters; segment
   and anchor compatibility are stable local contracts.
4. Context assembly owns segment semantics, sessions own durable local anchors,
   adapters own provider request shape.
5. Coupling is reduced by giving adapters structured context while keeping core
   away from provider-specific fields.
6. Stable core policy does not depend on concrete OpenAI or Anthropic request
   fields.
7. Provider side effects remain in adapter request construction and transport
   code.
8. Segment and anchor contracts are real: providers can ignore segments,
   consume cache breakpoints, or use continuation anchors without weakening the
   local transcript contract.
9. Existing sessions without anchors load; non-OpenAI providers can ignore
   anchors; Anthropic cache support never permits transcript omission.
10. No legacy prompt-cache path was removed beyond extracting request assembly
    helpers.
11. Expected invalidations happen through fingerprint/path/model checks;
    unexpected provider errors still surface through normal backend error paths.
12. Core behavior is covered by context, session, and harness tests; provider
    shape is covered by adapter request tests.

*** Finding 5: Compatibility omitted request-shaping options

**** Problem

The second review found that provider-anchor compatibility omitted request

=:max-tokens=.  Reusing an anchor across those changes would violate the
strict compatibility requirement.

**** Change

**** Evidence

=e-harness-test-provider-anchor-invalidates-on-instructions-change= and
=e-harness-test-provider-anchor-invalidates-on-max-tokens-change= cover the
new invalidation cases.

*** Finding 6: OpenAI unstored responses could persist anchors

**** Problem

The second review found that OpenAI parsing emits response-id anchor
candidates whenever a completed response contains an id, but Responses
requests without continuation are sent with =store: false=.  Persisting those
ids could later make an opted-in continuation request reuse an unstored
provider response id.

**** Change

The parser still reports provider output, but the harness now persists OpenAI
provider-anchor candidates only when the request context had OpenAI
continuation enabled.  Anthropic cache/context candidates remain persistable
because they are diagnostic cache anchors and do not imply transcript
omission.

**** Evidence

=e-harness-test-openai-anchor-candidates-require-continuation= proves OpenAI
response ids are skipped when continuation is disabled.  The existing
=e-harness-test-provider-anchor-candidates-are-persisted= covers the enabled
OpenAI continuation path.

*** Finding 7: Anthropic beta headers were active without context management

**** Problem

The second review found that Anthropic beta headers could be sent and recorded
even when no =context_management= body was sent.  The plan scopes those beta
headers to explicit context-management requests.

**** Change

Anthropic request context now activates beta headers only when context
management is also configured.  Request metadata records beta headers only
when the body includes =context_management=.

**** Evidence

=e-anthropic-test-request-context-omits-unused-beta-headers= proves beta
headers and metadata are absent when no context-management request is active.
=e-anthropic-test-request-context-sends-gated-context-management= continues to
cover the active gated path.
