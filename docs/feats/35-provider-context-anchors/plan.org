#+title: Provider Context Anchors
#+startup: overview

* Goal

Let provider adapters reuse provider-managed conversation state when it is safe
to do so, without making provider state authoritative for =e= sessions.

The MVP should add a backend-neutral context segmentation contract and
provider-owned continuation anchors.  OpenAI Responses can then use stored
response ids and =previous_response_id= for compatible follow-up requests,
while all providers can still fall back to the current full-replay request
shape.

* Status

In-review.

* Background

Feature 23 made context prompt-cache friendly by keeping stable request
prefixes stable, moving dynamic current-state context later, deriving
=prompt_cache_key= values, and forwarding cache usage into =token-usage=
events.  Recent long sessions show this works: stable prefixes often reach high
cached-input ratios.

The remaining inefficiency is that the OpenAI/Codex adapter is deliberately
stateless:

- =e= owns sessions, transcript stack, compaction, live context attachments,
  tools, and events.
- =e-openai-codex-request-body= sends =store: false= to the Responses-style
  endpoint.
- No adapter path persists a provider response id or sends
  =previous_response_id=.
- Every provider round trip gets a freshly flattened view of the current
  context and transcript.

Prompt caching reduces the cost of repeated stable prefixes, but each new tool
result, assistant message, canvas state refresh, compaction summary, or user
message changes the request suffix.  The model still receives a full logical
conversation window, and the client still serializes and sends it.

Current OpenAI Responses guidance separates these concerns:

- =previous_response_id= chains responses and lets the API manage multi-turn
  continuation state.
- =store: true= makes the response available as a continuation anchor.
- Prior input tokens in a response chain are still billed as input tokens.
- =instructions= from a previous response are not carried over, so clients must
  send the current instructions for each request.
- Prompt caching remains the cost optimization for reused prefixes.

Current Anthropic Messages guidance points to a different implementation shape:

- Messages API multi-turn conversations are stateless: the client sends the
  prior conversational turns in =messages= on each request.
- Anthropic prompt caching can be enabled with top-level =cache_control=, which
  automatically applies a cache breakpoint to the last cacheable block, or with
  explicit =cache_control= on individual blocks.  The cached prefix spans
  =tools=, =system=, and =messages= in that order, and cache usage is reported
  through =cache_creation_input_tokens= and =cache_read_input_tokens=.
- Beta =container= is exposed as a reusable provider identifier, but current
  public SDK/API descriptions do not make it a transcript-continuation handle;
  it must not justify omitting local history.
- Beta =context_management= can clear old tool results or thinking blocks
  server-side, but Anthropic documents that the client maintains the full,
  unmodified conversation history locally.
- Beta server-side compaction can emit =compaction= blocks that the client
  appends for later requests; adopting that would require explicit local
  modeling of those blocks and should not be confused with =e='s own
  transcript-stack compaction.

This feature should treat provider continuation as an optional request
optimization and state-management aid, not as a replacement for local
transcript semantics, compaction, or prompt-cache discipline.

* Non-goals

- Do not make OpenAI or any provider the source of truth for =e= session
  history.
- Do not remove transcript-stack session semantics.
- Do not make =previous_response_id= a core harness concept.
- Do not require providers to support server-managed continuation.
- Do not make provider continuation mandatory for correctness.
- Do not blindly chain provider responses across changed current-state context.
- Do not solve token minimization only with provider state; compaction and tool
  output discipline still matter.
- Do not change user-facing chat, canvas, or Org Canvas behavior in the first
  slice.

* Current behavior

- =e-harness-context= builds backend-neutral context from active capability
  messages and the session transcript.
- =e-capabilities-context-messages= sorts capability instructions and context
  providers by cache-placement rank, priority, capability order, provider
  order, and message order.
- Context providers can classify messages as =static-prefix=,
  =stable-context=, or =dynamic-context=.
- =e-chat-session-context-attachments-provider= marks live attachments and
  canvas content as =dynamic-context= because they represent current state,
  not transcript history.
- =e-context-transcript-stack-create= returns transcript messages, optionally
  replacing compacted history with a compaction summary plus a kept suffix.
- =e-openai-codex-request-body= moves backend-neutral system messages into
  top-level Responses =instructions=, sends non-system messages as =input=,
  and sets =store= to false.
- =e-openai-codex--usage-item= normalizes cached input tokens into the
  provider-neutral =token-usage= event payload.

* Accepted decisions

** Preserve local session authority

=e-session= remains the canonical record of user messages, assistant messages,
tool calls, tool results, compactions, activity events, and branches.

Provider continuation state is a disposable cache.  If it is absent, rejected,
expired, ambiguous, or incompatible with the current context, the adapter must
fall back to the existing full-replay request without changing behavior.

** Segment context before flattening

Core context assembly should expose enough structure for providers to reason
about what changed.  The old flat =:messages= result should remain available
for existing providers and tests.

The segment model should describe semantics, not a provider feature:

| Segment kind | Meaning |
|--------------+---------|
| =static-prefix= | capability instructions and stable policy that should be resent every request |
| =stable-context= | project, layer, resource, skill, and catalog context that changes rarely |
| =current-state= | live buffer, canvas, visible-window, and other replace-not-append state |
| =history= | durable session transcript through a known entry |
| =delta= | new transcript items after a chosen provider anchor |

Names can change during implementation, but the contract should avoid
provider-specific terms such as =cached= and =new=.

** Fingerprints prove compatibility

Segments should carry deterministic fingerprints derived from the backend
neutral content they contribute.  Provider adapters can persist those
fingerprints alongside their provider-specific anchor data.

A continuation anchor is valid only when all provider-relevant fingerprints
still match:

- model id;
- reasoning mode when it affects request semantics;
- active layer ids;
- tool schema names and definitions;
- static instruction content;
- stable context content;
- current-state epoch or fingerprint;
- compaction boundary;
- transcript entry id covered by the anchor.

If any fingerprint differs, the provider should start a new anchor from a full
request.

** Keep provider state below the adapter boundary

Core may store opaque provider-anchor metadata under the session, but only the
owning provider interprets provider-specific fields such as OpenAI response ids
or Anthropic cache/context-management metadata.

A provider anchor should be shaped like:

#+begin_src elisp
(:provider openai
 :wire-api responses
 :response-id "resp_..."
 :covered-entry-id "01..."
 :context-fingerprint "..."
 :static-fingerprint "..."
 :stable-fingerprint "..."
 :current-state-fingerprint "..."
 :tool-schema-fingerprint "..."
 :model "gpt-5.5"
 :created-at "2026-06-21T...")
#+end_src

Core code should not branch on =:response-id=, =previous_response_id=,
Anthropic =cache_control= placement, or Anthropic =context_management= edits.

Anthropic anchors should use the same opaque provider-owned slot, but their
first implementation should not pretend that Messages has an OpenAI-style
response-chain id.  Useful Anthropic anchor fields are likely to be:

#+begin_src elisp
(:provider anthropic
 :wire-api messages
 :mode prompt-cache
 :container-id "..."             ; only when explicitly enabled by profile
 :covered-entry-id "01..."
 :static-fingerprint "..."
 :stable-fingerprint "..."
 :current-state-fingerprint "..."
 :tool-schema-fingerprint "..."
 :model "claude-opus-4-8"
 :cache-ttl "5m"
 :context-management (...opaque provider options...)
 :created-at "2026-06-21T...")
#+end_src

The Anthropic adapter owns the meanings of =:container-id=,
=:context-management=, cache-control placement, and any future compaction block
metadata.  Core should only persist and select anchors by provider id,
covered-entry id, and provider-supplied compatibility fingerprints.

** Use provider continuation conservatively

The first useful target is OpenAI Responses continuation for requests where the
context fingerprints are unchanged and only transcript delta has been appended.

When valid, the adapter can send:

- current =instructions=, because OpenAI does not carry prior instructions
  forward;
- =previous_response_id= from the anchor;
- only the new input items after =:covered-entry-id=;
- the current tool definitions when required by provider semantics;
- =store: true= so the new response can become the next anchor.

When invalid, the adapter sends the existing full request, with =store: true=
only if continuation mode is enabled and the provider accepts it.

** Reset on replace-current-state changes

Current-state context is not append-only.  Canvas attachments, Org Canvas
focus, visible buffers, and similar provider inputs replace earlier state.

The provider must not chain through an anchor if the current-state fingerprint
changed, because the prior provider state may still contain stale live context.

This rule is stricter than prompt caching.  Prompt caching can tolerate dynamic
context late in the request; provider continuation must prove that replacing
state has not changed before it omits older items.

For Anthropic, a changed =current-state= fingerprint should not imply that the
adapter may omit old history, because Anthropic does not have that
continuation path.  It should instead force the cache/context anchor to rebuild
or move its breakpoint so stale live context is not hidden behind an old
cache-control decision, container id, or context-management diagnostic.

** Preserve prompt cache behavior

=prompt_cache_key= should continue to be sent for OpenAI requests where it is
supported.  Provider continuation and prompt caching are complementary:

- provider continuation can reduce client-side replay and preserve response
  chain semantics;
- prompt caching reduces cost and latency for repeated prefixes;
- compaction and transcript pruning reduce the logical input window.

For Anthropic, this means preserving the existing =cache_control= support and
making it segment-aware.  Anthropic cache hits should be reported through
=cache_read_input_tokens= and writes through =cache_creation_input_tokens=; an
Anthropic cache anchor should explain breakpoint choice and invalidation, not
claim that any local transcript prefix was omitted.

** Support Anthropic with cache and context-management anchors

Anthropic Messages is still a stateless multi-turn API: callers send the
conversation history in =messages=, and the service generates the next assistant
message.  The Anthropic provider should therefore not try to implement an
OpenAI-style =previous_response_id= path unless Anthropic later exposes an
explicit continuation id.

The comparable Anthropic support should use the same segment fingerprints for
three provider-local optimizations instead:

- place =cache_control= breakpoints on stable prefixes, or enable top-level
  automatic prompt caching, so tools, system content, and stable history can be
  reused by Anthropic's prompt cache;
- optionally enable Anthropic =context_management= beta edits for long-running
  conversations, especially tool-result clearing and thinking-block clearing;
- optionally enable Anthropic server-side compaction only behind an explicit
  profile gate, because compaction returns provider-specific =compaction=
  blocks that must be appended to subsequent requests and reconciled with =e='s
  local compaction semantics.

Anthropic prompt caching does not let the client omit prior messages.  Cache
anchors for Anthropic should be diagnostic and request-construction metadata,
not transcript coverage proofs.  A cached-prefix anchor can record the segment
fingerprints, cache TTL, beta headers, and chosen breakpoint, but a valid
Anthropic cache anchor still sends the full logical request.  Its purpose is to
keep cache breakpoints stable, report cache hits and writes, and decide when to
move or rebuild the breakpoint.

Anthropic context editing also must not become authoritative local state.  For
tool-result and thinking clearing, the API edits context server-side before the
model sees it; =e= should keep its full transcript and continue sending the
canonical history.  Diagnostics should report that server-side clearing was
requested and usage should surface =cache_read_input_tokens= and
=cache_creation_input_tokens=, but local transcript entries remain unchanged.

Server-side Anthropic compaction is stronger and should be treated separately
from provider continuation.  If enabled, the adapter must capture any returned
=compaction= blocks as provider-specific artifacts and either persist them as
opaque Anthropic continuation metadata or translate them into an explicit =e=
compaction record through a deliberate core path.  It must never silently drop
local transcript history just because Anthropic would drop blocks before the
provider compaction marker on the next request.

** Anthropic adapter implementation guidance

The current Anthropic adapter already has the right first building block:
=e-anthropic-request-body= can attach an ephemeral =cache_control= breakpoint to
the system block, or to the last tool definition when there is no system prompt.
The feature should evolve that behavior rather than replace it.

The Anthropic-specific implementation should proceed in this order:

1. Keep the existing flat =:messages= request path as the compatibility path.
   Add segment-aware request construction only when the richer context result
   includes =:segments=.
2. Split request construction into small helpers that can be tested without
   performing HTTP:
   - choose an Anthropic cache mode: =off=, =top-level= automatic caching, or
     =explicit= breakpoint placement;
   - choose the breakpoint segment from =static-prefix= and =stable-context=
     content, stopping before =current-state= and uncached transcript =delta=;
   - attach =cache_control= to the exact Anthropic block when explicit placement
     is required;
   - build optional =context_management= and beta header data from provider
     profile/config;
   - build a compact diagnostic plist that records cache mode, breakpoint
     segment id/fingerprint, TTL, context-management edits, container id, and
     =:full-history t=.
3. Preserve Anthropic's full-history invariant.  Even when an Anthropic cache
   anchor is valid, =:messages= should contain the full logical conversation
   produced by =e=, subject only to =e='s own transcript-stack compaction.
4. Treat Anthropic =container= as provider-owned metadata.  A configured
   reusable container id may be sent and recorded, but it is not a transcript
   continuation proof.  Container id changes should invalidate Anthropic cache
   anchors that depended on that container.
5. Gate =context_management= behind profile/config flags.  The adapter should
   add the required beta header only when it sends =context_management=, and
   diagnostics should state which edits were requested.  Tool-result clearing
   and thinking-block clearing must not mutate =e= session entries.
6. Keep server-side Anthropic compaction disabled in this feature unless a
   separate explicit integration captures returned =compaction= blocks.  If that
   later becomes enabled, tests must prove that local transcript history is not
   silently dropped or replaced outside an =e= compaction record.

The practical target is therefore not fewer Anthropic =messages= items.  It is
more stable cache breakpoint selection, clearer invalidation reasons, optional
server-side context editing, and better diagnostics while preserving local
session authority.

* Proposed API

The implementation should introduce a backend-neutral context result that can
serve both old and new providers:

#+begin_src elisp
(:strategy transcript-stack
 :messages (...flat backend-neutral messages...)
 :segments ((:kind static-prefix
             :id instructions
             :fingerprint "..."
             :messages (...))
            (:kind stable-context
             :id agents-std-context
             :fingerprint "..."
             :messages (...))
            (:kind current-state
             :id chat-session-attachments
             :fingerprint "..."
             :messages (...))
            (:kind history
             :through-entry-id "01..."
             :fingerprint "..."
             :messages (...)))
 :options (...))
#+end_src

The exact plist keys should follow existing =e= naming style.  The important
requirement is that old consumers can keep using =:messages= while provider
adapters can opt into =:segments=.

Provider anchor metadata can live in session metadata or a dedicated session
record type.  A dedicated record type is preferable if anchors will change
often during tool loops, because session metadata is better for user-visible
session facts.

* Implementation slices

** Slice 1: Context segments with flat compatibility

Add segment metadata to context assembly without changing provider request
behavior.

Acceptance criteria:

- Existing context callers still receive =:messages= with the same order.
- Context results also include segment metadata.
- Static capability instructions, stable providers, dynamic providers, and
  transcript history are represented as distinct segment kinds.
- Segment fingerprints are deterministic and covered by tests.
- No provider uses the segments yet.

** Slice 2: Provider anchor persistence

Add a replay-safe way to store opaque provider anchors against a session.

Acceptance criteria:

- A provider can append an anchor with provider id, covered transcript entry,
  fingerprints, model, and provider-specific opaque fields.
- Session reload reconstructs anchors.
- Branching, compaction, or transcript truncation cannot make a stale anchor
  look current.
- Existing sessions without anchor records load normally.

** Slice 3: OpenAI stream response id capture

Teach the OpenAI Responses adapter to observe completed response ids and emit
or return anchor candidate data without changing request replay behavior.

Acceptance criteria:

- OpenAI stream parsing captures response ids from completed response events.
- Token usage parsing remains unchanged.
- The adapter can produce an anchor candidate that core can persist.
- Missing response ids do not fail a turn.

** Slice 4: OpenAI continuation request construction

Add request construction for valid anchors.

Acceptance criteria:

- When an anchor is valid, the request body includes =previous_response_id=.
- The request still sends current =instructions=.
- The request includes only transcript delta after the anchor plus required
  current input.
- The request sets =store= true only when continuation mode is enabled and the
  provider profile supports it.
- When any relevant fingerprint differs, the request body is identical to the
  full-replay path except for allowed storage configuration.

** Slice 5: Context epoch resets for current-state providers

Make current-state changes invalidate continuation anchors.

Acceptance criteria:

- Changing a canvas attachment invalidates the previous anchor.
- Changing visible-buffer context invalidates the previous anchor when that
  provider is active.
- A stable-only prompt change does not invalidate more than required.
- Tests prove stale current-state context is not silently hidden behind
  =previous_response_id=.

** Slice 6: Anthropic cache and context-management support

Use context segment fingerprints to make Anthropic request construction more
cache-aware without pretending Anthropic has OpenAI-style continuation.

Implementation notes:

- Start by extracting the current cache-control placement logic from
  =e-anthropic-request-body= into a helper that accepts already-built system,
  tool, message, cache, and segment inputs.
- Add a pure breakpoint-selection helper that returns an explanatory plist such
  as =(:mode explicit :segment stable-context :segment-id ... :fingerprint ...
  :ttl "5m" :reason before-current-state)=.
- Prefer =(:cache_control (:type "ephemeral" ...))= at the top level for simple
  automatic caching when no segment boundary needs exact placement.
- Use explicit block-level =cache_control= when the segment boundary matters:
  for example, cache tools + system + stable context, but stop before
  =current-state= and appended transcript =delta=.
- Add a provider-profile option for =context_management= edits and beta headers,
  but keep it off by default.
- Emit diagnostics/anchor candidates that include =:full-history t= so future
  readers can see that Anthropic cache anchors are not continuation proofs.

Acceptance criteria:

- Anthropic requests can choose stable =cache_control= breakpoints from segment
  boundaries while preserving existing prompt-cache behavior.
- Anthropic requests can use top-level automatic =cache_control= when exact
  segment placement is unnecessary.
- Anthropic cache metadata records provider id, model, segment fingerprints,
  cache TTL, beta headers, breakpoint choice, and =:full-history t=, but does
  not allow omitting prior transcript messages.
- Optional =container= ids are sent only when configured and recorded only as
  opaque provider metadata.
- Optional =context_management= edits are gated by provider profile/config and
  include the required beta headers.
- Tool-result clearing and thinking-block clearing leave local =e= transcript
  state unchanged.
- Server-side Anthropic compaction remains disabled unless a separate explicit
  integration maps returned =compaction= blocks into provider metadata or =e=
  compaction records.
- Usage normalization keeps reporting cache creation/read tokens.

** Slice 7: Live verification and status evidence

Expose enough diagnostics to prove when a turn used continuation, full replay,
Anthropic cache breakpoints, or Anthropic context-management edits.

Acceptance criteria:

- Provider request events or debug diagnostics show =:continuation used|full=,
  anchor id, and invalidation reason without dumping full prompts.
- Anthropic diagnostics show chosen cache breakpoint, cache TTL, cache
  creation/read tokens, and any requested context-management edits without
  dumping full prompts.
- Live Emacs verification covers one full-replay request, one valid
  continuation request, and one invalidated continuation after current-state
  change.
- Anthropic live verification covers a cached-prefix request and a request with
  context-management edits enabled when a supported profile is configured.
- Token usage events still show input, cached input, output, reasoning, and
  total tokens.

* Compatibility expectations

- Existing sessions without provider anchors continue to load.
- Non-OpenAI providers continue to receive flat =:messages=.
- OpenAI profiles can disable continuation.
- Anthropic profiles can keep today’s prompt-cache-only behavior and ignore
  anchor metadata.
- Anthropic context-management beta features are opt-in and profile-gated.
- Anthropic cache anchors never permit transcript omission, so they cannot
  diverge from local session authority.
- ChatGPT-backed Codex endpoints may reject public API fields differently from
  token-auth Responses endpoints; adapter profile gates must keep this
  explicit.
- If provider continuation fails during a turn, the user should get a normal
  provider error or a safe retry through full replay, not silent transcript
  divergence.

* Risks

- Provider state can become a hidden second source of truth if invalidation is
  too weak.
- Current-state context can become stale if replacement semantics are not
  modeled explicitly.
- Tool loops may append many anchors; persistence should avoid noisy metadata
  churn.
- OpenAI billing still includes prior input tokens in response chains, so this
  work may improve replay/state handling more than direct token cost.
- ChatGPT-backed Codex endpoint behavior may differ from public OpenAI
  Responses; live validation must use the configured provider.
- Anthropic server-side context editing can hide old tool results from the
  model while the local transcript still contains them; diagnostics must make
  that provider-side edit visible.
- Anthropic server-side compaction can become a competing compaction system if
  returned =compaction= blocks are not explicitly modeled.
