* Plan Acceptance Review

** Accepted

- Context stays provider-neutral.  Capability aggregation still returns ordered
  backend-neutral message plists; backend adapters remain responsible for
  provider-specific mapping/collapsing.
- Lower numeric priorities appear earlier in generated context.
- Capabilities without explicit priorities keep the default =200= behavior.
- Equal priorities preserve capability traversal order and provider message
  order.  Sorting keys are priority, capability traversal index, provider index,
  and message index.
- Provider build functions still return ordinary message plists.  Provider
  priority is provider-level and applies to all messages in that provider's
  returned list.
- Capability instruction priority is capability-owned and defaults to =200=.
- AGENTS.md context is assigned priority =100=.
- =agents-std-context= instructions/global skill catalog and project skill
  catalog are assigned priority =220=.
- OS base guidance is assigned priority =230=.
- Harness self-management guidance is assigned priority =240=.
- Emacs awareness instructions are assigned priority =300=.
- Visible buffers are assigned priority =320=.
- The default chat layer order still remains meaningful as a tie-breaker:
  =(chat-session agents-std-context harness-base e os-base emacs-base)= in the
  live default harness, while priority moves stable project policy ahead of
  later lower-band guidance where needed.
- Context export text now uses "Backend-neutral context fragment count", labels
  sections as "Context fragment N", and includes an adapter-mapping note:
  "System fragments may be mapped or collapsed by backend adapters before
  provider submission."
- Skill ordering remains separate from context-fragment priority.  The current
  =agents-std-context= skill precedence documentation states that global and
  project skills are distinct scoped resources, nearest project skill wins for a
  duplicate project slug, parent project skills remain available, and prompt
  ordering is global first then project skills from outer to inner roots after
  duplicate resolution.
- Later session-root/config changes did not break feature-09 ordering: the
  current AGENTS.md and project skill providers resolve through the active
  harness/session project root, then their messages enter the same priority
  ordering path.

** Current Implementation Notes

- The original review's only remaining UX note has been resolved.  I inspected
  an ordinary live chat context after reload.  The live default chat context had
  six fragments in this order: AGENTS.md, =agents-std-context= instructions/skill
  catalog, OS base guidance, harness guidance, Emacs awareness guidance, and
  visible buffers.
- The adapter note is intentionally generic rather than Codex-specific.  This
  is acceptable because the plan allowed provider-neutral wording and only
  required avoiding the impression that every backend receives separate system
  messages.
- Export metadata still returns =:message-count= in the tool result plist for
  API compatibility, while the Markdown body uses context-fragment wording.  I
  do not consider that a feature-09 blocker because the user-facing export body
  was the planned wording change and existing tests still assert the metadata
  key.

** Follow-up Notes

- Stale-record tolerance is partial.  The custom accessors
  =e-context-provider-priority= and =e-context-provider--build-function= can read
  short pre-priority =e-context-provider= vectors directly, and
  =e-capability-instruction-priority= defaults old =e-capability= vectors to
  =200=.  However, =cl-defstruct= predicates such as =e-context-provider-p= and
  =e-capability-p= do not recognize older vectors once the struct length has
  changed.  As a result, stale context-provider objects can still fail when they
  flow through =e-capabilities--provider-messages=, and stale capability objects
  can fail if generated accessors such as =e-capability-instructions= are used
  before tolerant accessors.  In normal startup/reload paths the default harness
  recreates configured stateless layers, so this is not showing up in the live
  default chat context.  If the project wants robust live-upgrade support for
  arbitrary already-active custom layers, add a small follow-up to make stale
  provider/capability detection explicit instead of relying on generated struct
  predicates.
- Test coverage is good for the main ordering behavior, but the stale-record
  compatibility edge above is not covered end-to-end.  Add regression coverage
  if that compatibility guarantee is promoted from "where possible" to a hard
  requirement.
