* Design Review

1. If this work package stopped here, would the project be better off than if it
   had not been built?
   Yes. Default chat and canvas requests now have more stable prefixes and can
   opt into provider cache controls without changing shell behavior.
2. What final direction does this move toward?
   Provider-neutral context policy in core with provider-specific request fields
   isolated in adapters.
3. Which decisions are likely to change, uncertain, or stable?
   The three placement buckets are stable enough to name. The cache-key version
   keeps future key-shape changes local.
4. What component owns this behavior, and why?
   =e-context= owns provider metadata, =e-capabilities= owns ordering, the
   harness owns default turn options, and OpenAI owns wire-field mapping.
5. Does this change increase or reduce coupling?
   It reduces adapter coupling by keeping OpenAI field names out of context
   providers and shells.
6. Did any dependency start pointing the wrong way?
   No. Core exposes neutral options and metadata; the adapter translates them.
7. Could any side effect be moved outward into an adapter or shell?
   No new side effects were introduced. Cache-key derivation is pure.
8. Are the abstractions semantically real?
   Yes. Static, stable, and dynamic prompt regions affect cache reuse and have
   concrete providers.
9. What compatibility expectations apply?
   Old provider records without cache placement default to stable context during
   live reload or incremental bytecode use.
10. What legacy code can we remove now?
    None in this slice.
11. Where are expected errors handled?
    Invalid placement values surface as =wrong-type-argument=; unsupported
    provider/model cache options surface at the provider request boundary.
12. What tests prove the core behavior?
    Context, capability ordering, OpenAI request, default harness, Emacs
    provider, and Org Canvas provider tests cover the behavior independently of
    a live OpenAI request.
