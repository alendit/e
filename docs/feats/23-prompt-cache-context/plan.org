#+title: Plan: Prompt Cache Friendly Context
#+author: e planning
#+date: 2026-06-07
#+startup: overview

* Status

Planned.

* Goal

Make default chat, canvas, and Org Canvas context assembly more friendly to
OpenAI prompt caching by keeping stable request prefixes stable, moving
per-turn dynamic context later, and wiring explicit cache controls through the
OpenAI adapter.

* Motivation

Cached input tokens are cheaper and can reduce latency. Current E requests can
benefit from OpenAI's automatic prompt caching when long stable prefixes repeat,
but the default chat context strategy does not deliberately maximize that
benefit:

- the OpenAI adapter records cached token usage after the response, but does not
  send =prompt_cache_key= or =prompt_cache_retention=;
- all capability context is built as prefix messages before transcript history;
- dynamic canvas, Org Canvas focus, visible-buffer, and attachment content can
  appear early enough to shorten reusable prefixes;
- canvas and Org Canvas use the default chat harness, so their cache behavior is
  a consequence of the shared context pipeline rather than a separately designed
  strategy.

Current OpenAI guidance is to keep static or repeated content at the beginning,
place dynamic user-specific context at the end, use =prompt_cache_key=
consistently for repeated traffic with common prefixes, optionally use
=prompt_cache_retention= where supported, and monitor cached token counts.

* Non-goals

- Do not switch E to OpenAI-managed conversation state or =previous_response_id=
  in this feature.
- Do not remove transcript-stack session semantics.
- Do not change the user-facing behavior of chat, canvas, or Org Canvas prompts.
- Do not hide live buffer, canvas, or visible-window context from the model.
- Do not make cache behavior OpenAI-only in core harness abstractions. Core may
  expose neutral metadata or options, but provider-specific request fields belong
  in the OpenAI adapter.
- Do not optimize every token in the prompt. This feature is about stable prefix
  structure and measurable cache controls, not prompt minification.

* Current behavior

- =e-default-chat-layer-ids= activates =agents-std-context=, =harness-base=,
  =e=, =os-base=, =emacs-base=, =web=, =text-editing=, =org-canvas=, and
  =project-local= by default.
- =e-default-chat-harness-create= creates an OpenAI harness, attaches the
  internal =chat-session= layer, then activates configured layers.
- =e-harness-context= asks capabilities for context messages, then passes those
  messages as =:prefix-messages= to =e-context-build=.
- =e-context-transcript-stack-create= returns transcript messages, optionally
  compacted, and =e-context-build= prepends all capability context messages.
- =e-capabilities-context-messages= sorts capability instructions and context
  provider messages by priority and stable tie-breakers.
- =e-openai-codex-request-body= moves all backend-neutral system messages into
  the Responses API top-level =instructions= string, then sends non-system
  messages as =input= items.
- =e-openai-codex--usage-item= and
  =e-openai-chat-completion--usage-item= normalize =cached_tokens= into
  =:cached-input-tokens=, so E can already observe prompt-cache results.
- =e-chat-session-context-attachments-provider= emits current attachment and
  canvas contents as a system context message with priority 120.
- =e-org-canvas-context-provider= emits current scope, document uri, heading
  path, focus summary, and visibility state as a system context message with
  priority 118.
- =e-emacs-visible-buffers-context-provider= emits visible-buffer state with
  priority 320.

* Accepted decisions

** Preserve the shared default context pipeline

Default chat, canvas, and Org Canvas should continue to use the default chat
harness and transcript-stack strategy. Canvas-specific behavior should be
expressed through capability context and metadata, not by forking a separate
backend path.

Why:

- Canvas and Org Canvas already rely on chat sessions, tools, compaction, token
  usage, and presentation state.
- A shared fix improves every surface and avoids divergent provider behavior.
- The underlying issue is ordering and request metadata, not a need for a new
  shell-specific harness.

** Add cache controls at the OpenAI adapter boundary

Introduce backend-neutral turn options that the OpenAI adapter maps to API
fields:

- =:prompt-cache-key= -> =prompt_cache_key=
- =:prompt-cache-retention= -> =prompt_cache_retention=

The OpenAI adapter should include these fields only when present. Other
backends should ignore these options unless they explicitly support comparable
request fields later.

Default chat should derive a stable cache key when no explicit key is present.
The key should be stable for the repeated prefix shape but not include prompt
text, transcript text, canvas contents, or tool outputs. A suitable first key is
based on:

- provider profile or wire API;
- model;
- normalized project root when available;
- active layer ids;
- a small version string for the cache-context strategy.

** Classify context fragments by cache stability

Do not rely only on numeric provider priorities to express cache behavior. Add a
small cache-placement concept to context providers or capability fragments, with
at least:

- =static-prefix= :: repeated instructions and project policy that should appear
  earliest;
- =stable-context= :: project/layer context that may change across projects but
  is stable across many turns in one session;
- =dynamic-context= :: per-turn state such as live attachments, Org focus,
  visible buffers, current selection, and token/status observations.

Keep current priority ordering within a placement bucket so existing feature 09
ordering semantics remain useful. Existing providers without an explicit
placement should default to =stable-context= or the current behavior in a way
that preserves compatibility.

** Move dynamic context after stable instructions and policy

The model still needs dynamic context every turn, but it should not interrupt
the reusable prefix before stable guidance. The request should be shaped as:

1. base/provider instructions;
2. static capability instructions;
3. stable project and AGENTS context;
4. stable skill catalog/resource pointers;
5. dynamic live context for the current turn;
6. transcript and current user prompt.

For the Responses API adapter this still becomes a top-level =instructions=
string plus =input= items, but the top-level string should keep stable sections
before dynamic sections.

** Treat canvas and Org Canvas as dynamic context sources

Canvas and Org Canvas providers should explicitly mark their per-turn state as
=dynamic-context=:

- =e-chat-session-context-attachments-provider= for attachment and canvas
  contents;
- =e-org-canvas-context-provider= for focus, scope, heading path, document uri,
  and visibility state;
- =e-emacs-visible-buffers-context-provider= for visible-buffer summaries.

Their short, stable instructions can stay early. Their changing state should be
late enough that repeated default chat/project policy remains cacheable.

** Measure cache benefit with existing token usage events

Do not add a separate analytics system. Extend tests and, if needed, developer
status/reporting helpers to make =:cached-input-tokens= visible enough for
review. The feature review should compare at least two repeated requests with
the same cache key and stable prefix shape, recording:

- input tokens;
- cached input tokens;
- model/provider;
- cache key shape, not the full key if it contains local paths;
- surface used: default chat, canvas, or Org Canvas.

* Proposed API

Backend-neutral options:

#+begin_src emacs-lisp
(:prompt-cache-key "e:openai:gpt-5.5:/path/hash:layers/hash:v1"
 :prompt-cache-retention "24h")
#+end_src

OpenAI Responses body fields when present:

#+begin_src json
{
  "prompt_cache_key": "e:openai:gpt-5.5:project:layers:v1",
  "prompt_cache_retention": "24h"
}
#+end_src

Context provider creation should remain compact. One possible shape:

#+begin_src emacs-lisp
(e-context-provider-create
 :name 'org-canvas
 :priority 118
 :cache-placement 'dynamic-context
 :build #'e-org-canvas-context-provider)
#+end_src

If adding a struct slot is too reload-sensitive, use provider metadata or a
small wrapper helper instead. Do not add compatibility shims unless live reload
requires them for intended extension seams.

* Compatibility expectations

- Existing sessions remain readable.
- Existing context provider constructors without cache-placement continue to
  work.
- Current prompt behavior remains semantically equivalent: the model still
  receives stable instructions, project context, live Emacs context, canvas
  content, Org focus, transcript history, tools, and user prompts.
- Provider-specific prompt-cache fields are emitted only for OpenAI-compatible
  request bodies that support them.
- A full Emacs restart may be acceptable if adding a provider struct slot makes
  old live records stale. Do not add long-lived compatibility shims solely for
  stale in-memory struct layouts.

* Risks

- Reordering context can change model behavior even when all content is still
  present. Mitigate with focused chat, canvas, and Org Canvas tests plus live
  probes.
- Adding a struct slot can make live-reloaded provider records stale. Keep the
  change simple, document restart expectations if needed, and avoid masking
  unexpected wrong-type errors.
- A cache key that is too broad can miss cache opportunities; a key that is too
  narrow can cause cache churn. Start with project/model/layer granularity and
  record observations in review.
- =prompt_cache_retention= support may vary by model/provider. The OpenAI
  adapter should only send it when intentionally configured or clearly supported.
- Dynamic context moved too late could be easier for the model to overlook.
  Preserve clear headings and source labels.

* Review questions

1. Does the request prefix keep static instructions and project policy before
   per-turn live context?
2. Do default chat, canvas, and Org Canvas all use the same cache-friendly
   context pipeline?
3. Are provider-specific cache fields isolated in the OpenAI adapter?
4. Is the cache key stable for repeated traffic but sensitive to model, project,
   and active layer shape?
5. Does the model still receive all live context needed for correct canvas and
   Org Canvas edits?
6. Are cached token counts visible enough to verify the feature without adding a
   separate telemetry system?
7. What compatibility expectations apply to live reload and stale in-memory
   provider structs?
8. Where do unsupported provider/model cache options surface?
9. What tests prove ordering independently of a real OpenAI request?
10. What live evidence shows cached-token behavior improved?
