#+title: Investigation: OpenAI Codex Prompt Cache Retention
#+startup: overview

* Investigation

** Source inspection

- =lisp/adapters/openai/e-openai.el= serialized backend-neutral
  =:prompt-cache-retention= as the Responses request field
  =prompt_cache_retention= whenever the option was present.
- The built-in OpenAI provider profile is =codex=, which targets
  =https://chatgpt.com/backend-api/codex/responses= with ChatGPT-managed auth.
- The previous prompt-cache work intentionally added the request mapping, and
  its review recorded a live probe where retention was present.
- Current OpenAI public Responses API documentation and schema still list
  =prompt_cache_retention= as a supported public API field, so the failure is
  specific to the ChatGPT-backed Codex endpoint rather than the generic public
  Responses API shape.

** Root cause

The adapter treated every Responses-profile backend as accepting
=prompt_cache_retention=. The ChatGPT-backed Codex endpoint rejects that field
even though public/token-auth OpenAI Responses profiles may support it.

** Fix

- Added provider-profile gating for =prompt_cache_retention=.
- Profiles using Codex-managed ChatGPT auth now suppress
  =:prompt-cache-retention= unless a profile explicitly forces support with
  =:prompt-cache-retention= non-nil.
- Token-auth Responses providers keep explicit retention by default, and can set
  =:prompt-cache-retention= nil if a compatible gateway rejects the field.
- =prompt_cache_key= is still sent to the Codex endpoint.
