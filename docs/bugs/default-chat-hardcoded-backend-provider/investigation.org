#+title: Investigation: Default Chat Hardcoded Backend Provider
#+startup: overview

* Investigation

** Source inspection

- =lisp/defaults/e-default-harnesses.el= hardcoded the default chat harness to
  require =e-anthropic=, call =e-anthropic-create-harness=, and fall back to
  =e-anthropic-default-provider=.
- The same factory appended Anthropic-specific prompt-cache defaults to the
  harness default options.
- =e-default-harnesses-startup= syncs cached default instances from their
  factories after reload, so a live reload could replace a previously configured
  default chat backend with the package's Anthropic factory result.

** Root cause

The package default chat factory selected a concrete backend provider. That
violated the provider-neutral harness contract and made reload capable of
overwriting a user-configured backend with Anthropic.

** Fix

- Added =e-default-chat-harness-factory= as an explicit user-configuration
  variable. Its default is nil.
- =e-default-chat-harness-create= now creates an explicit unconfigured backend
  when no factory is configured from config.el/config.org and the user has not
  replaced =:chat-default= in =e-default-harness-specs=.
- The =:provider= argument is rejected so provider selection cannot slip back
  into package defaults.
- Default chat creation no longer requires =e-anthropic= and no longer adds
  provider-specific prompt-cache defaults.
- Startup sync refreshes backend/default option/runtime fields only when a
  configured factory or custom default spec exists. Without a configured default
  chat factory, startup still repairs shifted session slots and reconciles
  layers, but replaces cached backend/default options with an explicit
  unconfigured backend.

  - rejected =:provider= argument;
  - configured factory/session-store delegation;

    when no configured factory exists.
