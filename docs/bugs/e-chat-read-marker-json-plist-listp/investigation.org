#+title: Investigation: e-chat read-marker JSON plist listp restart failure

* Finding

The failure reproduced during =e-chat--attach-buffer= after
=e-session-load-session=, before provider context or request-body construction.
The failing string was the latest assistant message id stored as the
=chat-default= read marker, not a provider anchor.

* Root Cause

=e-session--json-read-line= parses JSON objects with =:object-type 'plist=.
When =:e-chat-read-markers= is persisted as a JSON object, replay returns a plist
such as:

#+begin_src elisp
(:chat-default "01KW92NEZQHYRCDN2KH538MPCT")
#+end_src

=e-chat-overview--session-read-marker= and
=e-chat-session-set-read-marker= used =alist-get= directly on that value.  On a
plist, =alist-get= eventually inspects the marker string as if it were an alist
cell and raises =wrong-type-argument listp=.

The underlying design error was persisting =:e-chat-read-markers= in session
metadata.  The marker only answers whether the current UI has already shown the
latest assistant message for an unread indicator.  That state is local to the
running Emacs presentation shell and is safe to lose on restart.

* Fix

Keep read markers out of durable session state:

- =e-chat= stores read markers in a process-local hash keyed by harness,
  session id, and chat instance;
- buffer attach and unread checks ignore any stale =:e-chat-read-markers=
  value loaded from old session metadata;
- marking a session read no longer calls =e-session-set-metadata=;
- chat-session metadata writes strip stale =:e-chat-read-markers= so future
  legitimate metadata updates do not carry presentation-only state forward.

This removes the restart crash and restores the correct ownership boundary:
durable session history contains conversation and session configuration, while
unread presentation state lives only in the UI process.

* Performance

The corrected read-marker path does less work than the persisted version.
Marking a session read now performs one process-local hash update and invalidates
the workspace unread cache.  It no longer copies session metadata, appends a
metadata event to the session store, rewrites persistent store indexes, or
serializes unread state to disk.

The only added metadata scan happens on existing chat-session metadata writes,
where the code already had to copy/update a plist before persisting it.  That
scan is linear in metadata size and replaces the prior copy-only step for rare
project-root/context-attachment writes, not for the frequent focus/read path.
