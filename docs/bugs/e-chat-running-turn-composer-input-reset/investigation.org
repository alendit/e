#+title: Investigation
#+startup: overview

* Source inspected

- =lisp/shells/chat/e-chat.el=: =e-chat-submit= deletes the real composer,
  sets status to queued, and inserts =e-chat--insert-pending-separator=.
- =lisp/shells/chat/e-chat.el=: =e-chat--insert-pending-separator= inserts
  bottom prompt chrome and the composer glyph, but intentionally does not set
  =e-chat--composer-start-marker=. Its docstring says it inserts prompt chrome
  without an editable composer.
- =lisp/shells/chat/e-chat.el=: =e-chat--composer-active-p= requires both
  =e-chat--transcript-end-marker= and =e-chat--composer-start-marker=. Pending
  prompt chrome therefore looks like a composer but is not an active composer.
- =lisp/shells/chat/e-chat.el=: =e-chat--start-progress-indicator= starts a
  repeating timer. Each tick calls =e-chat--advance-progress-indicator=, which
  calls =e-chat--render-progress-indicator= and then
  =e-chat--render-running-status=.
- =lisp/shells/chat/e-chat.el=: =e-chat--render-running-status= deletes the
  running status and any bottom prompt/composer region, redraws the transient
  activity/progress block, and when progress is active reinserts only
  =e-chat--insert-pending-separator=.
- =test/e-chat-test.el=: =e-chat-test-progress-rerender-keeps-pending-composer-glyph=
  explicitly asserts that progress redraw keeps a protected bottom composer
  glyph visible while =e-chat--composer-active-p= is false.

* Conclusion

The likely root cause is a mismatch between visual affordance and state
ownership during running turns. Running progress redraw intentionally removes
the real composer and replaces it with pending prompt chrome. That chrome uses
the same composer glyph, but it has no =e-chat--composer-start-marker= and is
therefore not treated as an editable composer by the composer helpers.

Because the progress timer redraws the running status every
=e-chat-progress-interval=, any text that appears after the prompt-shaped
pending chrome is outside the tracked composer region. The next progress or
activity redraw can delete and recreate the bottom region, which presents as
composer input being constantly reset.

This also explains why the bug is visible only while a turn is running: after
the turn finishes, =e-chat--ensure-composer= creates a real composer again.

* Next step

Decide whether running turns should support a real editable follow-up composer
or should show non-editable pending chrome that cannot be mistaken for input.

If follow-up composition during active turns is desired, the fix should give
the running-turn bottom area a real composer owner and preserve its text across
progress/activity redraws. If active-turn editing should be disabled, the fix
should make the pending chrome visibly non-editable and prevent ordinary text
insertion after the glyph.

In either direction, add a regression test that simulates typing at the bottom
prompt while a progress timer redraw happens, and assert that the chosen
behavior is stable.

* Correction

The conservative behavior was the wrong product choice.  Running turns should
keep a real editable follow-up composer, and redraws must preserve its draft
text and inline context-reference properties.

* Fix evidence

- Added =e-chat-test-progress-rerender-keeps-editable-composer-draft=.  It
  failed before the correction because =turn-started= replaced the active
  composer with pending chrome.
- Added =e-chat-test-progress-rerender-preserves-context-reference=.  It failed
  before the correction because progress redraw removed the composer owner,
  which would also drop inline reference properties if the draft was recreated
  from plain text.
- Added =e-chat-test-submit-keeps-follow-up-composer-editable=.  It failed
  before the correction because =e-chat-submit= deleted the composer and
  inserted pending chrome.
- Changed running status redraws to capture the composer content with text
  properties and restore an editable composer after repainting activity or
  progress status.
- Removed the pending bottom chrome path from the running-turn composer flow.
- Verified the focused regression tests:
  =rtk eldev test -f test/e-chat-test.el e-chat-test-progress-rerender-keeps-editable-composer-draft e-chat-test-progress-rerender-preserves-context-reference e-chat-test-submit-keeps-follow-up-composer-editable=.

* Follow-up fix evidence: final message render

The active-turn composer could still be cleared when the assistant's final
message arrived after the user had already typed a follow-up draft.
=e-chat--insert-entry= captured only whether a composer existed, then deleted
the bottom region and recreated an empty composer after inserting the final
assistant entry.

- Added =e-chat-test-final-message-preserves-follow-up-draft=. It failed
  before the fix because the draft became the empty string immediately after
  the assistant =message-added= event.
- Changed =e-chat--insert-entry= to capture full composer state before deleting
  bottom chrome and restore that state after inserting the entry. For active
  turns, the captured composer state is restored before the running-status
  renderer redraws so the shared status path can preserve it normally.
- Verified the focused regression, adjacent composer redraw selectors, and the
  full chat test file:
  =rtk eldev test -f test/e-chat-test.el e-chat-test-final-message-preserves-follow-up-draft=,
  =rtk eldev test -f test/e-chat-test.el e-chat-test-progress-rerender-keeps-editable-composer-draft e-chat-test-progress-rerender-preserves-scrollback-focus e-chat-test-progress-rerender-strips-composer-presentation-properties e-chat-test-progress-rerender-preserves-context-reference e-chat-test-submit-keeps-follow-up-composer-editable e-chat-test-final-message-preserves-follow-up-draft=,
  and =rtk eldev test -f test/e-chat-test.el=.
- Verified the running Emacs after =e-dev-reload= with a temporary chat buffer
  live probe. After final assistant =message-added= and =turn-finished=,
  the probe returned =:composer-active t= and
  =:composer-text "follow-up draft"=.
