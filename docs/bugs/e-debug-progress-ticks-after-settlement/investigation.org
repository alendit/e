* Source inspected

- =lisp/shells/chat/e-chat.el=: =e-chat--render-event= starts the progress
  timer on =turn-started= and stops it when an assistant =message-added= event
  arrives, or on explicit failure/cancellation/empty-output paths.
- =lisp/shells/chat/e-chat.el=: =turn-finished= updates status and composer
  position, but does not itself stop a progress timer.  That relies on a final
  assistant message having arrived first.
- =lisp/shells/chat/e-chat.el=: =e-chat--render-replayed-active-activity= can
  restore a running =Thinking for ...= row from durable activity after a reload
  or popup reopen when the turn has no terminal event in durable activity.
- =lisp/shells/chat/e-chat.el=: =e-chat--clear= already cancels old timers
  during reload-buffer reattachment, so the stale timer is not just an old
  pre-reload timer surviving buffer clearing.
- =lisp/core/e-harness.el=: async turns are stored in =active-turns= while
  running, but the presentation layer is event-driven and did not reconcile a
  restored progress turn against the harness active-turn state on subsequent
  timer ticks.

* Conclusion

The immediate bad state is a chat presentation reconciliation bug.  The debug
chat buffer can restore or keep a progress timer for a durable turn whose
provider activity has no terminal event, while the harness no longer considers
that session to have a running active turn.  Once in that state, the progress
timer only advances/redraws; it never asks whether its turn still matches the
harness active-turn truth.

=e-dev-reload= is a plausible trigger because reload/popup reopen can restore
running activity from durable events, and in-flight async callback/subscription
state can be disrupted.  The existing reload hardening cancels old timers during
buffer reattach and preserves composer state, but it did not add a timer-side
reconciliation guard.

* Fix

Add a chat presentation guard before each progress tick schedules another
redraw:

- Compare =e-chat--progress-turn-id= with the attached harness
  =active-turns= entry for the session.
- If the harness has a different running turn, a settled entry, or no running
  entry for a durable progress turn, treat the progress turn as stale.
- Settle the open thinking row to a terminal =Thought for ...= line before
  stopping the progress indicator, because stopping without settling would
  immediately restart the timer from the still-active provider round record.
- Keep synthetic presentation tests working by only treating a no-entry turn as
  stale when that turn is known in durable messages or activity.
