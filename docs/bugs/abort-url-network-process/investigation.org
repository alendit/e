#+title: Abort Url Network Process Investigation
#+startup: overview

* Status

Closed.

* Investigation

The abort command flows from =e-chat-abort= through =e-chat-session-abort= to
=e-harness-abort=.  When the active turn has a backend request, the harness
delegates to =e-harness--cancel-active-request=.

For active OpenAI/Codex calls, the request handle comes from
=e-openai-codex--http-request-start=.  Its cancellation path cleaned up the
=url-retrieve= request buffer by getting the buffer process and calling
=kill-process=.

That is incorrect for url.el network processes.  A live url.el connection can
have a process name such as =chatgpt.com<1>=, but it is a network process, not
an OS subprocess.  Calling =kill-process= on that process raises:

#+begin_example
Process chatgpt.com is not a subprocess
#+end_example

The harness also assumed provider/tool cancellation callbacks would not raise.
If a callback raised during abort, the turn could fail to settle as cancelled.

* Fix

The OpenAI request-buffer cleanup now checks the attached process type.  It uses
=kill-process= for real subprocesses and pipes, and =delete-process= for network
processes such as url.el connections.

The harness abort path now treats cancellation callback failures as cancellation
cleanup failures, not as a reason to leave the active turn unsettled.  It records
the condition on the active entry as =:cancel-error= and continues to append
cancelled tool results, set the status to =cancelled=, and emit
=turn-cancelled=.
