#+title: Investigation
#+startup: overview

* Source inspected

- =lisp/shells/e-org-canvas.el=: =e-org-canvas-new-file= used interactive code
  =FOrg Canvas file:=, then passed the selected value directly to
  =find-file-noselect= and forced =org-mode= in the returned buffer.
- Selecting an existing directory therefore visited a Dired buffer. Changing
  that buffer into =org-mode= while Dired's directory listing process was still
  active could leave Dired's process sentinel running against a buffer without
  Dired's expected =dired-subdir-alist= state, matching the reported
  =No subdir-alist= sentinel error.
- =docs/feats/17-org-canvas/plan.org= separates the internal workflows:
  file-backed canvases create or visit an Org file, while
  =e-org-canvas-new-buffer= asks for a target folder and marks the canvas with
  =:needs-file-name= so the first prompt can drive model-suggested file naming.
- The command UX should allow selecting an existing directory from
  =e-org-canvas-new-file= and then delegate into the existing
  =e-org-canvas-new-buffer= workflow.
- After that delegation was added, directory selection created the right
  unsaved canvas metadata but did not display the newly created Org buffer.
  =e-org-canvas--open-session-for-buffer= returns the backing chat buffer and
  does not select any buffer, so the visible window could remain on the
  previous canvas.
- =test/e-org-canvas-test.el= covered normal file-backed canvas behavior and
  the folder-targeted new-buffer workflow, but did not cover directory input to
  =e-org-canvas-new-file= or the selected-window presentation result.

* Live access

- The user's live run already reproduced the sentinel failure after selecting a
  directory.
- An initial narrow fix rejected directory paths with =user-error=. The user
  clarified that selecting a directory should be supported as the
  model-derived-file-name workflow.
- The user then reported that running =e-org-canvas-new-file= again still kept
  the previous canvas visible. The focused regression test was tightened to
  require the selected window to show the newly created Org Canvas buffer.
- After the final display fix, =e-dev-reload= reloaded the repository into the
  running Emacs.
- A live =emacsclient --eval= probe called =e-org-canvas-new-file= with a
  temporary directory path while an older Org Canvas buffer was visible. The
  running Emacs now returns Org Canvas metadata for an unsaved buffer and shows
  that the selected window displays the new target:

  #+begin_example
  (:ok (:needs-file-name t
        :target-folder "/.../e-org-canvas-live-dir-.../"
        :selected-buffer "*e-org-canvas:org-canvas*"))
  #+end_example

* Conclusion

The root cause was that =e-org-canvas-new-file= accepted a directory path even
though its file-visiting path only worked for file paths. The directory path
went through =find-file-noselect=, producing a Dired buffer, and Org Canvas then
forced =org-mode= over that Dired buffer.

The command should treat an existing directory as a request for the
folder-targeted canvas workflow and delegate to =e-org-canvas-new-buffer= before
visiting anything. That keeps Dired out of the path and preserves
model-derived file naming on the first prompt.

For newly-created file or folder canvases, Org Canvas should also display the
Org buffer as the primary presentation surface while still returning the backing
chat buffer for existing call sites and tests that inspect session state.

* Fix evidence

- Replaced the narrow rejection test with
  =e-org-canvas-test-new-file-directory-starts-unsaved-folder-canvas=. It failed
  before the final fix because =e-org-canvas-new-file= still signaled
  =user-error= for directories.
- Updated =e-org-canvas-new-file= to expand the selected path and delegate
  existing directories to =e-org-canvas-new-buffer=.
- Tightened the regression test to assert that the selected window displays the
  newly created unsaved Org Canvas buffer. It failed before the display fix
  because the selected window still showed the prior buffer.
- Added =e-org-canvas--open-session-for-buffer-and-display= and used it for
  newly-created file and folder canvases.
- Verified the focused regression test passes.
- Verified all =test/e-org-canvas-test.el= tests pass.
- Reloaded the running Emacs with =e-dev-reload= and verified the live directory
  probe now returns unsaved Org Canvas metadata with =:needs-file-name= and the
  selected =:target-folder= while selecting the new Org Canvas buffer.
