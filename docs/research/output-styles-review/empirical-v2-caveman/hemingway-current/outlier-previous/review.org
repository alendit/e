* Review summary

The implementation matches the main feature contract. =output-style= is a
built-in =os-base= capability, resolves its active style through
=e-capability-config=, contributes style prose through the existing system
context path, stays inert when no style is selected, and ships the planned
=concise=, =explanatory=, and =hemingway= styles.

Focused tests and adapter-adjacent tests pass. The one open finding is a small
plan-contract mismatch in the unknown configured style diagnostic: capability
construction rejects the value, but the error does not list the registered style
ids as the plan requested.

* Findings

** TODO Low: Configured unknown style error does not list known style ids

The plan says an unknown style id should produce a clear error naming the bad id
and the known ids. The resolver itself does this:
=e-output-style--resolve= signals =Unknown output style `ID'; known styles: ...=,
and the tests cover that path.

The capability construction path rejects unknown configured styles earlier,
inside =e-capability-config-resolve= value validation. A configured value such as
=(:style bogus)= signals the generic capability-config invalid-value error:

#+begin_src emacs-lisp
(progn
  (require 'e-output-style)
  (let ((e-capability-config '((output-style :style bogus))))
    (condition-case err
        (e-output-style-capability-create default-directory)
      (error (list :type (car err) :message (error-message-string err) :data (cdr err))))))
#+end_src

#+begin_example
(:type e-capability-config-invalid-value
 :message "Invalid capability config value: output-style, :style, bogus"
 :data (output-style :style bogus))
#+end_example

This still rejects the invalid config, so the runtime does not silently use a bad
style. The gap is diagnostic quality and plan fidelity. A user who misspells a
style in =e-capability-config= does not see the registered alternatives.

Smallest useful fix direction: move style-id validation for =:style= into a
normalizer or validator path that can signal the resolver's message, or let the
capability constructor call =e-output-style--resolve= for non-nil ids before a
generic invalid-value error wins. Preserve the typed
=e-capability-config-invalid-value= error only if the message still includes the
known ids.

* Plan coverage

** Step 1: Style registry and resolver

Covered.

=e-output-style-registry= is a =defcustom= seeded with =concise=,
=explanatory=, and =hemingway=. =e-output-style-register= adds or replaces
entries. =e-output-style--resolve= returns prose for known ids and signals a
clear unknown-style error for direct resolver calls. =e-dev-reload= includes the
new module and re-evaluates uncustomized =e-output-style-registry= defaults.

** Step 2: =output-style= capability constructor

Mostly covered.

=e-output-style-capability-create= declares the =:style= config option, resolves
configuration through =e-capability-config-resolve= for the given directory, and
creates =output-style= with priority 260. With no style configured, instructions
are nil. With =:style hemingway=, instructions match the style prose. Unknown
option keys signal =e-capability-config-unknown-option=.

The only gap is the open diagnostic finding for unknown configured style ids.

** Step 3: Wire into the default layer

Covered.

=e-base-layer-create= includes =e-output-style-capability-create= after
=base-guidance=, file handling, and shell process capabilities. =test/e-base-test.el=
asserts =output-style= appears in =os-base=, remains inert by default, and sorts
after =base-guidance= when configured.

** Step 4: Interactive set/describe commands

Covered for helpers and command construction.

=e-output-style-set= uses =completing-read= over registered style ids and clears
on empty input. =e-output-style--set-config= writes the =:style= option under
=output-style= without disturbing other capability config. =e-output-style-describe=
prints the active style text in a help window when interactive and returns the
text when called noninteractively.

** Step 5: End-to-end system-channel assertion

Covered in the backend-neutral context layer and adapter-adjacent checks.

=e-output-style-test-style-prose-is-system-fragment= proves the style prose is a
system message and not a non-system conversation turn. Existing adapter tests
plus the live spot check show Anthropic removes system messages from the turns
array, OpenAI Responses moves them into =:instructions=, and OpenAI Chat
Completions keeps them as system-role messages before the user turn.

** Step 6: Document the contract

Covered in architecture documentation.

=docs/architecture.md= lists =output-style= under =os-base= with priority 260,
configuration through =e-capability-config=, inert default behavior, and no
tools/resources.

* Architecture review

The design follows the repository's capability-first direction.

- Ownership is cohesive. =e-output-style.el= owns style registry, config
  resolution, capability construction, and user commands.
- The harness does not gain style-specific logic. It only rebuilds layer-owned
  capabilities for the effective session root, which is the existing runtime
  config path.
- Provider adapters do not gain style-specific branches. They keep handling
  backend-neutral system messages.
- Layers stay presets. =os-base= wires the capability but does not own style
  policy.
- Side effects are narrow: the command mutates =e-capability-config=; context
  assembly only reads configured style state.
- The extension point is real. Additional styles are data in
  =e-output-style-registry= and do not require a new capability or adapter path.

The priority choice is sound. Style prose sorts after operational guidance, so
it shapes response voice without being positioned ahead of safety and tool-use
rules.

* Design self-check

1. If this work stopped here, the project would be better off because users can
   select a stable output voice through the existing capability and system
   prompt path.
2. It moves toward live-configurable agent behavior that is expressed as
   capability-owned guidance, not shell or provider special cases.
3. The likely changing decision is the set of built-in styles. That decision is
   local to the registry. The stable decision is that style is one active
   capability config value.
4. =e-output-style.el= owns the behavior because it is the capability owner for
   style prose and style selection.
5. Coupling stays low. The public surface is small: registry, resolver,
   constructor, set, and describe.
6. Stable core policy does not depend on concrete providers. Existing adapters
   consume system messages as before.
7. Side effects are already outward: the command writes config, and adapters
   handle provider wire shapes.
8. The abstraction is semantically real. A style is named guidance. A capability
   contributes the active guidance.
9. Compatibility expectations are explicit. Nil style preserves the old default
   voice and contributes no extra system fragment.
10. No legacy output-style path exists to remove.
11. Expected errors are handled by config validation and resolver errors.
    Unexpected errors surface.
12. Core behavior is proven by focused registry/config/context tests and
    adapter-adjacent request-body tests.
