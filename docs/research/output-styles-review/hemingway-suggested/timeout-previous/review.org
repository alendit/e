* Review summary

The implementation delivers the core feature described in =docs/feats/38-output-style/plan.org=.
It adds a provider-neutral =output-style= capability, a built-in style registry, config-backed style selection, default inert behavior, =os-base= layer wiring, set and describe commands, focused tests, and architecture documentation.
The capability uses the existing capability instruction path, so style prose reaches the system channel without adding provider-specific request logic.

Two gaps remain.
First, a fresh package load does not expose the user-facing output-style commands.
Second, bad style ids configured through =e-capability-config= are rejected with a generic error instead of the plan's clearer known-style message.

* Findings

** TODO Medium: Fresh package loads do not expose the output-style commands

*** Expected

The plan says the deliverable includes a command to switch the active style.
It names =M-x e-output-style-set= and =M-x e-output-style-describe= as user-facing entrypoints.

Step 1 also says to modify the =e.el= require list or the reload list to include the new module.

*** Observed

=e-dev-reload= includes =lisp/layers/base/e-output-style.el=, and =e-base.el= requires =e-output-style= for the layer factory.
But =e.el= does not require =e-output-style= and the command definitions do not have package autoload cookies.
The =os-base= layer is loaded lazily through the layer registry, so normal package startup can finish without loading the output-style module.

The fresh-load probe returned:

#+begin_example
(:set nil :describe nil :create nil)
#+end_example

*** Why it matters

The feature is meant to be selected by command.
A user can start e, try =M-x e-output-style-set=, and fail to find it unless another workflow already loaded =e-base= or =e-output-style=.
That makes the feature feel absent and misses a direct plan contract.

*** Smallest useful fix

Load or autoload the command-owning module from the package entry path.

The direct fix is to add =(require 'e-output-style)= to =e.el= after load-path setup and before startup completes.
Another acceptable fix is to add autoload cookies for =e-output-style-set= and =e-output-style-describe= and ensure package autoload generation installs them.
Keep =e-base= requiring =e-output-style= because the layer factory depends on the constructor.

*** Verification needed after fix

Run a fresh-load probe that requires only =e.el= and confirms the entrypoints exist:

      :describe (commandp 'e-output-style-describe)

#+end_src

Expected result:

#+end_example

Then rerun focused ERT and warning-as-error compile.

** TODO Low: Configured unknown style ids do not list known style ids

*** Expected

The plan says an unknown style id should raise a clear error naming the bad id and the known ids.

The direct resolver path meets this contract.
=e-output-style--resolve= signals an error like:

#+begin_example
Unknown output style `nonesuch'; known styles: concise, explanatory, hemingway
#+end_example

*** Observed

When the bad id comes through =e-capability-config=, the config option validator rejects it before =e-output-style--resolve= emits the specific message.
The result is generic:

#+begin_example
e-capability-config-invalid-value
Invalid capability config value: output-style, :style, bogus
#+end_example

*** Why it matters

Global and directory-local config are core selection paths for this feature.
A user who mistypes a style id there should see the valid choices.
The current message is safe, but less useful than the plan promised.

*** Smallest useful fix

Move non-nil style-id validation into a normalizer or validator path that calls =e-output-style--resolve= and preserves its known-id message.
If the typed =e-capability-config-invalid-value= error is retained, include the bad id and known ids in that error path.

*** Verification needed after fix

Add a focused test with:

(let ((e-capability-config '((output-style :style bogus))))
  (e-output-style-capability-create default-directory))
#+end_src

Assert that the error message contains =bogus= and a known id such as =hemingway=.
Then rerun focused ERT.

** Step 2: =output-style= capability constructor

Mostly covered.

=e-output-style-capability-create= registers and declares the =:style= config option, resolves config for the supplied directory, creates =output-style= with priority 260, and returns nil instructions when no style is configured.
Unknown option keys signal =e-capability-config-unknown-option=.

Finding 2 remains because unknown configured style ids use a generic capability-config invalid-value error.

Covered.

=e-base-layer-create= includes =e-output-style-capability-create= in =os-base=.
=test/e-base-test.el= asserts =output-style= appears in =os-base=, remains inert by default, activates when configured, and sorts after =base-guidance=.

** Step 4: Interactive set/describe commands

Implemented but not reliably available after a fresh package load.

=e-output-style-set= uses =completing-read= over registered style ids and clears on empty input.
=e-output-style--set-config= writes =(:style ID)= under =output-style= without disturbing other capability config.
=e-output-style-describe= prints the active style in a help window when interactive and returns the text when called noninteractively.

Finding 1 blocks reliable command discovery.

** Step 5: End-to-end system-channel assertion

Covered at the backend-neutral context layer and supported by adapter source inspection.

=e-output-style-test-style-prose-is-system-fragment= proves the style prose becomes a =:role 'system= message and not a non-system conversation turn.
The Anthropic adapter removes system messages from the conversation turns and maps them to the top-level =:system= field.
The OpenAI Chat Completions adapter emits backend-neutral system messages as leading ="system"= messages.
There is no style-specific adapter regression test, but no output-style-specific adapter branch is present.

** Step 6: Document the contract

Mostly covered.

=docs/architecture.md= records =output-style= as an =os-base= capability and states that it is instructions-only, priority 260, selected through =e-capability-config=, and inert when unset.

* Architecture review

The decomposition is sound.
The output-style registry, resolver, config contract, capability constructor, and commands live in =e-output-style.el=.
The =os-base= layer only assembles the capability.
The harness and provider adapters do not know about output-style-specific behavior.

Dependency direction is acceptable.
=e-output-style.el= depends on stable core capability and config contracts.
=e-base.el= depends on =e-output-style= because it creates the =os-base= capability set.
Core code does not depend on the base-layer style module.

The feature keeps side effects at the boundary.
=e-output-style--set-config= mutates =e-capability-config= for the interactive command path.
The resolver and constructor remain simple and testable.

The main architecture issue is load ownership.
The commands are global user entrypoints, but the command-owning module is behind lazy layer loading.
Global commands need package entry load ownership or package autoload ownership.

* Design self-check

1. If this work package stopped here, would the project be better off than if it had not been built?
   Yes. The core output-style capability works and is tested, but command discoverability needs a fix.
2. What final direction does this move toward?
   It moves toward configurable, capability-owned system-channel guidance without provider or shell special cases.
3. Which decisions are likely to change, uncertain, or stable?
   The registry shape, single active style, capability config ownership, and priority 260 contract are stable for v1.
   The built-in style set and command UX can change locally in =e-output-style.el=.
4. What component owns this behavior, and why?
   =e-output-style.el= owns it because output style is a named guidance capability with its own registry and config option.
5. Does this change increase or reduce coupling?
   It adds a narrow =os-base= dependency and avoids coupling providers, shells, or the harness to style ids.
6. Did any dependency start pointing the wrong way?
   No. Core and adapters do not depend on output-style-specific behavior.
7. Could any side effect be moved outward into an adapter or shell?
   No. The only new side effect is writing =e-capability-config= from a user command.
8. Are the abstractions and interfaces semantically real?
   Yes. Style registry, config option, and capability are real extension seams used by the current feature.
9. What compatibility expectations apply?
   The capability is inert when unset, so adding it to =os-base= preserves default behavior.
10. What legacy code can we remove now?
    None found.
11. Where are expected errors handled, and where do unexpected errors surface?
    Direct unknown style ids surface through the resolver.
    Config-path unknown style ids currently surface through generic capability-config validation; finding 2 asks to improve that expected error.
12. What tests prove the core behavior independently of the full system?
    =test/e-output-style-test.el= proves the registry, constructor, helpers, and system-message contribution.
    =test/e-base-test.el= proves default-layer presence, inert behavior, activation, and ordering.
