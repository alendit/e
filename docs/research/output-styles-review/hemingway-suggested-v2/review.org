* Review summary

The implementation delivers the main =output-style= feature.

It adds a built-in output-style capability, a style registry, config-backed
resolution, inert default behavior, =os-base= layer wiring, set and describe
commands, focused tests, and architecture documentation. The capability uses the
existing system-channel path. It does not add provider-specific prompt logic.

Three review notes remain open.

First, a fresh =e.el= package load does not expose the output-style commands or
constructor. The module is loaded through the lazy =os-base= path, not through the
package entry path or package autoloads.

Second, invalid style ids configured through =e-capability-config= are rejected,
but the error does not list the known style ids as the plan requested.

Third, the current built-in registry ships five styles. The attached v1 plan
named three built-ins. The broader set may be fine, but it should be intentional
and documented.

* Findings

** TODO Medium: Fresh package loads do not expose the output-style commands

*** Expected

The plan says the deliverable includes a command to switch the active style. It
names =M-x e-output-style-set= and =M-x e-output-style-describe= as user-facing
entrypoints.

Step 1 also says to modify the =e.el= require list or the live reload list to
include the new module.

*** Observed

=e-dev-reload= includes =lisp/layers/base/e-output-style.el=, but =e.el= does
not require =e-output-style= and the command definitions do not have package
autoload cookies.

A fresh =-l e.el= package load returns:

#+begin_example
(:set nil :describe nil :create nil)
#+end_example

for:

#+begin_src emacs-lisp
(list :set (commandp 'e-output-style-set)
      :describe (commandp 'e-output-style-describe)
      :create (fboundp 'e-output-style-capability-create))
#+end_src

*** Why it matters

The feature is meant to be selectable by command. A user can start e, try
=M-x e-output-style-set=, and fail to find it unless another workflow has
already loaded =e-base= or =e-output-style=. That makes the feature feel absent
and misses a direct plan contract.

*** Smallest useful fix

Load or autoload the command-owning module from the package entry path.

The direct fix is to add =(require 'e-output-style)= to =e.el= after load-path
setup and before startup finishes. Another acceptable fix is to add proper
autoload cookies for =e-output-style-set= and =e-output-style-describe= and make
sure package autoload generation installs them. Keep =e-base= requiring
=e-output-style= because the layer factory depends on the constructor.

*** Verification needed after fix

Run a fresh-load probe that requires only =e.el= and confirms all entrypoints
exist:

#+begin_src emacs-lisp
(list :set (commandp 'e-output-style-set)
      :describe (commandp 'e-output-style-describe)
      :create (fboundp 'e-output-style-capability-create))
#+end_src

Expected result:

#+begin_example

#+end_example

Then rerun focused ERT and warning-as-error compile.

** TODO Low: Configured unknown style error does not list known style ids

*** Expected

The plan says an unknown style id should raise a clear error naming the bad id
and the known ids.

The direct resolver path meets this contract. =e-output-style--resolve= signals
an error that names the unknown id and the registered ids.

*** Observed

When the bad id comes through =e-capability-config=, the config option validator
rejects it before =e-output-style--resolve= can emit the specific message.

The error is generic:

#+begin_example
e-capability-config-invalid-value
Invalid capability config value: output-style, :style, bogus
#+end_example

*** Why it matters

Directory-local and global config are core selection paths for this feature. A
user who mistypes a style id there should see the valid choices. The current
message is safe but less useful than the plan promised.

*** Smallest useful fix

Move non-nil style-id validation into a normalizer or validator path that can
call =e-output-style--resolve= and preserve its known-id message. If the typed
=e-capability-config-invalid-value= error is retained, include the known ids in
its message.

*** Verification needed after fix

Add a focused test with:

#+begin_src emacs-lisp
(let ((e-capability-config '((output-style :style bogus))))
  (e-output-style-capability-create default-directory))
#+end_src

Assert that the error message contains =bogus= and a known id such as
=hemingway=. Then rerun focused ERT.

** TODO Low: Built-in style set is broader than the v1 plan

*** Expected

The plan names a small v1 built-in set:

- =concise=
- =explanatory=
- =hemingway=

It also says to keep the set small in v1 and use the registry to make adding
more styles cheap later.

*** Observed

The current =e-output-style-registry= ships five built-in styles:

- =concise=
- =explanatory=
- =hemingway=
- =hemingway-suggested-v2=
- =caveman=

The tests now assert all five are present.

*** Why it matters

This does not break the mechanism. It changes the product contract from the
attached plan. Built-ins are user-visible through completion, config, and help.
A broader style list should be intentional and documented, especially because
=caveman= was listed in the high-level goal examples but excluded from the v1
built-in decision.

*** Smallest useful fix

Either trim the built-in registry and tests back to the three planned v1 styles,
or update the plan and architecture docs to state that =hemingway-suggested-v2=
and =caveman= are accepted built-ins for this slice.

*** Verification needed after fix

If trimming, rerun focused ERT and confirm =e-output-style-ids= returns only the
three planned ids by default. If documenting, update the feature plan or review
handoff and confirm completion lists all intended built-ins.

* Plan coverage

** Step 1: Style registry and resolver

Mostly covered.

=e-output-style-registry= is a =defcustom=. It contains =concise=,
=explanatory=, =hemingway=, =hemingway-suggested-v2=, and =caveman=.
=e-output-style-register= adds or replaces entries. =e-output-style--resolve=
returns prose for known ids and raises a clear unknown-id error for direct calls.

The module is included in =e-dev-reload=. It is not loaded or autoloaded from a
fresh =e.el= package load. This is finding 1.

** Step 2: Capability constructor

Covered.

=e-output-style-capability-create= registers config options, resolves the active
style for a directory, creates an =output-style= capability, keeps instructions
nil when unset, and uses priority 260.

Unknown option keys still raise the feature-07 unknown-option error. Unknown
configured style values are rejected, but their error text does not list known
style ids. This is finding 2.

** Step 3: Default layer wiring

Covered.

=e-base-layer-create= includes =output-style= in =os-base=. Focused tests assert
capability presence, inert default behavior, and system-fragment ordering after
base guidance.

** Step 4: Interactive set and describe commands

Implemented with a loadability gap.

=e-output-style-set= uses =completing-read= over registered ids and writes
=:style= through =e-output-style--set-config=. =e-output-style-describe= renders
the active style through a help window when called interactively. Pure helpers
are tested.

The commands are not available after a fresh =e.el= load. This is finding 1.

** Step 5: End-to-end system-channel assertion

Covered at the capability level.

=e-output-style-test-style-prose-is-system-fragment= builds context messages
from a configured =output-style= capability and asserts that the style prose is a
=:role 'system= message and not a conversation turn.

There is no style-specific Anthropic or OpenAI adapter test. That is acceptable
for this implementation because the feature rides the same system-message shape
that adapter tests already cover.

** Step 6: Document the contract

Mostly covered.

=docs/architecture.md= records =output-style= as an =os-base= capability with
priority 260 and =:style= config. If the five-style registry is intended, the
feature documentation should state that accepted built-in set.

* Architecture review

The dependency direction is sound. The base layer depends on a base-layer
capability module. The capability module depends on stable capability and config
contracts. No provider or shell code depends on output-style-specific behavior.

The side effect in =e-output-style--set-config= is at the interactive/config
boundary. The resolver and constructor stay small and testable.

The main architecture gap is load ownership. The command-owning module lives
behind a lazy layer factory, but the commands are global user entrypoints.
Global entrypoints need package-level load ownership or autoload ownership.
