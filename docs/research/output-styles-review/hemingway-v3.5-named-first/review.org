* Review summary

The implementation delivers the core =output-style= capability design.

It adds a built-in output-style capability, a style registry, config-backed
style resolution, inert default behavior, =os-base= layer wiring, set and
describe commands, focused tests, and architecture documentation. The
capability uses the existing system-channel path. It does not add
provider-specific prompt logic.

Three review findings remain open.

First, a fresh =e.el= package load does not expose the output-style commands or
constructor. The module is loaded through the lazy =os-base= layer path, not
from the package entry path or package autoloads.

Second, invalid style ids configured through =e-capability-config= are rejected,
but the error does not list the known style ids as the plan requested.

Third, the source registry ships more built-in styles than the v1 plan named.
That may be intentional later work, but it changes the user-visible product
surface and should be documented or trimmed.

* Findings

** TODO Medium: Fresh package loads do not expose the output-style commands

*** Expected

The plan says the deliverable includes a command to switch the active style. It
names =M-x e-output-style-set= and =M-x e-output-style-describe= as user-facing
entrypoints.

Step 1 also says to modify the =e.el= require list or the live reload list to
include the new module.

*** Observed

=e-dev-reload= includes =lisp/layers/base/e-output-style.el=, but =e.el= does
not require =e-output-style= and the command definitions do not have package
autoload cookies.

A fresh =-l e.el= package load returns:

#+begin_example
(:set nil :describe nil :create nil)
#+end_example

for:

#+begin_src emacs-lisp
(list :set (commandp 'e-output-style-set)
      :describe (commandp 'e-output-style-describe)
      :create (fboundp 'e-output-style-capability-create))
#+end_src

*** Why it matters

The feature is meant to be selectable by command. A user can start e, try
=M-x e-output-style-set=, and fail to find it unless another workflow has
already loaded =e-base= or =e-output-style=. That makes the feature feel absent
and misses a direct plan contract.

*** Smallest useful fix

Load or autoload the command-owning module from the package entry path.

The direct fix is to add =(require 'e-output-style)= to =e.el= after load-path
setup and before startup finishes. Another acceptable fix is to add proper
autoload cookies for =e-output-style-set= and =e-output-style-describe= and make
sure package autoload generation installs them. Keep =e-base= requiring
=e-output-style= because the layer factory depends on the constructor.

*** Verification needed after fix

Run a fresh-load probe that requires only =e.el= and confirms all entrypoints
exist:

#+begin_src emacs-lisp
(list :set (commandp 'e-output-style-set)
      :describe (commandp 'e-output-style-describe)
      :create (fboundp 'e-output-style-capability-create))
#+end_src

Expected result:

#+begin_example
(:set t :describe t :create t)
#+end_example

Then rerun focused ERT and warning-as-error compile.

** TODO Low: Configured unknown style error does not list known style ids

*** Expected

The plan says an unknown style id should raise a clear error naming the bad id
and the known ids.

The direct resolver path meets this contract. =e-output-style--resolve= signals
an error like:

#+begin_example
Unknown output style `nonesuch'; known styles: concise, explanatory, hemingway, hemingway-suggested-v2, hemingway-v3, caveman
#+end_example

*** Observed

When the bad id comes through =e-capability-config=, the config option validator
rejects it before =e-output-style--resolve= can emit the specific message.

The error is generic:

#+begin_example
e-capability-config-invalid-value
Invalid capability config value: output-style, :style, bogus
#+end_example

*** Why it matters

Global and directory-local config are core selection paths for this feature.
When a user mistypes a style id there, the error should show valid choices. The
current message names the bad value but does not help recovery.

*** Smallest useful fix

Preserve capability-config validation while moving style-id validation through
the resolver.

One direct approach is to use an option normalizer for =:style= that accepts
nil, requires a symbol, calls =e-output-style--resolve= for non-nil values, and
returns the original symbol. Then the resolver message can surface for unknown
ids.

*** Verification needed after fix

Add a test with =e-capability-config='((output-style :style bogus))= that
asserts the error message contains =bogus= and at least one known id such as
=hemingway=.

Rerun focused ERT.

** TODO Low: Source ships extra built-in styles beyond the v1 plan

*** Expected

The plan's v1 built-in style set is small:

- =concise=
- =explanatory=
- =hemingway=

It says to keep the set small in v1. The registry exists so more styles can be
added later.

*** Observed

=e-output-style-registry= currently includes six source-default styles:

#+begin_example
(concise explanatory hemingway hemingway-suggested-v2 hemingway-v3 caveman)
#+end_example

=test/e-output-style-test.el= also treats all six as built-ins.

*** Why it matters

The style registry is user-visible through =e-output-style-set=. Extra built-ins
change the product surface, completion choices, and documentation burden. They
also make the implementation drift from the v1 acceptance text.

This may be intended follow-up work from output-style experiments, but the code
should make that decision explicit.

*** Smallest useful fix

Pick one of two paths:

- trim source defaults and tests back to =concise=, =explanatory=, and
  =hemingway=; or
- update the feature contract and documentation to state that =caveman=,
  =hemingway-suggested-v2=, and =hemingway-v3= are now shipped built-ins.

Do not leave the tests as the only record of the expanded public surface.

*** Verification needed after fix

If trimming, assert =e-output-style-ids= only includes the three v1 ids by
default.

If documenting, update architecture or feature notes and keep the built-in
style test aligned with the documented list.

* Architecture review

The ownership and dependency direction are sound. The output-style behavior
lives in a base-layer capability module. That module depends on stable
capability and config contracts. No shell, provider adapter, or core harness
code knows about specific style ids.

The feature adds one narrow dependency from =e-base.el= to
=e-output-style.el=. That is appropriate because =os-base= assembles the
capability set.

The side effect in =e-output-style--set-config= sits at the interactive/config
boundary. The resolver and constructor stay small and testable.

The main architectural issue is load ownership. The command-owning module is
behind a lazy layer feature, but the commands are global package entrypoints.
Global entrypoints need package-level load ownership or autoload ownership.

* Design self-check

1. If this work package stopped here, would the project be better off than if it
   had not been built?
   Yes. The core provider-neutral style capability works and is tested, but
   command discoverability still needs a fix.
2. What final direction does this move toward?
   It moves toward configurable system-channel guidance owned by capabilities,
   not providers or shells.
3. Which decisions are likely to change, uncertain, or stable?
   The registry shape, single active style, config ownership, and priority
   contract are stable enough for v1. The built-in style set and command UX may
   change, and both remain local to =e-output-style.el=.
4. What component owns this behavior, and why?
   =e-output-style.el= owns it because output style is a base guidance
   capability with its own registry and config contract.
5. Does this change increase or reduce coupling?
   It adds a narrow =os-base= dependency and avoids provider coupling.
6. Did any dependency start pointing the wrong way?
   No. Core and adapters do not depend on output-style-specific behavior.
7. Could any side effect be moved outward into an adapter or shell?
   No. The only new side effect is writing =e-capability-config= from an
   interactive command.
8. What state does this change add or mutate?
   It adds the customizable registry and mutates =e-capability-config= when the
   set command runs. That is the right storage lifetime for user-selected style
   config.
9. What is the performance impact?
   It adds small alist and plist lookups during capability construction. There
   is no meaningful hot-path risk.
10. Are the abstractions semantically real?
    Yes. Style registry, config option, and capability are real extension seams
    tied to current behavior.
11. What compatibility expectations apply?
    The capability is inert when unset, so adding it to =os-base= preserves
    default behavior. No legacy path is needed.
12. What legacy code can we remove now?
    None found.
13. Where are expected errors handled, and where do unexpected errors surface?
    Direct unknown style ids surface through the resolver. Config-path unknown
    ids currently surface through generic capability-config validation; finding
    2 asks to improve that expected error.
14. What tests prove the core behavior independently of the full system?
    =test/e-output-style-test.el= proves the registry, constructor, helpers,
