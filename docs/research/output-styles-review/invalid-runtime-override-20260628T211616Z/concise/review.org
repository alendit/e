* Review summary

The implementation delivers the core feature from =docs/feats/38-output-style/plan.org=.
It adds a named style registry, resolves the active style through =e-capability-config=, contributes style prose through the existing system-channel capability path, wires =output-style= into =os-base=, adds set and describe commands, and covers the main paths with focused tests.

Two gaps remain.
Fresh package loads do not expose the user commands.
Invalid configured style ids report a generic config error instead of the planned resolver error that lists known ids.

* Findings

** TODO Medium: Fresh package loads do not expose the output-style commands

*** Expected

The plan says the deliverable includes a command to switch the active style.
Step 1 says to modify the =e.el= require list or the reload list to include the new module.
Step 4 adds =M-x e-output-style-set= and =M-x e-output-style-describe= as user-facing commands.

*** Observed

=e-dev.el= includes =lisp/layers/base/e-output-style.el= in the reload list.
=e-base.el= requires =e-output-style= for layer creation.
But =e.el= does not require =e-output-style= and does not autoload =e-output-style-set= or =e-output-style-describe=.
The =os-base= layer spec is lazy through =:feature e-base=.
A fresh package load finishes without defining the output-style entrypoints.

Evidence:

#+begin_example
(:set nil :describe nil :create nil)
#+end_example

*** Why it matters

The feature is meant to be selectable by command.
If the command is missing after package startup, the feature feels absent.
The user should not need to create an =os-base= layer first to discover the global style command.

*** Smallest useful fix

Load the command-owning module from =e.el= or add real autoloads for the user commands.
The direct plan-matching fix is to add =(require 'e-output-style)= to the package startup path after load-path setup.
Keep =e-base= requiring it too, because the layer factory depends on the capability constructor.

*** Verification needed after fix

Run a fresh-load probe that requires only =e.el= and confirms the command entrypoints exist:

#+begin_src emacs-lisp
(list :set (fboundp 'e-output-style-set)
      :describe (fboundp 'e-output-style-describe)
      :create (fboundp 'e-output-style-capability-create))
#+end_src

Expected result:

#+begin_example
(:set t :describe t :create t)
#+end_example

Then rerun focused ERT and warning-as-error compile.

** TODO Low: Invalid configured style ids do not list known ids

*** Expected

The plan says an unknown style id should raise a clear error naming the bad id and the known ids.
The resolver implements this for direct calls to =e-output-style--resolve=.
The set helper also calls the resolver before writing config.

*** Observed

When the bad id comes through =e-capability-config=, the config option validator rejects it before =e-output-style--resolve= can emit its specific message.
The result is generic:

#+begin_example
e-capability-config-invalid-value
Invalid capability config value: output-style, :style, bogus
#+end_example

*** Why it matters

Global and directory-local config are core selection paths.
When a user mistypes a style id there, the error should tell them which ids are valid.
The current message names the bad value but does not help the user recover.

*** Smallest useful fix

Preserve capability-config validation while moving style-id validation through the resolver.
One direct approach is to use an option normalizer for =:style= that accepts nil, requires a symbol, calls =e-output-style--resolve= for non-nil values, and returns the original symbol.

*** Verification needed after fix

Add a test with =e-capability-config='((output-style :style bogus))= that asserts the error message contains =bogus= and at least one known id such as =hemingway=.
Rerun focused ERT.

* Architecture review

Ownership is sound.
The output-style behavior lives in a base-layer capability module.
That module depends on the stable capability and config contracts.
No shell, provider adapter, or core harness code knows about specific style ids.

The feature adds one narrow dependency from =e-base.el= to =e-output-style.el=.
That is appropriate because =os-base= assembles the capability set.

The side effect in =e-output-style--set-config= sits at the interactive/config boundary.
The resolver and constructor stay small and testable.

The main architectural issue is load ownership.
The command-owning module is behind a lazy layer feature, but the commands are global package entrypoints.
Global entrypoints need package-level load ownership or autoload ownership.

* Design self-check

1. If this work package stopped here, would the project be better off than if it had not been built?
   Yes. The core provider-neutral style capability works and is tested, but command discoverability still needs a fix.
2. What final direction does this move toward?
   It moves toward configurable system-channel guidance owned by capabilities, not providers or shells.
3. Which decisions are likely to change, uncertain, or stable?
   The registry shape, single active style, config ownership, and priority contract are stable enough for v1.
   The built-in style set and command UX may change, and both remain local to =e-output-style.el=.
4. What component owns this behavior, and why?
   =e-output-style.el= owns it because output style is a base guidance capability with its own registry and config contract.
5. Does this change increase or reduce coupling?
   It adds a narrow =os-base= dependency and avoids provider coupling.
6. Did any dependency start pointing the wrong way?
   No. Core and adapters do not depend on output-style-specific behavior.
7. Could any side effect be moved outward into an adapter or shell?
   No. The only new side effect is writing =e-capability-config= from an interactive command.
8. Are the abstractions semantically real?
   Yes. Style registry, config option, and capability are real extension seams tied to current behavior.
9. What compatibility expectations apply?
   The capability is inert when unset, so adding it to =os-base= preserves default behavior.
10. What legacy code can we remove now?
    None found.
11. Where are expected errors handled, and where do unexpected errors surface?
    Direct unknown style ids surface through the resolver.
    Config-path unknown ids currently surface through generic capability-config validation; finding 2 asks to improve that expected error.
12. What tests prove the core behavior independently of the full system?
    =test/e-output-style-test.el= proves the registry, constructor, helpers, and system-message contribution.
    =test/e-base-test.el= proves default-layer presence and ordering.
