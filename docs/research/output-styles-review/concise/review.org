- The feature reuses the existing capability instruction path instead of adding provider-specific prompt plumbing.
- =e-output-style-capability-create= uses =:instruction-priority 260=, matching the plan's placement after base guidance.

* UX and performance playthrough

- First use after a fresh package load is not coherent: =M-x e-output-style-set= and =M-x e-output-style-describe= are not commands after loading =e.el= alone.  This is finding 1.
- Once the base module is loaded, the registry, describe helper, and capability constructor work and return scalar results immediately.
- Bad config recovery is only partly helpful.  The generic config error names the bad value, but it does not list valid style ids.  This is finding 2.
- No profiling was needed.  The feature performs small alist/plist lookups and does not add a redraw, network, transcript traversal, or provider hot path.

* Findings

** TODO Medium: Fresh package loads do not expose the output-style commands

*** Expected

The plan says the deliverable includes a command to switch the active style.
Step 1 says to modify the =e.el= require list or the reload list for the new module.
Step 4 adds =M-x e-output-style-set= and =M-x e-output-style-describe= as interactive entrypoints.

*** Observed

=e-dev.el= reloads =lisp/layers/base/e-output-style.el=, and =e-base.el= requires it, but =e.el= does not require it or autoload its commands.
A fresh =-l e.el= package load leaves these entrypoints unavailable:

#+begin_example
(:set nil :describe nil :create nil)
#+end_example

*** Why it matters

The feature is meant to be selectable by a global command.
A user can start e and fail to discover =M-x e-output-style-set= until some other path has forced =e-base= to load.
That misses the command-delivery part of the plan.

*** Smallest useful fix

Add package-level load ownership for the user commands.
The simplest fix is to require =e-output-style= from =e.el= after load-path setup, or to add proper autoload cookies and ensure the package autoload path exposes =e-output-style-set= and =e-output-style-describe=.
Keep =e-base= requiring =e-output-style= because the layer factory depends on the constructor.

*** Verification needed after fix

Run a fresh-load probe requiring only =e.el= and expect:

#+begin_example
(:set t :describe t :create t)
#+end_example

Then rerun focused ERT and warning-as-error compile.

** TODO Low: Invalid configured style ids do not list known ids

*** Expected

The plan says an unknown style id should raise a clear error naming the unknown id and the known ids.
=e-output-style--resolve= does this for direct resolver calls.
=e-output-style--set-config= also does this before writing config.

*** Observed

When the unknown id comes from =e-capability-config=, the option validator rejects it before the resolver message can surface:

#+begin_example
e-capability-config-invalid-value
Invalid capability config value: output-style, :style, bogus
#+end_example

*** Why it matters

Global and directory-local config are core selection paths for this feature.
A mistyped style id should tell the user how to recover without requiring them to inspect the registry separately.

*** Smallest useful fix

Validate configured non-nil style ids through a path that preserves the resolver's message.
One direct approach is to use an option normalizer that accepts nil, checks symbol type, calls =e-output-style--resolve= for non-nil symbols, and returns the original style id.

*** Verification needed after fix

Add a test for =e-capability-config='((output-style :style bogus))= that asserts the error message contains =bogus= and a known id such as =hemingway=.
Rerun focused ERT.

* Plan coverage

- Step 1, style registry and resolver: mostly covered.  Registry, resolver, register helper, and tests exist.  The module is included in live reload, but not in the fresh =e.el= package entry path.
- Step 2, capability constructor: covered.  It resolves config, declares =:style=, is inert when unset, and uses priority 260.
- Step 3, default layer wiring: covered.  =os-base= includes =output-style= and tests assert presence and ordering.
- Step 4, interactive set/describe commands: implemented, but not reliably discoverable after a fresh package load.
- Step 5, end-to-end system-channel assertion: covered at the backend-neutral capability-message level.  Adapter code already maps/removes system messages correctly, but there is no style-specific Anthropic/OpenAI adapter test.
- Step 6, document the contract: covered in =docs/architecture.md=.  This review did not edit =docs/feats/index.org= by request.

* Architecture review

The design is sound overall.
Output style is capability-owned, provider-neutral, and shell-independent.
The base layer activates it without making provider adapters know about style ids.
The only stateful side effect is the interactive helper writing =e-capability-config=, which is the right boundary.

The main ownership gap is command load ownership.
A lazy layer module can own capability construction, but a global user command needs either package-level loading or autoload exposure.

* Design self-check

1. If this work package stopped here, would the project be better off than if it had not been built?
   Yes.  The core capability works and is tested, but command discoverability blocks full acceptance.
2. What final direction does this move toward?
   Provider-neutral system-channel guidance controlled by capability configuration.
3. Which decisions are likely to change, uncertain, or stable?
   The registry shape, one active style, config ownership, and priority are stable for v1.  Built-in styles and command UX can change locally in =e-output-style.el=.
4. What component owns this behavior, and why?
   =e-output-style.el= owns it because output style has its own registry, config option, resolver, and capability constructor.
5. Does this change increase or reduce coupling?
   It adds a narrow base-layer dependency and avoids provider coupling.
6. Did any dependency start pointing the wrong way?
   No provider, adapter, or shell depends on output-style-specific logic.
7. Could any side effect be moved outward into an adapter or shell?
   No.  The only new side effect is an interactive config write.
8. Are the abstractions semantically real?
   Yes.  Style registry, config option, and capability all map to current behavior.
9. What compatibility expectations apply?
   The capability is inert when unset, so default behavior is preserved.  No legacy compatibility path is needed.
10. What legacy code can we remove now?
    None found.
11. Where are expected errors handled, and where do unexpected errors surface?
    Direct unknown style errors surface through the resolver.  Config-path error quality needs the low-severity fix above.
12. What tests prove the core behavior independently of the full system?
    =test/e-output-style-test.el= covers registry, constructor, helpers, and system-fragment contribution.  =test/e-base-test.el= covers default-layer presence and ordering.
