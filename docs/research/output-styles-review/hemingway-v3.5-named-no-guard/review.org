* Review summary

The implementation delivers the core =output-style= feature.

It adds a built-in output-style capability, a style registry, config-backed
style resolution, inert default behavior, =os-base= layer wiring, set and
describe commands, focused tests, and architecture documentation. The capability
uses the existing system-channel path. It does not add provider-specific prompt
logic.

Three plan-contract gaps remain open.

First, a fresh =e.el= package load does not expose the output-style commands or
constructor. The module is loaded through the lazy =os-base= layer path, not from
the package entry path or package autoloads.

Second, invalid style ids configured through =e-capability-config= are rejected,
but the error does not list the known style ids as the plan requested.

Third, the current built-in registry is broader than the small v1 set in the
plan. The plan named =concise=, =explanatory=, and =hemingway=. The source also
ships =hemingway-suggested-v2=, =hemingway-v3=, and =caveman=.

* UX and performance playthrough

- First use after a fresh package load is not coherent. The planned command
  entrypoints are not available until another path loads =e-output-style=. This
  is finding 1.
- First use with no style is coherent after the module is loaded. The capability
  exists in =os-base= but contributes no instructions.
- Active style description is clear after the module is loaded. The helper
  returns the active id and the style prose.
- Style switching helper is covered by tests. It writes =(:style ID)= under
  =output-style= without disturbing other capability config.
- Error recovery for a bad configured style is safe but incomplete. The bad
  value is rejected, but the message does not show valid choices. This is
  finding 2.
- Completion choices include more built-in styles than the v1 plan names. This
  may be acceptable, but it should be an explicit product decision. This is
  finding 3.
- Responsiveness is acceptable. The feature performs small alist and plist
  lookups and adds no redraw, network, or transcript hot path. Profiling was not
  needed.

The interactive =M-x e-output-style-set= UI itself was not driven visually in
this pass. Its pure helper and describe text are covered, and the command uses
standard =completing-read= plus =message=.

* Findings

** TODO Medium: Fresh package loads do not expose the output-style commands

*** Expected

The plan says the deliverable includes a command to switch the active style. It
names =M-x e-output-style-set= and =M-x e-output-style-describe= as user-facing
entrypoints.

Step 1 also says to modify the =e.el= require list or the live reload list to
include the new module.

*** Observed

=e-dev-reload= includes =lisp/layers/base/e-output-style.el=, but =e.el= does
not require =e-output-style= and the command definitions do not have package
autoload cookies.

A fresh =-l e.el= package load returns:

#+begin_example
(nil nil nil)
#+end_example

for:

#+begin_src emacs-lisp
(list (fboundp 'e-output-style-set)
      (fboundp 'e-output-style-describe)
      (fboundp 'e-output-style-capability-create))
#+end_src

*** Why it matters

The feature is meant to be selectable by command. A user can start e, try
=M-x e-output-style-set=, and fail to find it unless another workflow has
already loaded =e-base= or =e-output-style=. That makes the feature feel absent
and misses a direct plan contract.

*** Smallest useful fix

Load or autoload the command-owning module from the package entry path.

The direct fix is to add =(require 'e-output-style)= to =e.el= after load-path
setup and before startup finishes. Another acceptable fix is to add proper
autoload cookies for =e-output-style-set= and =e-output-style-describe= and make
sure package autoload generation installs them. Keep =e-base= requiring
=e-output-style= because the layer factory depends on the constructor.

*** Verification needed after fix

Run a fresh-load probe that requires only =e.el= and confirms all entrypoints
exist:

(list :set (commandp 'e-output-style-set)
      :describe (commandp 'e-output-style-describe)
      :create (fboundp 'e-output-style-capability-create))
#+end_src

Expected result:

#+begin_example

#+end_example

Then rerun focused ERT and warning-as-error compile.

** TODO Low: Configured unknown style error does not list known style ids

*** Expected

The plan says an unknown style id should raise a clear error naming the bad id
and the known ids.

The direct resolver path meets this contract. =e-output-style--resolve= signals
an error like:

#+begin_example
Unknown output style `nonesuch'; known styles: concise, explanatory, hemingway, hemingway-suggested-v2, hemingway-v3, caveman
#+end_example

*** Observed

When the bad id comes through =e-capability-config=, the config option validator
rejects it before =e-output-style--resolve= can emit the specific message.

The error is generic:

#+begin_example
e-capability-config-invalid-value
Invalid capability config value: output-style, :style, bogus
#+end_example

*** Why it matters

Directory-local and global config are core selection paths for this feature. A
user who mistypes a style id there should see the valid choices. The current
message is safe but less useful than the plan promised.

*** Smallest useful fix

Move non-nil style-id validation into a normalizer or validator path that can
call =e-output-style--resolve= and preserve its known-id message. If the typed
=e-capability-config-invalid-value= error is retained, include the known ids in
its message.

*** Verification needed after fix

Add a focused test with:

#+begin_src emacs-lisp
(let ((e-capability-config '((output-style :style bogus))))
  (e-output-style-capability-create default-directory))
#+end_src

Assert that the error message contains =bogus= and a known id such as
=hemingway=. Then rerun focused ERT.

** TODO Low: Built-in style set is broader than the v1 plan

*** Expected

The plan names a small v1 built-in set:

- =concise=
- =explanatory=
- =hemingway=

It also says to keep the set small in v1 and use the registry to make adding
more styles cheap later.

*** Observed

The current source =e-output-style-registry= ships six built-in styles:

- =concise=
- =explanatory=
- =hemingway=
- =hemingway-suggested-v2=
- =hemingway-v3=
- =caveman=

The focused tests assert all six are present.

*** Why it matters

This does not break the mechanism. It does change the product contract from the
attached plan. Built-ins are user-visible through completion, config, and help.
A broader style list should be intentional and documented, especially because
=caveman= was listed in the high-level goal examples but excluded from the v1
built-in decision.

*** Smallest useful fix

Either trim the built-in registry and tests back to the three planned v1 styles,
or update the plan and architecture/docs handoff to state that
=hemingway-suggested-v2=, =hemingway-v3=, and =caveman= are accepted built-ins
for this slice.

*** Verification needed after fix

If trimming, rerun focused ERT and confirm =e-output-style-ids= returns only the
three planned ids by default. If documenting, update the feature handoff and
confirm completion lists all intended built-ins.

* Plan coverage

** Step 1: Style registry and resolver

Mostly covered.

=e-output-style-registry= is a =defcustom=. =e-output-style-register= adds or
replaces entries. =e-output-style--resolve= returns prose for a known id and
lists known ids for direct unknown-id calls. The new module is in the live reload
list.

The package entry path does not load or autoload the module, so fresh package
loads do not expose the new commands or constructor.

The built-in set is broader than the plan's v1 list.

** Step 2: Capability constructor

Covered.

=e-output-style-capability-create= resolves =:style= through
=e-capability-config=, declares =:style= as its only config option, returns nil
instructions when unset, and uses =e-output-style-instruction-priority= 260.

The unknown-option path is covered. The unknown-style configured-value message
is weaker than the plan requested.

** Step 3: Wire into the default layer

Covered.

=e-base-layer-create= includes =output-style= in the =os-base= layer. Tests
assert the capability is present, inert by default, and sorted after base
guidance when active.

** Step 4: Interactive set and describe commands

Implemented with a load/discovery gap.

=e-output-style-set= uses =completing-read= over registered style ids and clears
on empty input. =e-output-style-describe= writes a help-window description when
called interactively and returns the string in noninteractive use.

The helper behavior is tested. The commands are not available after a fresh
=e.el= load.

** Step 5: End-to-end system-channel assertion

Covered at the provider-neutral capability level.

=e-output-style-test-style-prose-is-system-fragment= builds messages from a
configured output-style capability and asserts the style prose appears in a
system-role message and not a conversation turn. The implementation uses the
same capability instruction path that the adapters already consume.

** Step 6: Document the contract

Partly covered.

=docs/architecture.md= records =output-style= as an =os-base= capability with
priority 260 and =e-capability-config= =:style= config.

The broader built-in style set is not clearly documented as an intentional
change from the attached v1 plan.

* Architecture review

The dependency direction is sound. The base layer depends on a base-layer
capability module. The capability module depends on the stable capability and
config contracts. No provider or shell code depends on output-style-specific
behavior.

The side effect in =e-output-style--set-config= is appropriately at the
interactive/config boundary. The resolver and capability constructor remain
small and testable.

The main architectural gap is load ownership. The command-owning module lives
behind a lazy layer factory, but the commands are global user entrypoints.
Global entrypoints need either package-level load ownership or autoload
ownership.

* Design self-check

1. If this work stopped here, would the project be better off than if it had not
   been built?
   Yes. The core mechanism works and is tested. The command discoverability gap
   still blocks full acceptance.
2. What final direction does this move toward?
   It moves toward configurable system-channel guidance that remains
   provider-neutral and capability-owned.
3. Which decisions are likely to change, uncertain, or stable?
   The registry shape, single active style, and config ownership are stable
   enough for v1. The built-in style set and command UX may change, and those
   choices remain local to =e-output-style.el= and package loading.
4. What component owns this behavior, and why?
   =e-output-style.el= owns it because output style is a base guidance
   capability with its own registry and config contract.
5. Does this change increase or reduce coupling?
   It adds a narrow dependency from =os-base= to the new capability and avoids
   provider coupling.
6. Did any dependency start pointing the wrong way?
   No provider, adapter, or shell depends on the style implementation.
7. Could any side effect be moved outward into an adapter or shell?
   No. The only new side effect is writing =e-capability-config= from an
   interactive command.
8. What state does this change add or mutate?
   It adds the style registry and mutates =e-capability-config= when the user
   sets a style. That is the correct durable scope for global and project style
   selection.
9. What is the performance impact?
   It is small. The feature adds small alist/plist lookups and one optional
   static-prefix instruction fragment.
10. Are the abstractions semantically real?
    Yes. Style registry, config option, and capability are real extension seams
    tied to current behavior.
11. What compatibility expectations apply?
    The capability is inert when unset, so adding it to =os-base= preserves
    default behavior.
12. What legacy code can we remove now?
    None found in this review.
13. Where are expected errors handled?
    Unknown style ids surface through the resolver in direct/helper paths.
    Config-path error quality needs the low-severity fix above.
14. What tests prove the core behavior independently of the full system?
    =test/e-output-style-test.el= proves the registry, constructor, helpers, and
    system-message contribution. =test/e-base-test.el= proves default-layer
    presence and ordering.
