* Review summary

Core feature works.

=output-style= is a built-in capability. It resolves style prose through
=e-capability-config=. It is inert by default. It is wired into =os-base=. Active
style prose becomes a system fragment with priority 260. Tests pass for the
focused layer, output-style, OpenAI, and Anthropic surfaces.

Three review findings remain:

1. Fresh =e= package load does not expose =M-x e-output-style-set= or
   =M-x e-output-style-describe=.
2. Bad configured style ids fail with the generic capability-config error, not
   the plan's clear unknown-style error listing known ids.
3. Built-ins drifted from the v1 plan: current registry ships five styles, while
   the plan's v1 set named three.

- Configured style path: works. With =((output-style :style caveman))=, the
  =os-base= layer creates an =output-style= capability with caveman prose in its
  instructions.
- Describe path after module load: works. =e-output-style--describe-string=
  reports the active style and prose.
- Bad config path: recoverable but vague. It reports a generic invalid value
  instead of naming the bad style and known ids. See Finding 2.
- Responsiveness: no profiling needed. Registry lookup and context contribution
  are small alist/plist operations; focused tests run in milliseconds.

* Findings

** TODO Medium: Fresh package load does not expose output-style commands

Plan expected user-facing commands:

- =M-x e-output-style-set=
- =M-x e-output-style-describe=

Current source defines both commands in =lisp/layers/base/e-output-style.el=,
but fresh =(require 'e)= does not load that module and there are no autoload
cookies on the commands in the reviewed source. =e.el= loads defaults and shell
modules, but not =e-output-style=. =e-default-layers.el= registers the =os-base=
spec lazily with =:feature e-base=, so =e-output-style= appears only after the
layer path loads =e-base=.

Evidence:

#+begin_example
(:e t :output nil :set nil :describe nil :create nil)
#+end_example

Why it matters: the main switching flow is an =M-x= command. If users cannot
find it after package load, the feature is present in the lazy layer but not
usable as documented.

Small fix direction: add autoload cookies for the interactive commands and make
package generation expose them, or require =e-output-style= from a startup path
that intentionally owns globally available commands. Keep capability creation
lazy if desired; only the command entrypoints need to be discoverable.

** TODO Medium: Invalid configured style id does not list known ids

Plan expected unknown style ids to signal a clear error naming the bad id and
known ids. The direct resolver does that:

#+begin_example
Unknown output style `nonesuch'; known styles: concise, explanatory, hemingway, hemingway-suggested-v2, caveman
#+end_example

But capability construction resolves config through the option validator first.
For =((output-style :style bogus))=, the validator returns nil and
=e-capability-config-resolve= signals the generic invalid-value condition before
=e-output-style--resolve= can produce the helpful message.

Observed evidence:

#+begin_example
e-capability-config-invalid-value
Invalid capability config value: output-style, :style, bogus
#+end_example

Why it matters: project/global config is the documented selection mechanism.
Bad ids there should be easy to diagnose without reading source.

Small fix direction: make the =:style= option normalizer call
=e-output-style--resolve= for non-nil values and return the id, or teach the
validator/error path to include the style registry. Preserve the existing
=e-capability-config-invalid-value= condition if callers depend on it, but add
the actionable text.

** TODO Low: Built-in style set exceeds the v1 plan without a documented contract change

The v1 plan lists these built-ins:

- =concise=
- =explanatory=
- =hemingway=

Current registry ships:

- =concise=
- =explanatory=
- =hemingway=
- =hemingway-suggested-v2=
- =caveman=

Evidence:

#+begin_example
:ids (concise explanatory hemingway hemingway-suggested-v2 caveman)
#+end_example

Why it matters: extra built-ins are user-facing API. =caveman= may be desired
for this comparison run and is mentioned in the high-level goal examples, but
the v1 decision said to keep the built-in set small. =hemingway-suggested-v2=
looks like experimental review content rather than a stable built-in name.

Small fix direction: either trim the shipped v1 registry back to the planned
three styles, or update the feature plan/docs to state that =caveman= and
=hemingway-suggested-v2= are intentional supported built-ins.

* Architecture review

The design is sound.

- Ownership is cohesive: output style behavior lives in one base-layer module.
- Dependency direction is acceptable: layer code depends on core capability and
  config contracts; core does not depend on this feature.
- Side effects are small: =e-output-style--set-config= mutates global
  =e-capability-config= only when the user asks to set a style.
- Provider boundary is clean: no adapter-specific output-style branch.
- Compatibility is simple: nil style keeps old prompt behavior.
- Unexpected errors surface. The one issue is error quality for invalid style
  config, not swallowed failure.

* Design self-check

1. Better if stopped here? Yes. The core prompt-behavior feature works, with
   discoverability and polish gaps.
2. Final direction? Capability-owned guidance selected by config, injected into
   the system channel.
3. Stable vs uncertain? System-channel reuse and priority are stable. Built-in
   style inventory is uncertain; keep that decision explicit.
4. Owner? =e-output-style.el= owns style registry, resolution, commands, and
   capability creation.
5. Coupling? Low. It reuses existing capability/config contracts.
6. Wrong dependency? No new core dependency on layer/provider/UI code found.
7. Side effects outward? Already mostly outward. The config mutation helper is
   command-owned and acceptable.
8. Abstractions real? Yes. Style registry and capability config are real
   variation points.
9. Compatibility? Nil style preserves old behavior.
10. Legacy removable now? None found.
11. Expected errors? Unknown resolver ids are handled well; unknown configured
    ids need better text.
12. Tests? Focused tests prove registry, config, layer, ordering, and system
    fragment behavior independent of a full provider call.
