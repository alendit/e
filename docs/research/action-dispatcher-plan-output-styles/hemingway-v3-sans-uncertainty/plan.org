#+title: Action Dispatcher
#+startup: overview

* Status

In-progress.

Feature 43 adds a safe action call path for agents, shells, and tool-backed Elisp.

* Problem

Actions are a real runtime surface, but their caller API is too low-level.

Today a caller can read an action function from a capability plist:

#+begin_src elisp
(funcall (e-capabilities-action capability :compact) ...)
#+end_src

That form works only when the caller already knows the capability, the action key, the function signature, and the current harness or session values.

Agents do not have that context in a clean form.

The motivating direct call is too long:

#+begin_src elisp
(let ((ctx (e-tools-current-context)))
  (e-chat-session-compact
   (plist-get ctx :harness)
   (plist-get ctx :session-id)))
#+end_src

It also teaches the wrong boundary.  The caller reaches into chat-session functions instead of asking the active capability to run an action.

The runtime already has a better pattern for tools:

#+begin_src elisp
(e-tools-call NAME ARGUMENTS &optional OPTIONS)
(e-tools-call! NAME ARGUMENTS &optional OPTIONS)
#+end_src

Actions need the same context-bound shape.

* Goal

Add one dispatcher for active capability actions.

The Elisp API is:

#+begin_src elisp
(e-actions-call 'chat-session :compact)
(e-actions-call 'chat-session :rename '(:name "Focused chat"))
#+end_src

The model-facing bridge uses the same names:

#+begin_src json
{"capability":"chat-session","action":"compact","arguments":{}}
#+end_src

The dispatcher will:

- resolve the capability from the active harness capability set;
- resolve the action from that capability;
- use =e-tools-current-context= by default;
- accept explicit harness, session, and turn options for shell code;
- normalize symbols, keywords, strings, plists, alists, and hash tables;
- validate required arguments before the handler runs;
- inject declared harness, session, and turn context;
- preserve legacy action functions;
- report clear errors for missing context, missing capability, missing action, missing session, and bad arguments.

* Design

** Capability-owned actions

Actions remain owned by capabilities.

The harness provides the active capability set and lifecycle state.  It does not learn chat-session, layer-selection, Agent Shell, Goal, Grimoire, or Doom action details.

Add an action descriptor shape to the capability contract.

The descriptor records:

- =:handler=, the direct function returned to legacy callers;
- =:caller=, the function used by the dispatcher;
- =:description=, human-facing text for docs and future listing;
- =:parameters=, a compact schema for required arguments;
- =:requires-session=, whether the action needs an active session;
- =:tool-metadata=, metadata returned by the model-facing bridge.

Plain function actions remain valid:

#+begin_src elisp
:actions (list :example #'my-action)
#+end_src

The dispatcher calls a plain function action with one normalized arguments plist.

=e-capabilities-action= keeps returning a function.  Existing shell code and tests keep working.

=e-capabilities-action-spec= returns the descriptor or legacy spec for dispatcher code.

** Dispatcher module

Add =lisp/core/e-actions.el=.

Public API:

#+begin_src elisp
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

The dispatcher owns lookup, normalization, validation, and context injection.

It performs these steps:

1. Normalize the capability name.
2. Normalize the action name.
3. Normalize arguments to a plist.
4. Read explicit options when provided.
5. Read =e-tools-current-context= when options omit context.
6. Resolve effective capabilities from the harness.
7. Resolve the capability by id.
8. Resolve the action by key.
9. Validate required argument keys.
10. Build an action context plist.
11. Call descriptor actions through =:caller=.
12. Call legacy plain functions with one arguments plist.
13. Return the action result.

The dispatcher signals specific action errors instead of falling back to silent nils.

** Elisp and tool chaining alignment

Expose actions beside tools in =run_elisp= guidance:

#+begin_src elisp
(e-tools-call NAME ARGUMENTS &optional OPTIONS)
(e-tools-call! NAME ARGUMENTS &optional OPTIONS)
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

Use =e-tools-call= for model-facing tools.

Use =e-actions-call= for capability actions.

Both APIs use the active harness and session context during a tool turn.

Shell code can pass explicit options:

#+begin_src elisp
(e-actions-call
 'chat-session
 :rename
 '(:name "Focused chat")
 (list :harness harness :session-id session-id))
#+end_src

** Model-facing bridge

Add one generic tool named =call_action=.

Input:

- =capability= :: active capability id;
- =action= :: action name;
- =arguments= :: optional argument object.

The tool delegates to the dispatcher.  It does not duplicate action lookup, context injection, or validation.

The tool returns the action result.

When an action descriptor supplies tool metadata, the bridge returns it with the result.  This preserves the compact-session context refresh path for =chat-session/:compact=.

** Built-in action conversion

Convert standard action surfaces to descriptors.

Chat session actions are session-scoped.  They inject harness and session and call the existing semantic functions.

Expected chat actions include:

- =:submit=
- =:steer=
- =:queue=
- =:compact=
- =:rename=

Layer selection actions are harness-scoped.  They inject the harness and validate the layer argument.

Agent Shell actions already use one plist.  Descriptors document that contract and make those actions dispatcher-visible.

Goal actions also use the plist convention.  Descriptors make them callable through the same API.

Direct functions stay public.

* Non-goals

- Do not move action ownership into the harness.
- Do not make every action model-facing by default.
- Do not add one model tool per action.
- Do not remove direct shell functions.
- Do not break plain function actions in =:actions= plists.
- Do not guess handler signatures from function arity.
- Do not add Grimoire or Doom branches to core dispatch.
- Do not rewrite historical plans that are not active guidance.

- Built-in chat-session actions dispatch through descriptors.
- Layer-selection actions dispatch through descriptors.
- Agent Shell and Goal actions keep their plist convention and gain descriptors.
- Legacy function-valued actions still work.
- Missing context, capability, action, session, and required arguments fail clearly.
- The =call_action= tool delegates to the dispatcher.
- =chat-session/:compact= through =call_action= preserves context-refresh metadata.

- Docs define actions, tools, resource methods, and context providers as distinct runtime concepts.
- Active Grimoire or Doom guidance no longer teaches old direct chat-session calls.
- Running Emacs is reloaded after Lisp changes, or the exact reload failure is recorded.

* Design notes

This slice improves the project even if no later action work lands.  Agents and shells get one safe action call path instead of ad hoc capability plist lookup.

The final direction is clear.  Tools are model-facing operations.  Actions are capability-owned semantic operations.  The dispatcher connects callers to active actions without becoming a second harness.

Core dispatch owns lookup, context injection, normalization, validation, and error reporting.  Each capability owns its own action behavior.

Coupling goes down because callers stop depending on chat-session, layer-selection, Agent Shell, or Goal function signatures.

Dependencies stay in the right direction.  Core dispatch depends on the generic capability and harness contracts, not concrete layer internals.

Side effects stay inside action handlers and services that already own those effects.

No durable state is added.  Dispatch uses current harness, session, and turn state and returns plain Elisp values.

The descriptor abstraction is real.  Built-in actions already use mixed call shapes, and generic dispatch needs an explicit contract.

Compatibility is explicit.  Plain function actions still work, and legacy lookup still returns handlers.

Expected dispatch failures use specific conditions.  Unexpected handler failures surface to the caller.
