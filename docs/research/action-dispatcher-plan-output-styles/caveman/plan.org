#+title: Action Dispatcher
#+startup: overview

* Status

In-progress.

* Problem

Actions are real runtime things, but there is no good call path.

Today capabilities expose =:actions= as a plist.  That is enough for a shell that already knows the concrete function.  It is bad for agent Elisp and generic callers.

Current direct shape:

#+begin_src emacs-lisp
(let ((ctx (e-tools-current-context)))
  (e-chat-session-compact
   (plist-get ctx :harness)
   (plist-get ctx :session-id)))
#+end_src

This leaks implementation details:

- concrete layer function names;
- harness/session threading;
- mixed calling conventions;
- no generic validation;
- no clear error surface;
- no alignment with =e-tools-call=.

Desired shape:

#+begin_src emacs-lisp
(e-actions-call 'chat-session :compact)
(e-actions-call 'chat-session :rename '(:name "Focused chat"))
#+end_src

Matching model-facing bridge:

#+begin_src json
{"capability":"chat-session","action":"compact","arguments":{}}
#+end_src

The harness context should supply lookup data.  Capabilities should still own the actions.

* Goal

Add an action dispatcher.

It must:

- dispatch active capability actions by capability id and action key;
- use active =e-tools-current-context= by default;
- allow explicit harness/session/turn options for shell or test code;
- normalize Elisp and JSON-ish arguments;
- validate required arguments;
- inject harness/session/turn context when declared;
- preserve legacy function-valued actions;
- expose =e-actions-call= to Elisp;
- expose generic =call_action= as the model tool bridge;
- update built-in actions to the declared descriptor shape;
- update guidance and docs so agents learn the new path.

* Non-goals

- Do not move action ownership into =e-harness=.
- Do not make actions global.
- Do not remove direct semantic functions such as =e-chat-session-compact=.
- Do not force every external capability to migrate now.
- Do not guess function calling convention from arity.
- Do not expose every action as its own model-facing tool.
- Do not rewrite old historical plans except where they are active guidance.

* Current state

=action= exists as a capability contribution, but the contract is underdefined.

Known built-in action styles:

- =chat-session= :: direct functions need harness/session.
- =layer-selection= :: direct functions need harness, usually no session.
- =agent-shell-fleet= :: plist arguments already.
- =goal= :: plist arguments already.

=run_elisp= already has tool chaining:

#+begin_src emacs-lisp
(e-tools-call NAME ARGUMENTS &optional OPTIONS)
(e-tools-call! NAME ARGUMENTS &optional OPTIONS)
#+end_src

It has no parallel action chaining API.

Special case: =compact_session= tool returns refresh metadata.  If =call_action= can compact chat sessions, it must preserve that context-refresh behavior.

* Design

** Capability action descriptor

Add a canonical action descriptor while keeping old plist entries valid.

Descriptor responsibilities:

- name the underlying handler;
- optionally provide a caller wrapper;
- describe required parameters;
- declare whether a session is required;
- carry optional tool-result metadata.

Expected Elisp shape can be a struct, not a raw plist, as long as the contract is explicit.

Core fields:

- handler/caller;
- description;
- parameters, including required keys;
- requires-session;
- tool metadata for =call_action=.

Legacy shape stays valid:

#+begin_src emacs-lisp
:actions (list :x #'handler)
#+end_src

Legacy handlers are called with one normalized arguments plist.

=e-capabilities-action= should preserve old behavior: return the function/handler for direct callers.  Add a separate spec lookup for dispatcher use.

** Dispatcher module

Add =lisp/core/e-actions.el=.

It owns only dispatch plumbing:

- normalize capability ids from symbols or strings;
- normalize action names from keywords, symbols, or strings;
- normalize arguments from nil, plist, alist, hash-table, and nested JSON-ish objects;
- read active context from =e-tools-current-context=;
- accept explicit =:harness=, =:session-id=, =:turn-id=, or =:context= options;
- resolve capability from =e-harness-effective-capabilities=;
- get action spec from capability;
- validate required parameters;
- enforce session requirement;
- call descriptor caller or legacy function;
- return plain Elisp results.

Public API:

#+begin_src emacs-lisp
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
(e-actions-dispatch CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

=e-actions-call= returns the raw action result.
=e-actions-dispatch= returns metadata useful to tools/tests.

Errors should be explicit:

- no active harness;
- no active session when required;
- unknown capability;
- unknown action;
- invalid/missing arguments.

** Generic model tool

Add =call_action= to the core layer.

Parameters:

- =capability=, string;
- =action=, string;
- =arguments=, object, default empty.

Behavior:

1. dispatch through =e-actions-dispatch=;
2. return the action result;
3. merge any action tool metadata into the tool result.

Do not duplicate dispatch logic inside the tool.

=chat-session/compact= must return refresh metadata through this path.

** Built-in action conversion

Convert built-in actions to descriptors.

Chat session actions:

- =:submit=;
- =:steer=;
- =:queue=;
- =:compact=;
- =:rename=.

They are session-scoped.  Descriptors inject harness/session and validate inputs like =:text= and =:name=.

Layer selection actions:

- select current layer;
- enable layer;
- disable layer.

They are harness-scoped.  Descriptors validate =:layer= and inject harness.

Agent Shell actions:

- already plist-oriented;
- wrap with descriptors for clear surface and docs;
- do not change core Agent Shell behavior.

Goal actions:

- already plist-oriented;
- wrap with descriptors;
- keep goal capability ownership unchanged.

** Guidance

Teach action calls beside tool calls.

Run-Elisp guidance should show:

#+begin_src emacs-lisp
(e-tools-call NAME ARGUMENTS &optional OPTIONS)
(e-tools-call! NAME ARGUMENTS &optional OPTIONS)
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

Example:

#+begin_src emacs-lisp
(e-actions-call 'chat-session :rename '(:name "Focused chat"))
#+end_src

Rules:

- use tools for model-facing operations;
- use actions for capability-owned semantic operations;
- do not teach direct =e-chat-session-*= calls as normal agent path.

** Docs

Add durable runtime vocabulary.

Define:

- harness;
- session;
- turn;
- capability;
- layer;
- model-facing tool;
- shell-facing action;
- resource method;
- context provider;
- lifecycle hook;
- shell;
- backend adapter.

Update compact bootstrap docs so future agents know what actions are before source spelunking.

* Risks

** Boundary blur

=call_action= is a tool, but the operation it calls is still an action.  Docs must say this plainly.

** Compatibility

User/project capabilities may still publish raw functions.  Keep them valid.

** Calling convention drift

Do not infer arity.  Use descriptors for context injection.

** Context refresh

Compaction is special.  Losing refresh metadata would make =call_action= worse than =compact_session=.

** Over-broad docs edits

Update active guidance and durable docs.  Do not churn historical plans unless they are used as guidance.

* Design notes

The dispatcher is glue, not a second harness.

The harness owns lifecycle, sessions, turns, tool execution, hooks, and active capability sets.  Capabilities own actions.  The dispatcher only resolves active capability action specs, validates the declared contract, injects context, and calls.

This keeps dependency direction clean:

- core knows the generic descriptor;
- chat owns chat actions;
- layer selection owns layer actions;
- Agent Shell owns work actions;
- Goal owns goal actions.

The descriptor shape is additive because compatibility is real.  Existing direct shell code and project-local capabilities should keep working while new callers get the safe path.
