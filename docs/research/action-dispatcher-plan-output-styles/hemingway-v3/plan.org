#+title: Action Dispatcher
#+startup: overview

* Status

In-progress.

* Problem

Capability actions are real runtime behavior, but the caller surface is too low-level.

Today a caller can fetch an action function from a capability:

#+begin_src elisp
(funcall (e-capabilities-action capability :compact) ...)
#+end_src

That works for code that already owns the capability and knows the function contract.  It is a poor fit for agents and generic Elisp.  A caller must find the active capability, know whether the action wants a harness or session id, pass arguments in the right shape, and handle missing state itself.

The common case should be small and context-bound:

#+begin_src elisp
(e-actions-call 'chat-session :compact)
(e-actions-call 'chat-session :rename '(:name "Focused chat"))
#+end_src

This should line up with tool chaining from =run_elisp=:

#+begin_src elisp
(e-tools-call NAME ARGUMENTS &optional OPTIONS)
(e-tools-call! NAME ARGUMENTS &optional OPTIONS)
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

Actions also need a model-facing bridge for cases where the model should call an action without writing Elisp:

#+begin_src json
{"capability":"chat-session","action":"compact","arguments":{}}
#+end_src

* Goal

Add an action dispatcher that calls active capability actions by capability id and action name.

The dispatcher should:

- keep actions owned by capabilities;
- resolve actions through the active harness capability set;
- use the current tool context by default;
- inject harness, session, and turn state only when an action declares that need;
- validate required arguments before the action runs;
- keep legacy function-valued actions working;
- expose =e-actions-call= for Elisp and =run_elisp= code;
- expose a generic =call_action= tool that uses the same dispatcher;
- update guidance and docs so action calls and tool calls use aligned shapes.

* Non-goals

- Do not move action ownership into the harness.
- Do not make actions model-facing by default.
- Do not create one model tool for each action.
- Do not remove direct shell functions such as =e-chat-session-compact=.
- Do not break existing =:actions= plists that map keys to functions.
- Do not guess action calling conventions from function arity.
- Do not add Grimoire or Doom-specific branches to core dispatch.
- Do not rewrite historical plans that are not active guidance.

* Design

** Action descriptors

Add a canonical descriptor shape for actions that need safe generic dispatch.

A descriptor should declare:

- the handler function;
- required argument keys;
- whether the current harness is required;
- whether the current session id is required;
- whether the current turn id is required;
- optional tool-result metadata.

Plain function actions remain valid:

#+begin_src elisp
:actions (list :example #'my-action)
#+end_src

A legacy action receives one normalized arguments plist.  This preserves project-local and external capability code while built-in actions move to descriptors.

=e-capabilities-action= should keep returning a callable for old callers.  New dispatcher code should use a richer action spec lookup.

** Dispatcher

Add =lisp/core/e-actions.el=.

Public API:

#+begin_src elisp
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

The dispatcher should:

1. normalize capability names from symbols or strings;
2. normalize action names from keywords, symbols, or strings;
3. normalize arguments from nil, plist, alist, or hash table input;
4. read =e-tools-current-context= by default;
5. allow explicit harness and session overrides through =OPTIONS=;
6. resolve the capability from the active harness capability set;
7. resolve the action from that capability;
8. validate required arguments;
9. inject declared context values;
10. call descriptor actions through their declared caller;
11. call legacy actions with one arguments plist;
12. signal clear errors for missing context, capability, action, session, or arguments.

The dispatcher should not know about chat, layer selection, Agent Shell, Goal, Grimoire, or Doom.  Those layers own their actions.

** Model-facing bridge

Add one generic tool:

#+begin_src text
call_action
#+end_src

Input:

- =capability= :: active capability id;
- =action= :: action name;
- =arguments= :: optional argument object.

The tool delegates to =e-actions-call=.  It must not duplicate lookup or validation.

The tool should return the action result.  If the action descriptor provides tool-result metadata, the bridge should preserve it.  This is needed for =chat-session/compact= so a compact action can refresh context like the existing =compact_session= tool.

** Built-in actions

Convert standard action surfaces to descriptors.

Chat session actions are session-scoped.  Their descriptors should inject harness and session and call the existing semantic functions.

Expected chat actions include:

- =:submit=
- =:steer=
- =:queue=
- =:compact=
- =:rename=

Layer selection actions are harness-scoped.  Their descriptors should inject the harness and validate the layer argument.

Agent Shell actions already use one arguments plist.  Wrap them in descriptors to document the contract without changing behavior.

Goal actions also use the plist convention.  Wrap them in descriptors so =e-actions-call= can call them the same way as other actions.

** Guidance

Update =run_elisp= guidance so agents see both APIs together:

#+begin_src elisp
(e-tools-call NAME ARGUMENTS &optional OPTIONS)
(e-tools-call! NAME ARGUMENTS &optional OPTIONS)
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

Use =e-tools-call= for model-facing tools.  Use =e-actions-call= for capability actions.

Show a small action example:

#+begin_src elisp
(e-actions-call 'chat-session :rename '(:name "Focused chat"))
#+end_src

Do not teach agents to call =e-chat-session-rename= or =e-chat-session-compact= directly unless they are editing that layer.

** Docs

Add durable vocabulary for runtime primitives.

Define:

- action :: shell-facing semantic operation contributed by a capability;
- tool :: model-facing operation in the tool registry;
- resource method :: URI-addressed data operation;
- context provider :: model-input contributor, not a command surface.

Update bootstrap docs so agents do not have to infer the action contract from old examples.

* Acceptance criteria

- =run_elisp= code can call active capability actions with =e-actions-call=.
- The action API mirrors =e-tools-call= ergonomics.
- The dispatcher resolves actions through active capabilities, not a global table.
- Built-in chat-session actions dispatch through descriptors.
- Layer-selection actions dispatch through descriptors.
- Agent Shell and Goal actions keep their plist convention and gain descriptors.
- Legacy function-valued actions still work.
- Missing context, capability, action, session, and required arguments fail clearly.
- The =call_action= tool delegates to the dispatcher.
- =chat-session/compact= through =call_action= preserves context-refresh metadata.
- Guidance teaches =e-actions-call= instead of direct chat-session functions for normal agent use.
- Docs define actions, tools, resource methods, and context providers as distinct runtime concepts.
- Active Grimoire or Doom guidance no longer teaches the old direct chat-session call shape.
- Running Emacs is reloaded after Lisp changes, or the exact reload failure is recorded.

* Risks

Mixed action conventions are the main risk.  Descriptor calls must be explicit.  Arity guessing would make failures hard to see.

The descriptor shape can grow too large.  Keep it to dispatch needs: handler, required arguments, context requirements, and optional result metadata.

The generic =call_action= tool can blur the action/tool boundary.  Docs must state that =call_action= is a tool bridge.  The called operation remains an action owned by its capability.

Live compaction has special context behavior.  The bridge must preserve refresh metadata or compacting through the generic action path will leave stale model context.

* Design notes

This slice should make the project better even if no more action work follows.  It removes hand-written context plumbing from agent-authored Elisp and gives actions a durable contract.

The final direction is a small, capability-owned action system.  Tools remain model-facing.  Actions remain shell-facing.  The dispatcher connects callers to active capability actions without becoming a second harness.

The uncertain part is how many descriptor fields future actions will need.  Keep that local to =e-actions.el= and =e-capabilities.el=.  Do not add broad abstractions until a real action needs them.
