#+title: Action Dispatcher
#+startup: overview

* Status

In-progress.

* Problem

Actions are a first-class runtime concept, but callers do not have a stable action dispatch API.

Today a capability can expose an =:actions= plist.  That gives shell code a place to find functions, but it does not give agents or generic Elisp code a safe call surface.  A caller that knows it wants to compact a chat session still has to know the concrete function, the action calling convention, and how to inject harness and session identity:

#+begin_src emacs-lisp
(let ((ctx (e-tools-current-context)))
  (e-chat-session-compact
   (plist-get ctx :harness)
   (plist-get ctx :session-id)))
#+end_src

That is too low-level.  It leaks capability internals into agent-authored Elisp.  It also diverges from the new nested tool API, where =run_elisp= code can call active tools through a small context-bound surface.

The runtime needs the same shape for actions:

#+begin_src emacs-lisp
(e-actions-call 'chat-session :compact)
(e-actions-call 'chat-session :rename '(:name "Focused chat"))
#+end_src

The model-facing bridge should match that shape:

#+begin_src json
{"capability":"chat-session","action":"compact","arguments":{}}
#+end_src

The harness should own lookup, context injection, validation, and clear errors.

* Goal

Add a generic action dispatcher that lets Elisp and model-facing tools call active capability actions by capability id and action name.

The dispatcher should:

- keep actions capability-owned;
- resolve only against the active harness capability set;
- inject harness, session, and turn context when an action declares that need;
- validate required arguments before calling an action;
- preserve old =:actions= plist functions for compatibility;
- expose an Elisp API aligned with =e-tools-call=;
- expose a model-facing =call_action= tool using the same dispatcher;
- update built-in action definitions to the canonical descriptor shape;
- update guidance so agents learn =e-actions-call= next to =e-tools-call=.

* Non-goals

- Do not move action ownership from capabilities into the harness.
- Do not make every shell command a model-facing tool.
- Do not remove existing direct functions such as =e-chat-session-compact= or =e-chat-session-rename=.
- Do not require all external or project-local capabilities to migrate at once.
- Do not guess calling conventions from function arity.
- Do not make action dispatch a stronger sandbox than the active harness and capability set already provide.
- Do not rewrite historical plans or inactive examples outside the active guidance path.

* Current behavior

Capabilities can carry an =:actions= plist.  =e-capabilities-action= returns the value for a given action key.  The value is usually a function, but the runtime does not define a richer descriptor contract.

Built-in action conventions are mixed:

- =chat-session= actions call concrete functions that take harness and session values directly.
- =layer-selection= actions are harness-scoped and do not need a session.
- =agent-shell-fleet= actions already take one arguments plist.
- =goal= actions also use the plist convention.

The =run_elisp= tool already supports nested tool calls through =e-tools-call= and =e-tools-call!=.  There is no matching =e-actions-call= surface.

The existing =compact_session= tool has special result metadata that refreshes context after compaction.  A generic action bridge must preserve that behavior when it calls the chat compact action.

* Design

** Action descriptors

Add a canonical action descriptor shape while preserving the old function value shape.

A descriptor should be a plist with at least:

- =:function= :: action implementation function;
- =:required= :: required argument keys;
- =:session= :: non-nil when the dispatcher must inject a session id;
- =:harness= :: non-nil when the dispatcher must inject the harness;
- =:turn= :: non-nil when the dispatcher must inject a turn id;
- =:metadata= :: optional tool-result metadata for the model-facing bridge.

The exact field names can stay small, but the contract must be explicit.  The dispatcher must not infer whether a function expects harness/session arguments by counting parameters.

Legacy actions remain valid:

#+begin_src emacs-lisp
:actions (list :some-action #'some-handler)
#+end_src

A legacy action is called with one normalized arguments plist.  This keeps project-local and external capabilities working until they migrate.

** Dispatcher module

Create a small core module, =e-actions.el=.

It should own:

- name normalization for capability ids and action ids;
- argument normalization from JSON-style alists/hash tables and Elisp plists;
- active capability lookup through the current tool/harness context;
- descriptor validation;
- required argument checks;
- context injection;
- action invocation;
- clear condition types and messages for missing context, missing capability, missing action, and missing arguments.

The public Elisp API should be:

#+begin_src emacs-lisp
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

The default call should use the active =e-tools-current-context=, so code running under =run_elisp= can call actions without manually threading harness/session values.

=OPTIONS= can later accept explicit context overrides.  The first slice should keep that path narrow and test-oriented.

** Model-facing bridge

Add a generic =call_action= tool to the core layer.

Tool input:

- =capability= :: string or symbol name of the active capability;
- =action= :: string or keyword name of the action;
- =arguments= :: object/plist of action arguments.

The tool should delegate to =e-actions-call=.  It should not duplicate dispatch logic.

The bridge should return the action result as the tool content.  If the action descriptor supplies tool-result metadata, the bridge should merge that metadata into the structured tool result.  The chat compact action needs this so a =call_action= compact call refreshes context like the existing =compact_session= tool.

** Built-in actions

Convert built-in action declarations to descriptors where the dispatcher needs explicit context or validation.

Chat session actions:

- =:submit=
- =:steer=
- =:queue=
- =:compact=
- =:rename=

These are session-scoped.  Their descriptors should inject harness and session as needed, validate user-supplied fields such as =:text= or =:name=, and call the existing semantic functions.

Layer selection actions:

- =:select-layer=
- =:enable-layer=
- =:disable-layer= or equivalent current surface

These are harness-scoped.  Their descriptors should inject the harness and validate the =:layer= argument.

Agent Shell actions already use the one-plist convention.  Wrap them in descriptors to document the canonical surface without changing behavior.

Goal actions also use the plist convention.  Wrap them in descriptors so =e-actions-call= can call them consistently.

** Guidance alignment

The =run_elisp= guidance should teach two aligned APIs:

#+begin_src emacs-lisp
(e-tools-call NAME ARGUMENTS &optional OPTIONS)
(e-tools-call! NAME ARGUMENTS &optional OPTIONS)
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

Tools are model-facing operations.  Actions are capability-owned semantic operations for shell and host behavior.  Evaluated Elisp can call both only through the active context.

The guidance should show one compact action example:

#+begin_src emacs-lisp
(e-actions-call 'chat-session :rename '(:name "Focused chat"))
#+end_src

Do not teach agents to call =e-chat-session-rename= or =e-chat-session-compact= directly unless they are working inside that layer's implementation.

** Documentation

Add durable vocabulary for actions.  The docs should state the distinction between:

- action: shell-facing semantic operation contributed by a capability;
- tool: model-facing operation exposed through the provider tool registry;
- resource method: URI-addressed data access surface;
- context provider: model input contributor, not a command surface.

Update the bootstrap docs so future agents see the action contract before they inspect source examples.

* Acceptance criteria

- =run_elisp= code can call active capability actions with =e-actions-call=.
- The API mirrors the ergonomics of =e-tools-call=.
- The dispatcher resolves actions through active capabilities, not a global table.
- Missing context, capability, action, and required arguments fail clearly.
- Built-in action descriptors cover chat-session, layer-selection, Agent Shell, and Goal actions.
- Legacy function-valued actions still work.
- The model-facing =call_action= tool uses the same dispatcher.
- =chat-session= compaction through =call_action= preserves refresh-context behavior.
- Guidance no longer teaches direct chat action functions as the normal agent path.
- Runtime docs define action, tool, resource method, and context provider as distinct concepts.
- The running Emacs is reloaded after Lisp changes, or the exact reload failure is reported.

* Design notes

The dispatcher should stay small.  It should not become a second harness.

The harness owns lifecycle, sessions, turns, active capability sets, hooks, and tool execution.  Capabilities own actions.  The dispatcher only connects a caller to the active capability action and applies the declared contract.

This keeps dependency direction clean.  Chat, Agent Shell, Goal, and layer-selection logic stay in their own layers.  Core dispatch knows only the generic descriptor shape.

The descriptor shape is additive.  That matters because project-local capabilities and user config may already publish function-valued actions.  Compatibility here is real, not speculative.

The model-facing bridge is useful, but it must not blur the concept boundary.  =call_action= is a tool because the model needs a JSON surface.  The thing it calls remains an action because ownership and semantics live with the capability.
