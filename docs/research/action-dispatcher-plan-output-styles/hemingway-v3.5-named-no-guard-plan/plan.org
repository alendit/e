#+title: Action Dispatcher
#+startup: overview

* Status

In-progress.

* Problem

Capability actions are a first-class runtime concept, but they do not have a first-class call path.

Today an agent can call active tools from =run_elisp= with =e-tools-call= and =e-tools-call!=.
There is no matching action API.
An agent that knows it wants a chat action must either call a concrete function such as =e-chat-session-compact= or manually find the active capability, find the action entry, infer the calling convention, and inject harness/session state.

That is brittle.
It also teaches the wrong boundary.
Actions are capability-owned semantic operations.
They should be resolved through the active harness/session context, not through direct calls to one shell or layer module.

The current action table is also mixed.
Some actions take harness and session ids directly.
Some take one arguments plist.
Some are used by shell code.
Some are useful from agent-authored Elisp.
Without a small descriptor contract, a dispatcher would have to guess from function arity.
That would make errors vague and reload behavior fragile.

* Goal

Add an ergonomic action dispatcher that mirrors the existing Elisp tool chaining API.

The main call shape is:

#+begin_src elisp
(e-actions-call 'chat-session :compact)
(e-actions-call 'chat-session :rename '(:name "Focused chat"))
#+end_src

The dispatcher must:

- resolve actions against active capabilities;
- use the current =e-tools-current-context= when called from =run_elisp=;
- accept explicit =:harness= and =:session-id= options when shell code already owns context;
- normalize capability ids, action names, and arguments from Elisp or JSON-like values;
- validate required arguments from an action descriptor;
- inject harness/session/turn context for actions that need it;
- preserve the old direct action lookup contract;
- return plain Elisp values for shell and =run_elisp= callers.

Add a model-facing bridge named =call_action= that uses the same dispatcher and accepts this wire shape:

#+begin_example
{"capability":"chat-session","action":"compact","arguments":{}}
#+end_example

The Elisp and JSON shapes should teach the same names.

* Non-goals

- Do not replace direct shell functions such as chat submit, compact, or rename.
- Do not make every action model-facing by default.
- Do not move action ownership into the harness.
- Do not make the dispatcher know about Grimoire, Doom, Agent Shell, or Goal internals.
- Do not break legacy =:actions (list :x #'fn)= capability entries.
- Do not add a new planning language.
- Do not change unrelated dirty work in feature 42 or Org Canvas files.

* Design

** Action descriptors

Add an =e-action= descriptor to the capability contract.

The descriptor records:

- =:handler= for the old direct action function;
- =:caller= for dispatcher-aware invocation;
- =:description= for documentation and future listing;
- =:parameters= for a compact required-argument schema;
- =:requires-session= for actions that cannot run without a session;
- =:tool-metadata= for bridge behavior that must preserve tool metadata.

Keep =e-capabilities-action= backward compatible.
When an action entry is an =e-action= descriptor, it returns the descriptor handler.
That preserves old shell code and tests that expect a function.

Add =e-capabilities-action-spec= for callers that need the full descriptor.

Legacy function entries remain valid.
The dispatcher wraps them as one-plist handlers.
That gives existing project-local actions a safe migration path.

** Dispatcher module

Add =lisp/core/e-actions.el=.

This module owns action dispatch only.
It should not execute model tools and should not own harness lifecycle.
It should depend on the capability contract, harness effective capability lookup, and the current tool context.

Public API:

#+begin_src elisp
(e-actions-dispatch CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

=e-actions-dispatch= returns a plist with the capability object, normalized capability id, action key, action spec, and raw result.
=e-actions-call= returns only the raw action result.

Error symbols should be explicit:

- =e-actions-no-active-harness=;
- =e-actions-no-active-session=;
- =e-actions-unknown-capability=;
- =e-actions-unknown-action=;
- =e-actions-invalid-arguments=.

The dispatcher should normalize:

- capability ids from symbols or strings;
- action names from keywords, symbols, or strings;
- argument keys from keywords, symbols, or strings;
- hash tables and alists into keyword plists;
- nested JSON-like objects into plists.

** Context rule

The default context comes from =e-tools-current-context=.
That makes action calls work inside =run_elisp= without manual harness/session plumbing.

Explicit options override the current context:

#+begin_src elisp
(e-actions-call
 'chat-session
 :rename
 '(:name "Focused chat")
 (list :harness harness :session-id session-id))
#+end_src

This mirrors =e-tools-call= and keeps tool chaining and action dispatch aligned.

** Model-facing bridge

Add =call_action= as a core tool.

The tool should accept:

- =capability= string;
- =action= string;
- =arguments= object.

It should call =e-actions-dispatch= with the active harness/session/turn context and return the action result.

For =chat-session/:compact=, preserve the same refresh-context metadata as the existing =compact_session= tool.
The bridge must not make the compact path lose its context refresh behavior.

** Capability conversions

Convert built-in capability actions to descriptors where the dispatcher needs clear calling rules.

Convert =chat-session= actions first because this is the motivating use case.
The descriptor callers should map normalized action arguments to existing semantic functions.
Direct shell functions stay unchanged.

Convert =layer-selection= actions as harness-scoped descriptors.
They do not require a session.
Their callers inject the harness and normalize the selected layer id.

Wrap =agent-shell-fleet= actions with descriptors.
These already use a one-plist convention, so the descriptor mainly documents the canonical shape.

Wrap =goal= actions with descriptors.
These also use the one-plist convention.

** Guidance

Update =run_elisp= guidance so agents see both context-bound APIs together:

#+begin_src elisp
(e-tools-call NAME ARGUMENTS &optional OPTIONS)
(e-tools-call! NAME ARGUMENTS &optional OPTIONS)
(e-actions-call CAPABILITY ACTION &optional ARGUMENTS OPTIONS)
#+end_src

The guidance should say:

- use =e-tools-call= and =e-tools-call!= for active model-facing tools;
- use =e-actions-call= for active capability actions;
- do not add a separate action tool just to call an action;
- use the generic =call_action= bridge only when the model-facing tool surface is the right path.

Update durable docs so future agents do not infer the contract from source:

- add =docs/references/runtime_concepts.org=;
- add compact runtime vocabulary to =AGENTS.md=;
- update =README.org= capability docs;
- update =docs/architecture.md= where capabilities, actions, and core tools are described.

* Rollout Notes

This is a bridge slice.
It makes actions easy to call and safe to document.
It does not yet solve action telemetry or tool-surface cleanup.
A later slice should record action calls as first-class activity and then move UI or workflow verbs out of the default model-facing tool list where actions are the better contract.

* Design Self-Check

1. If this work package stopped here, would the project be better off than if it had not been built?
   Yes.  Agents and shell code get one stable action call path, and legacy actions keep working.

2. What final direction does this move toward?
   A capability-first runtime where tools are model-facing primitives and actions are capability-owned semantic operations.

3. Which decisions are likely to change, uncertain, or stable?
   The stable decision is the action/tool boundary.  The descriptor fields may grow later for listing and telemetry.  That uncertainty stays local to =e-action= and the dispatcher.

4. What component owns this behavior, and why?
   Capabilities own actions.  =e-actions.el= owns dispatch.  The harness owns active capability resolution and lifecycle context.  Shells own UI.

5. Does this change increase or reduce coupling?
   It reduces coupling.  Agents no longer need direct calls into chat, Goal, Agent Shell, or layer modules when they want an active action.

6. Did any dependency start pointing the wrong way?
   No.  The dispatcher depends on the capability contract and harness context.  Capability implementations do not depend on the dispatcher unless they want descriptors.

7. Could any side effect be moved outward into an adapter or shell?
   The dispatcher performs no UI side effects.  Action handlers keep side effects in their owning capability or shell-facing service.

8. What state does this change add or mutate?
   It adds no durable state.  It adds action descriptors to capability definitions and transient dispatch context during calls.

9. What is the performance impact?
   Dispatch adds one active capability lookup and small argument normalization.  That cost is bounded and only happens when an action is called.

10. Are the abstractions semantically real?
    Yes.  Actions already exist as capability contributions.  The descriptor names the real contract that callers were already relying on indirectly.

11. What compatibility expectations apply?
    Old =:actions= plist entries and direct shell functions must continue to work.  The new descriptor path is additive.

12. What legacy code can we remove now?
    None in this slice.  Later work can remove direct guidance that teaches concrete action functions where =e-actions-call= is better.

13. Where are expected errors handled?
    Missing context, unknown capability, unknown action, and invalid arguments are dispatcher errors.  Unexpected handler failures should surface.

14. What tests prove the core behavior independently of the full system?
    Capability contract tests and dispatcher tests prove normalization, lookup, required arguments, legacy compatibility, and bridge metadata without a full live session.
